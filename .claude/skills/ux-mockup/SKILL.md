# UX 목업 우선 (ux-mockup)

UI/UX·인터랙션·레이아웃 결정은 **글로 설명하면 이해가 어렵다**. 말 대신 **실제처럼 보이고 만져지는 목업**으로 보여주고 고르게 한다.

## 왜

- 화면 배치·여백·색·동작은 **봐야** 감이 온다. "우측 슬라이드오버 672px에 토글 + 바깥클릭 닫기"를 글로 읽는 것보다 한 번 클릭해 보는 게 빠르다.
- 텍스트 A/B/C 메뉴를 남발하면 사용자가 매 micro-decision을 머릿속에서 렌더해야 한다. **판단은 내가 하고**, 사용자는 결과를 **보고** 고른다.
- 인터랙션(열고닫기·애니메이션·hover·드래그)은 정적 그림으로는 전달이 안 된다 — **실제로 동작하는 목업**이라야 "느낌"이 온다.

## 언제 쓰나

- 트리거: 사용자가 "목업으로", "화면으로 보여줘", "보고 정하자" / 화면 배치·컴포넌트 동작·인터랙션·전환을 글로 설명하려는 순간 / brainstorming 중 시각·인터랙션 결정 지점.
- **안 쓰는 경우**: 백엔드·데이터 모델·순수 로직·API 계약 등 시각/인터랙션 표면이 없는 결정. (그건 글/표가 맞다.)

## 절차

1. **시각/인터랙션 결정으로 인식** — "이건 봐야 안다" 싶으면 즉시 목업 모드. 긴 산문 설명·텍스트 선택지 메뉴를 송출하기 **전**에 전환한다.
2. **세부는 직접 판단** — 모든 micro-choice를 사용자에게 묻지 않는다. 합리적 기본값(플랫폼 관행·기존 패턴·접근성)을 **내가 정해 목업에 반영**하고, **진짜 양자택일**(레이아웃 A vs B, 위치 후보)만 시각 옵션으로 나란히 보여준다.
3. **실제 앱 룩 차용(추측 금지)** — 색·치수·컴포넌트를 진짜처럼. 작업 전 `frontend/src/styles/tokens.css`(색 토큰)·관련 컴포넌트(예: `cardKinds.ts`)·`docs/research/`의 이전 목업에서 팔레트/치수를 grep해 그대로 쓴다. 화면 골격(패널 폭·툴바·시트)도 실제 레이아웃 코드로 확인 후 반영.
4. **인터랙션은 실제로 동작하게** — 열고닫기·토글·hover·드래그·슬라이드/페이드 전환이 결정 대상이면 **클릭되는 버튼 + CSS transition + 필요한 최소 JS**로 진짜 동작하는 목업을 만든다(정적 side-by-side 금지). 순수 레이아웃·비주얼 비교(예: 카드 배경 A/B)는 정적 나란히로 충분.
5. **파일 위치·네이밍** — `docs/research/YYYY-MM-DD-<topic>-mockup.html` (단일 자족 HTML, 외부 의존 0). 날짜는 인자로 받은 현재 날짜.
6. **브라우저로 띄우기** — `open "<절대경로>"`. 그다음 **무엇을 볼지·어느 안을 고를지** 짧게 안내(글 설명은 목업을 못 담는 보조 맥락만).
7. **피드백 반영 iterate** — "이 부분만 다르게" 오면 같은 파일을 고쳐 다시 `open`. 합의될 때까지 반복.
8. **확정은 파일에 박기** — 고른 안·세부 결정은 설계문서/spec(`docs/board/*` 또는 `specs/NNN-*`)에 기록한다. 목업과 대화에만 남기지 않는다(결정은 대화가 아니라 파일).

## 원칙

- **글 설명 최소화, 목업 최대화.** UX는 보여주고, 말은 보조.
- **판단은 내가** — 합리적 기본값을 정해 목업에 반영. 사용자는 결과를 보고 고친다.
- **진짜 선택만 시각화** — 의미 있는 양자택일만 옵션으로. 나머지는 내 판단으로 채운다.
- **실제처럼** — 앱의 진짜 색·치수·컴포넌트. 가짜 회색 박스 말고 그 화면처럼.
- **인터랙션은 만져지게** — 동작이 결정 대상이면 클릭·전환이 실제로 되게.

## 산출물

- `docs/research/YYYY-MM-DD-<topic>-mockup.html` — 자족 인터랙티브(또는 정적) HTML.
- 확정 결정은 별도로 설계문서/spec에 기록.
