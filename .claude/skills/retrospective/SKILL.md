# 작업 회고 (Retrospective)

방금 끝난 작업을 정직하게 돌아본다.
사실 진술 + 어긋난 점 명시 + 다음 작업 교훈 surfacing 이 핵심.

## 왜 회고하는가

세션이 끝나면 컨텍스트는 휘발된다.
회고 산출물은 **다음 세션의 Claude / 사용자가 같은 어긋남을 반복하지 않게 하는 영속 신호**다.
글로벌 룰 (`~/.claude/rules/`) 의 "회귀 사례" 섹션 자체가 회고 누적이므로,
정직한 어긋남 기록이 다음 룰을 만든다.

## 언제 트리거하나

### 사용자 명시 호출
- "회고해줘" / "끝났어 회고" / "마무리하자" / "정리하자" / "돌아보자"
- "/회고" / "/retrospective"

### Proactive suggest (사용자 명시 호출 없이도 본 스킬 적용)
- PR 머지 / 큰 commit 직후
- 다중 라운드 구현 완료 직후 (multi-round-implementation 패턴 종료 신호)
- 디버깅 사이클 (같은 에러 3+ 재시도 또는 한 영역에서 30분+ 작업) 종료 직후
- 사용자가 "이정도면 됐어" / "오늘 여기까지" / "다음 작업 가자" 발언

## 회고 산출물

### 파일 위치 / 네이밍
`docs/retrospectives/YYYY-MM-DD-{slug}.md`

- `YYYY-MM-DD` 는 회고 작성 일자 (한국 시각 기준)
- `slug` 는 작업 핵심 키워드 (영문 kebab-case 권장, 한글 가능)
- 예: `docs/retrospectives/2026-05-18-retrospective-skill-creation.md`

### 형식 — 5축

회고 본문은 다음 5섹션을 모두 포함한다. 각 섹션은 빈약하더라도 헤더는 유지 (작업의 어느 축이 빈약했는지 자체가 신호).

```markdown
# {작업 제목 — 한 줄}

- 일자: YYYY-MM-DD
- 워크트리 / 브랜치: {worktree-name} / {branch}
- 관련 PR / 커밋: (있으면 링크)
- 작업 시간 (대략): (시작 ~ 종료)

## 1. 무엇을 했는가 (사실)
- 산출물 / 파일 / 커밋 단위로 1~5줄
- "~~를 작성했다" / "~~를 수정했다" / "~~를 검증했다" 처럼 객체 + 동작 명확히
- 추측·계획이 아닌 **실제로 일어난 일**만

## 2. 어떻게 했는가 (접근)
- 사용한 도구 / 의사결정 흐름 / 분기 선택
- "왜 그 선택을 했는가" 1~2줄 동행 — 다음 작업의 판단 근거가 됨

## 3. 잘 된 점
1) ...
2) ...
- 사실 근거 (어떤 신호로 잘 됐다고 판단하는지) 함께
- 추측으로 "잘 된 점" 채우기 금지 — 근거 없으면 빈약하게라도 그 사실을 적음

## 4. 어긋난 점
- 사용자 멈춤 신호 발생 횟수 / 시점 (예: "잠깐만", "왜 ~ 했어?")
- 추측 사례 — 어떤 결정을 문서 미정독 / 사용처 미확인 채 박았는가
- 토큰 · 시간 낭비 — subagent 위임 비용 / 검증 명령 누적 / TestContainers wait 등
- 반복 디버깅 — 같은 에러 3+ 재시도 여부
- **회피 가능했던 시점 명시** — 어느 시점에 어떤 신호로 차단 가능했는가

"어긋난 점 없음" 단정 금지. 한 사이클이라도 멈춤 / 추측 / 재시도가 있었다면 기록.
완전히 매끄러웠다면 그 사실을 1줄 + 근거 ("재시도 0 / 멈춤 신호 0 / 토큰 N").

## 5. 다음 작업에 남길 교훈

### 5-1. 작업별 교훈 (이 프로젝트 한정)
- 다음에 같은 / 유사 작업을 할 때 따를 1~3 룰
- 본 프로젝트 (`.claude/rules/...` 또는 `CLAUDE.md`) 에 영구화 후보

### 5-2. 룰 갱신 후보 (사용자 컨펌 필요)

**룰은 포괄적·이식 가능한 원칙만 후보로 삼는다.** 특정 애플리케이션 구현 세부(특정 파일·함수·자료구조·버그 메커니즘, 예: "U+2028 렌더", "이 컴포넌트의 prop")에 종속된 교훈은 룰로 승격하지 않는다 — 그런 교훈은 §5-1(작업별 교훈) 또는 본 회고의 §4 회귀 사례 기록에만 남긴다.

후보 작성 전 self-check: **"이 룰이 다른 작업·다른 코드베이스·다른 도메인에도 적용되는 일반 원칙인가?"** NO(이 앱의 특정 구현에만 해당)면 §5-2 후보에서 제외하고 §5-1로 내린다. 일반 원칙으로 추상화 가능하면(예: 특정 버그 → "측정과 표시가 서로 다른 메커니즘에 의존하면 정합이 깨진다"는 일반 명제) 그 추상화된 형태로 올린다.

채택된 후보별로:
- (1) 갱신 대상 파일 — 글로벌 (`~/.claude/rules/...`) 또는 프로젝트 (`.claude/rules/...` / `CLAUDE.md`)
- (2) 추가 / 수정할 룰 본문 1~3줄 — **일반 원칙 진술** + 구체 사례는 근거로만 인용
- (3) 근거 회귀 사례 (본 회고의 §4 항목 참조)

**사용자 컨펌 전까지 실제 룰 파일 수정 금지.**
```

## 회고 작성 절차

1. **컨텍스트 수집**
   - `git log --oneline -20` / `git status` 로 변경 파일·커밋 파악
   - 본 세션 도구 호출 흐름 회상 — Read / Edit / Bash / Agent 위임 여부
   - 사용자 멈춤 신호 발생 여부 (`잠깐만`, `왜 ~?`, `step N 다시 봐`)
   - 같은 에러 재시도 / TestContainers wait / subagent verbose 흔적
2. **5축 정리** — 위 형식대로 채움. 추측 금지, 사실만.
3. **룰 갱신 후보 surfacing** — §5-2 에 후보 + 근거 사례 명시
4. **사용자에게 미리보기 제시** — 파일 저장 / 룰 갱신 진행 의사 확인
5. **파일 저장** — `docs/retrospectives/YYYY-MM-DD-{slug}.md` (디렉토리 없으면 mkdir)
6. **룰 갱신 (컨펌된 경우만)** — §5-2 후보 중 사용자가 채택한 항목만 실제 룰 파일에 반영

## 금지 패턴

- 추측으로 "잘 된 점" 채우기 — 사실 근거 없으면 빈약하게 둠
- "어긋난 점 없음" 단정 — 멈춤 / 추측 / 재시도 사이클이 1회라도 있었다면 기록
- 룰 갱신을 사용자 컨펌 없이 강행
- 대화 요약만 출력하고 파일 저장 생략
- 5축 중 일부 헤더 생략 (빈약해도 헤더는 유지)
- 영문 회고 작성 (CLAUDE.md 최상위 지침: 한글)

## 출력 언어

- 본 스킬의 모든 산출물·대화는 한글
- 코드 / 명령어 / 식별자 / 영문 고유명사는 영문 유지
- 회고 파일명의 slug 는 영문 kebab-case 권장 (cross-platform / git 친화)
