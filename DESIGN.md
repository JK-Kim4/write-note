## 문제 정의

취미로 소설·단막극을 쓰는 작가가 일상적으로 겪는 두 가지 통증, 그리고 그 뿌리:

1. **메모 컨텍스트 소실 ("죽은 메모")**
   영감이 튀어나오는 순간은 예측 불가능하다 — 지하철, 샤워, 산책. 그 자리에서 공책이나 메모 앱에 한 줄 적어둔다. 며칠~몇 주 후 다시 보면 그 메모가 *어떤 목적*이었는지 모른다. 메모 본체는 살아있는데 *맥락*이 죽어서 활용 불가능.

2. **세션 컨텍스트 소실 ("재진입 비용")**
   글쓰기는 한 호흡에 끝나는 일이 거의 없다. 한 작품이 수십 개의 끊긴 30-90분 세션에 걸쳐 진행된다. 매 세션마다 "내가 어떤 장르를 쓰고 있었지", "지금까지 몇 자 썼지", "목표 분량이 얼마였지", "이 인물이 어디까지 등장했지", "원래 톤이 뭐였지"를 재구성해야 한다. 매 진입에 10-15분의 오리엔테이션 세금.

**공통 뿌리:** 맥락은 *그 순간엔 자명하지만 시간 차원에서 휘발됨*. 현존 도구 중 누구도 "맥락의 영속성"을 정조준하지 않음.

## 무엇이 쩌는가 (Builder Mode 핵심)

> **세션이 끊겨도 컨텍스트가 살아있는 작가의 외장 기억장치.**

생성형 AI 작가 도구(Sudowrite, NovelCrafter)가 "당신 *대신* 글을 쓴다"면, 이 도구는 "당신이 *맥락을 잃지 않게* 한다". 다른 카테고리, 적은 경쟁.

차별점은 *자동화*가 아니라 *마찰 설계*. 사용자가 의도적으로 적어둔 정보를, 적시에 적절한 자리에서, 자연스럽게 다시 만나게 함.

## 타깃 사용자

취미 소설가/극작가/장르 단편 작가. 구체적으로:
- 본업 따로 있고, 주말/저녁의 30-90분짜리 세션으로 글 씀
- 세션 간 간격이 며칠~몇 주
- 영감/소재는 일상 곳곳에서 튀어나옴 (즉, 모바일 캡처가 핵심)
- Scrivener를 정복할 만큼 진지하진 않지만, 메모 앱+Google Docs로는 부족하다고 느낌
- 1차 사용자: 본인. 그리고 한국어 작가 커뮤니티(r/writing 한국어판, 브런치 작가 등)에 비슷한 사람들이 다수.

## 전제 (Premises)

1. 핵심 문제는 "글쓰기 효율"이 아니라 "컨텍스트 영속성"이다. 다른 카테고리.
2. 메모 캡처와 글쓰기 에디터는 **같은 시스템**이어야 한다. 외부 메모 앱과의 연결은 필연적으로 끊어진다.
3. 모바일↔데스크탑 동기화가 필요하므로 클라우드 백엔드. 로컬 우선(Obsidian 스타일) 아님.
4. 취미 작가 타깃. 전업 작가/Scrivener 사용자 대상 아님.
5. 한국어 우선. UI/콘텐츠/검색.
6. **MVP는 AI/LLM을 사용하지 않는다.** 컨텍스트 영속성은 UI 설계로 1차 해결. AI는 V2 이후 실사용 데이터 기반으로 결정.

## 검토한 접근

- **A-원안 (AI 포함 MVP)** — 거절됨. 본인의 우선순위 직감: 지금 당장 필요한 기능이 아님.
- **A-수정 (AI 없는 MVP)** — **채택**. 더 빠른 출시, AI 인프라 부담 없음, 실사용 데이터로 V2 설계.
- **B (AI 코파일럿)** — V2 로드맵으로 보존.
- **C (메모 큐레이터만, 에디터 없음)** — 거절. 외부 에디터에 붙이면 컨텍스트가 새서 핵심 가치 반감.

## 추천 접근 — A 수정안

### MVP 범위 (V1)

#### 메모 캡처
- **모바일**: iOS Shortcut → POST 엔드포인트로 본문만 던지면 inbox 진입. 필드 강제 없음 (마찰 0).
- **데스크탑**: 전역 단축키(⌘+N) 또는 사이드바 빠른 입력. 같은 방식 — 본문만, inbox로.
- 캡처 시점에 자동 저장되는 메타: 시각, 입력 출처(mobile/desktop), 활성 프로젝트(데스크탑이면).

#### 메모 inbox + 큐레이션
- 미분류 메모가 쌓이는 단일 큐.
- 큐레이션 화면에서 한 건씩(또는 다중 선택):
  - 프로젝트로 끌어다 놓기 (드래그 또는 단축키)
  - 태그 추가 (자유 텍스트)
  - 등장인물 연결 (해당 프로젝트의 인물 목록에서)
  - "왜 적었나" 한 줄 코멘트 (선택, 안 적어도 OK)
- 분류 후에도 inbox에서 보임 (필터로 미분류만/분류완료/전체 토글).

#### 프로젝트 메타 카드 (영원히 살아있는 진실)
프로젝트 페이지 상단에 항상 보이는 카드:
- 제목, 장르 (자유 텍스트 — "스릴러", "치유물 단막극" 등)
- 목표 분량 (자/단어)
- 톤·문체 노트 (자유 텍스트)
- 시놉시스 (한 단락)
- 등장인물 목록 (이름 + 한 줄 설명 + 자유 노트)
- 세계관/배경 메모 (자유 텍스트, 마크다운)

수동 편집. 모든 필드 선택. 부담 없게.

#### 에디터
- 본문(중앙) + 우측 사이드 패널.
- 사이드 패널 항상 표시되는 것:
  - 프로젝트 메타 카드 (접힘/펼침 가능)
  - 이 프로젝트에 연결된 메모 목록 (최근 추가순, 태그 필터)
  - 등장인물 목록 (클릭 시 카드 펼침)
- 본문에서 특정 단어/구절에 메모 핀 가능 → 그 부분 옆에 메모 아이콘 표시.
- **분량 카운터**: 현재 자수 / 목표 자수, 진행률 바.
- 자동 저장 (Supabase 실시간).

#### 세션 노트 흐름 (재진입 비용 죽이는 핵심)
- 세션 종료 버튼: 클릭 시 **"다음 세션을 위한 한 줄"** 입력 모달 (선택, 비워도 OK).
  - 예: "민지 등장 직전에 멈춤, 다음 장면은 카페에서 손녀와의 대화"
- 프로젝트 다음에 열 때 즉시 보이는 "지난번 메모" 카드 + 본문 마지막 200자 미리보기 + 마지막 작성 시간/분량 차이.

#### 검색/필터
- 메모: 텍스트 검색, 태그 필터, 프로젝트 필터, 미분류 토글, 등장인물 필터.
- 본문: 프로젝트 내 텍스트 검색.

#### 명시적으로 *안* 만드는 것 (MVP 범위 외)
- 모든 LLM/AI 기능 (자동 분류, 의미 검색, 브리핑 생성 등)
- 협업/다중 사용자
- 모바일 에디터 (모바일은 캡처만, 작성은 데스크탑)
- Export (PDF/EPUB 등) — V1에서는 복사붙여넣기로 충분
- 버전 관리/히스토리 (Supabase 백업으로 충분)
- 알림/뉴스레터/리마인더

### V2 로드맵 (AI 합류, 6주 실사용 후 결정)

후보들 — 실제 사용 데이터 기반으로 우선순위 결정:
- 미분류 메모 자동 분류 제안 (사용자가 컨펌만)
- 의미 기반 메모 검색 (글 쓰는 동안 관련 메모 자동 surface)
- 세션 시작 자동 브리핑
- 장르/톤 드리프트 감지 ("스릴러로 시작했는데 최근 가족 드라마 톤")
- 등장인물 일관성 체크 ("'민지'는 1장에서 22살로 설정, 3장에서 25살로 언급됨")
- 글쓰기 통계 (시간대별 생산성, 평균 세션 길이 등)

## 기술 스택

| 레이어 | 기술 | 이유 |
|--------|------|------|
| 프론트 | Next.js 15 (App Router) + TypeScript | 본인 친숙도, 풀스택 한 통, Vercel 무료 |
| 에디터 | TipTap (or Plate) | 한국어 IME 안정성, 핀/주석 확장 가능 |
| 상태 관리 | Zustand or React Query | 단순함 |
| 백엔드 | Supabase (Postgres + Auth + RLS + Realtime) | 풀스택 빨리, 무료 티어 충분, 모바일↔데스크탑 동기화 자연스러움 |
| 모바일 캡처 | iOS Shortcut → Supabase REST | 앱 안 만들어도 됨, 30분 셋업 |
| 배포 | Vercel | 무료, GitHub 연동 |
| 테스트 | Playwright (E2E 한두 개) + Vitest (유닛) | 핵심 흐름만 |

**비용**: $0/월 (Supabase 무료 + Vercel 무료, AI 안 쓰니 OpenAI 비용 없음). 사용자 증가 시에만 고민.

## 배포 계획 (Distribution)

- **Phase 0 (V1 개발 ~6주)**: localhost + Vercel preview, 본인만 사용.
- **Phase 1 (V1 출시 시점)**: Vercel 프로덕션 도메인. 본인 본격 사용.
- **Phase 2 (V1 + 4주)**: 자체 평가. 매주 쓰고 있는가? Yes면 → 친구 1-2명 초대해서 사용성 피드백.
- **Phase 3 (V2 결정 시점)**: 한국어 작가 커뮤니티(브런치 작가 모임, 글쓰기 디스코드, r/writing 한국어 스레드)에 공유. 무료 베타.

수익화는 V3 이후 고려 사항. MVP는 본인을 위한 도구.

## 성공 기준

- **하드 메트릭 (V1 출시 후 4주 기준)**: 본인이 주 1회 이상 글쓰기 세션 시작 시 이 도구를 연다.
- **소프트 메트릭**: 세션 시작 시 "지난번 어디까지 했지" 재구성 시간이 체감상 줄어든다.
- **실패 신호**: 4주 후에도 본인이 안 쓰고 있다 → 문제 정의 또는 UI가 잘못된 것. V2 가지 말고 MVP 재설계.

## 미해결 질문

1. **모바일 캡처 UX**: iOS Shortcut으로 시작하지만, 진짜 일상화되려면 PWA + 홈화면 추가가 더 나을지? V1에서 결정.
2. **에디터 한국어 IME**: TipTap/Plate의 한국어 입력 안정성. PoC에서 먼저 검증.
3. **모바일에서 메모 큐레이션 필요한가?** 일단 데스크탑 전용으로 시작. 실사용에서 필요하면 V1.5에 추가.
4. **메모를 본문에 핀할 때 위치 추적**: 본문 편집 후에도 핀 위치 유지 방법. TipTap 마크 vs 별도 ID 추적. PoC에서 결정.
5. **여러 문서 vs 단일 문서**: MVP는 프로젝트당 1문서로 시작. 단막극은 OK, 장편 소설은 챕터 분리 필요할 수 있음. V1.5 후보.

## UI/UX 설계 결정 (Design Review · 2026-05-17)

이 섹션은 plan-design-review 세션에서 결정된 UI/UX 내용을 캡처합니다. 시각적 결과물은 [`designs/wireframe.html`](designs/wireframe.html)에 있습니다 (탭으로 7개 view 비교 가능, 우상단 토글로 라이트↔다크 전환).

### 디자인 시스템

Apple 디자인 시스템 토큰 기반, 작가용 app UI로 변환 (마케팅 사이트 패턴은 제외).

**색상**
- Action Blue `#0066cc` 단일 accent (라이트), `#2997ff` (다크). 두 번째 brand color 없음
- Surface: `#ffffff` canvas, `#f5f5f7` parchment (기본), `#fbf8ee` manuscript cream
- Dark surface: `#1c1c1e` tile-1, `#242426` tile-2, `#2c2c2e` tile-3
- 원고지 다크 paper는 `#28231d` warm cream-dark — 촛불 정서 보존
- Text: `#1d1d1f` ink (라이트) / `#f4f4f6` on-dark, never pure black/white

**타이포그래피**
- Display: SF Pro Display (system-ui fallback)
- UI text: SF Pro Text 17px, line-height 1.47, letter-spacing -0.374px
- 본문 프로즈: Noto Serif KR 18-19px, line-height 1.85-1.9
- 원고지: Nanum Myeongjo
- Weight ladder: 300 / 400 / 600 / 700 (no 500)

**Radius**
- Cards: 14px (memo) / 16px (mode card) / 18px (project card)
- Buttons: pill (primary CTA) / 8px (utility)
- Tiles: 0 (no rounding on full-bleed)

**그림자**: 없음. Elevation은 hairline 1px 보더 (#e0e0e0) + surface 색상 변화로만.

**Active state**: 모든 인터랙티브 요소에 `transform: scale(0.95)`

| **홈** | 프로젝트 진입, 재방문 | 인사말 + 미분류 메모 alert + 프로젝트 카드 (지난 세션 인용 hero) + 최근 활동 피드 + 보관함 |
| **작성 — 원고지** | 격자 입력 | 200·400·1000자 지원, 매수 카운터, 5/10/15/20 컬럼 마커, 행 번호 |
| **작성 — 에디터** | Word 스타일 입력 | 풀 툴바 (스타일·폰트·크기·B/I/U/S·정렬·인용·목록·undo) |
| **미리보기** | 임시 읽기 view | 본문 페이지 break + sticky 하단 footer (진행률·페이지·목차·prev/next) |
| **메모 inbox** | 메모 큐레이션 | 필터 칩 (overlap 카운트), 카드 expand 큐레이션 폼 |
| **설정** | 환경 preferences | 작성 / 일반 / 계정 그룹 |

### 핵심 UX 결정

**1. 작성 모드는 설정에서 한 번 결정 (원고지 vs 에디터)**
- 작성 화면 top bar에 모드 토글 없음 (vim/emacs 같은 깊은 개인 취향)
- 설정 → 작성 모드 → 카드 선택하면 작성 화면이 그 모드로 고정
- 미리보기는 *임시 view 액션* — 작성 화면 top bar의 `📖 미리보기` 버튼으로 진입, "← 편집으로 돌아가기"로 복귀

**2. 원고지 3종 크기**
- 200자 (10×20) — 단편·단막극·시
- 400자 (20×20) — 장편 클래식
- 1000자 (25×40) — 대장편·제출용
- 설정에서 선택, 매수 카운팅 자동 변환

**3. 메모 다중 프로젝트 (many-to-many)**
- 한 메모가 여러 프로젝트에 동시 연결 가능 (예: "할머니 손 떨림" → 단막극 + 장편 둘 다)
- 등장인물도 연결된 프로젝트들의 인물 합집합에서 선택
- 필터 카운트는 overlap 반영 (단순 합산 ≠ 모든 메모 카운트)
- "미분류" = 0개 프로젝트 연결, 영구히 미분류 상태도 OK (loose observation 보존)

**4. 메모 캡처 컨텍스트 자동 기록 + 큐레이션 시 "왜 적었나"**
- 캡처 시점: 시각·요일·시간·출처(📱모바일/💻데스크탑)·당시 활성 프로젝트 자동 기록
- 큐레이션 시점 (며칠~몇 주 후): "왜 적었나" 한 줄 입력 (선택, 죽은 메모 방지 핵심)
- 캡처는 마찰 0, 큐레이션은 의도적 — 두 시점 분리가 작가의 실제 흐름

**5. 메모 분류 인터랙션**
- "분류 저장" 클릭 → 카드 녹색 톤 + ✓ 표시 → 800ms 후 슬라이드 + 페이드 아웃
- 등장인물 chip은 평소엔 이름만, 호버 시 툴팁으로 출처 프로젝트 표시 (시각적 잡음 ↓)

**6. 다크 모드 전 앱**
- 설정 → 일반 → 테마: 라이트 / 다크 / 시스템 따라가기
- 전 화면 일관 적용. 원고지 paper는 warm dark-cream (#28231d)로 변환 — 촛불 아래 원고 정서
- 본문 프로즈 weight 라이트→다크 시 400→300 자동 감소 (어두운 배경 부담 완화)
- accent 자동 #0066cc → #2997ff

**7. 분리 원칙 (어디에 무엇이 사는가)**
- **설정 (영구 환경)**: 작성 모드, 원고지 크기, 페이지 형식, 기본 폰트/크기, 테마, 자동 저장
- **에디터 툴바 (라이브 서식)**: 스타일·폰트·크기·B/I/U/S·정렬·인용·목록·undo
- **top bar (영구 컨텍스트)**: 프로젝트 타이틀, 진행률 ring, 미리보기 진입, 사이드 패널 토글
- **사이드 패널 (참조)**: 프로젝트 메타·등장인물·연결된 메모 — *편집*은 사이드 카드 클릭 또는 전용 관리 페이지

**8. 프로젝트 내용(메타·등장인물·태그)은 설정에 없음**
- 이들은 *환경 preferences*가 아니라 *프로젝트 내용*
- 작성 화면 사이드 패널 + 전용 관리 페이지에서 편집
- 설정 nav는 *환경*만 (작성 / 일반 / 계정)

### 세션 진입 ("재진입 비용 0") 메커니즘

여러 자리에 분산 배치된 컨텍스트 신호들이 합쳐서 작동 — *AI 없이* 마찰 설계로만:

1. **홈 화면 프로젝트 카드** — "지난 세션" 인용 hero, 분량 ring, 최근 작업 시각
2. **작성 화면 last-session-bar** — 본문 위에 파란 인용 띠 ("민지 등장 직전에 멈춤...")
3. **세션 종료 시 "다음 세션을 위한 한 줄"** — 사용자가 명시적으로 적어두는 작업 (강제 X, placeholder 명시적)
4. **활동 피드 (홈)** — 최근 작성 세션 시간/분량 + 최근 캡처 메모 본문 인용

각 신호는 *사용자가 의도적으로 적어둔 정보*를 적시에 적절한 자리에 노출. 자동화 아니라 마찰 설계.

### 데이터 모델 영향 (기존 섹션 보완)

```
Memo ↔ Project: many-to-many (이전 1:0 또는 1:1에서 변경)
Memo ↔ Character: 연결된 프로젝트들의 인물 합집합에서 선택
Memo: capture_context (timestamp, source, active_project_at_capture) 자동 기록
Memo: reason_note (큐레이션 시점에 사용자가 적는 "왜 적었나")
```

## What I noticed about how you think

- *"지금 당장 필요한 기능이 아니리서 MVP 범위에서는 LLM활용하지 않고"* — 우선순위 직감이 정확함. 2026년 거의 모든 사이드 프로젝트가 AI 우선으로 디자인되는데, 본인이 자기 통증의 본질을 정확히 짚고 "AI 없어도 풀리는 부분"과 "AI가 진짜 도움 되는 부분"을 구분함. 좋은 빌더의 특징.
- *"예측 불가능한 상황에서 튀어나오기 때문에"* — 캡처 제약을 일상의 진짜 형태로 묘사함. "책상에 앉아 영감을 적자"가 아니라 일상 곳곳에서 휘발되는 순간들. 이게 모바일 캡처 우선순위를 자연스럽게 결정해줌. 사용자(본인)를 잘 이해함.
- *"기존 글의 방향성을 금세 잃어버린다는 점"* — 세션 간 *재진입 비용*을 정확한 언어로 짚음. 작가 도구의 가장 큰 미해결 문제이고, 본인은 이미 본질을 알고 있음. 보통은 "글이 막힌다", "쓰는 게 어렵다" 같은 표면 증상으로 말하는데, 본인은 한 단계 깊은 곳을 봄.
- *"고도화된 텍스트 에디터 형태와... 메모 관리 애플리케이션이 결합된"* — 두 기능의 결합 *자체가* 차별점이라는 걸 알아챔. Scrivener는 결합돼있지만 무거움, Bear+Docs는 분리됨. 그 사이의 빈 자리를 본인이 정확히 봄.

## UI 상세 화면 추가 (Tier 1 · 2026-05-18)

1차 wireframe(8개 view) 검토 후, 프로토타입 구현 전 *반드시 필요한* 흐름을 추가했습니다. 모든 신규 패널은 [`designs/wireframe.html`](designs/wireframe.html)의 "로그인" 탭(인증 12개 패널 토글) + "홈 (빈)" 탭에서 비교 가능합니다.

### 추가된 13개 패널

**인증 흐름 12개** (한 탭 안에서 `data-mode`/`data-switch` panel toggle):

| 패널 ID | 화면 |
|---|---|
| `login` | 로그인 (이메일·비번 + Kakao) |
| `signup` | 회원가입 step 1 — 메서드 선택 (Kakao / 이메일) |
| `signup-email` | 회원가입 step 2 — 이메일 폼 + 약관 |
| `reset-request` | 비밀번호 재설정 step 1 — 이메일 입력 |
| `reset-sent` | 비밀번호 재설정 step 2 — 메일 보냄 안내 |

### 핵심 인증 UX 결정

1. **소셜은 Kakao 단독** — 한국어 작가 타깃. Google/Apple은 V2.
2. **회원가입은 Entry → Wizard** — 메서드 선택(메인) → 이메일 폼(보조). 카카오 가입자가 다수일 것이라는 가정. *Notion·Linear·ChatGPT 패턴*.
3. **로그인은 폼 default** — 재방문이라 즉시 입력 가능해야 빠름. 가입과의 비대칭은 의도된 결정.
4. **매직 링크 V1 제외** — 한국 사용자에게 친숙도 낮음. Supabase Auth는 지원하니 V2 후보로 보존.
5. **카카오는 로딩 디자인 없음** — Supabase 기본 OAuth가 full redirect라 우리 화면이 페이지를 떠남 (모바일 친화적, popup 차단 회피).
6. **에러 메시지에 해결 경로 인라인 링크** — "이미 가입된 이메일입니다. 로그인하기 →" 처럼 막다른 골목 회피.
7. **5회 실패 정책 명시** — alert-error 부제에 "남은 시도 N회. 5회 실패 시 30분 제한" 노출. 보안 정책이 사용자에게 예측 가능하게.
8. **success-icon 단일 accent blue로 통일** — info(✉)와 success(✓) 모두 같은 톤, 아이콘 형태로 의미 차별화 (Apple 패턴).

### 추가된 디자인 시스템 컴포넌트

- **`brand-block`** — 펜촉 SVG icon chip(ink 배경) + write-note wordmark(display 22px) + mode-label(accent blue, uppercase). 모든 입구 카드 상단 일관 적용.
- **`success-block`** — info/success 결과 화면 hero. accent blue 8% 배경 icon chip + display title + prose desc. brand-block 없이 단독 (결과 메시지가 hero).
- **`alert-error` / `form-error` / `form-input.error`** — Apple System Red 톤 (#d70015 라이트, #ff453a 다크). 폼 상단 박스 vs 필드 인라인 vs input border 변형.
- **`submit.is-loading` + `spinner`** — 16px 도넛 spinner + 라벨 변경. `.panel.is-loading`으로 폼 전체 dim.
- **`success-foot`** — 보조 액션 박스 ("다시 보내기" 등). canvas-parchment 배경.
- **`panel toggle`** — `data-mode`(패널 ID) + `data-switch`(전환 target) 메커니즘. JS 한 핸들러로 13개 패널 전환.
- **`empty-hero` / `empty-cta` / `empty-hint-card`** — 빈 상태 hero (display 30px + prose lede) + 큰 hero CTA 카드 (accent icon chip + cue chip) + 작은 hint card 2단 그리드.
- **`demo-hint`** — wireframe 전용 helper (italic + opacity 0.75). "데모: 메일 링크 클릭한 척 → 다음 단계" 같은 외부 트리거 시뮬레이션. **실제 앱엔 없음**.
