이번 라운드 변경

  • 홈 (프로젝트 진입) 신규 — 인사말 + 지난 세션 한 줄 → 미분류 메모 alert → 진행 중 프로젝트 카드 그리드 → 보관함. 각 프로젝트 카드: 상단 색상 accent, 제목 display, 분량 ring, 메모/등장인물/세션 카운트, "지난 세션" 인용. 클릭 시 작성 화면 진입. + 새 프로젝트는 dashed 카드.
  • 등장인물 출처 호버 표시 — inbox 큐레이션에서 등장인물 칩에 마우스 올리면 위에 작은 툴팁 "단막극: 시간" 등 출처 프로젝트가 뜸. 평소에는 "할머니"만 보임 — 시각적 잡음 제거. 첫 메모 expanded 상태에서 호버해보세요.
  • ✓ 후 사라지는 분류 저장 — 첫 메모의 "분류 저장" 버튼 클릭하면 카드가 ✓ 분류됨 상태로 전환 (녹색 success 톤) → 800ms 후 카드 자체가 위로 슬라이드 + 페이드 아웃 → DOM에서 제거. 실제 클릭해보세요.

  • 인사말은 본문체 — "오늘은 어떤 글을 쓸까요?" + 지난 세션 한 줄 인용. 데이터 대시보드가 아니라 *글쓰기 책상* 정서.
  • 프로젝트 카드 상단 색상 accent — 3px 짧은 라인. 작품마다 고유 색 (proj-color-1/2/3). 메모와도 시각적으로 연결됨.
  • "지난 세션" 인용을 카드의 hero로 — 진행률/매수보다 *어디서 멈췄나*가 재진입 신호. italic serif로 작가의 본인 목소리.
  • + 새 프로젝트는 dashed 카드 — 같은 격자 안에 있어서 "프로젝트는 더 만들 수 있다"는 시각 메시지. 강조 안 함.
  • 보관함은 collapsed 리스트 — 완료작은 보이되 주된 시각 무게는 진행 중 작품에.

로그인/회원가입 화면 디자인 결정 (Tier 1 #1)

  • 단일 패널 + 모드 토글 — 한 번에 한 모드만 노출. 카드 하단 "회원가입하러 가기 →" / "← 로그인하러 가기" 링크로 전환 (실제 인터랙션 동작 — 클릭해보세요). wireframe의 비교용 side-by-side가 사용자 인지를 흐림.
  • 회원가입은 Entry → Wizard 패턴 (Notion/Linear/ChatGPT 흐름) — step 1은 메서드 선택 (카카오 / 이메일) 두 버튼만, step 2는 이메일 폼. "이메일로 시작하기" 클릭 시 같은 자리에서 step 2로 전환. 카카오 가입자가 다수일 것이라 가정 — 이메일 폼을 default로 펴 둘 이유 없음. step 2 좌상단에 "← 뒤로" 버튼. mode-label은 "회원가입" → "이메일로 회원가입"으로 변화해서 step 인지.
  • 로그인은 폼 default 유지 (wizard 아님) — 로그인은 *재방문*이라 폼이 바로 보이는 게 빠름. 카카오 단축 경로도 동시 노출. 가입과 흐름 비대칭은 의도된 결정.
  • 카카오 vs 이메일 버튼 위계 — 카카오 = filled 노란 pill (primary), 이메일 = outlined hairline pill (secondary). 두 메서드의 사용 비중 차이를 시각 무게로 반영.
  • 비밀번호 재설정 (c) — 2 step 흐름 — step 1 이메일 입력 (back-btn으로 로그인 복귀, brand-block + helper "가입하신 이메일을 알려주세요"), step 2 메일 보냄 안내 (success-block: ✉ icon chip + 제목 + 본문에 이메일 강조, foot 박스에 "스팸함 확인 / 다시 보내기" 보조 액션, ink pill로 "로그인으로 돌아가기"). 로그인 패널 "재설정" 링크에서 진입.
  • success-block은 brand-block 없이 단독 — 완료 상태에선 브랜드 인지보다 *결과 메시지가 hero*. 메일 아이콘 chip이 success/info 정서 (#0066cc 8% 배경 + accent 색). 회원가입 메일 인증 대기 (b)에서도 동일 패턴 재사용 예정.
  • 비밀번호 재설정 입력 (d) — 2 step 흐름 — step 3 새 비밀번호 입력 (brand-block + helper "안전한 새 비밀번호" + 비밀번호/확인 두 필드 + 보기 toggle), step 4 변경 완료 (success-block: ✓ 체크 icon + "비밀번호가 변경되었어요" + "보안을 위해 모든 기기에서 로그아웃" 안내 + 로그인하기 ink pill). step 3엔 back-btn 없음 — 메일 링크로 진입한 사용자라 이전 단계로 갈 의미 없음.
  • success-icon 단일 accent blue로 통일 — info(메일 보냄)와 success(변경 완료) 모두 같은 톤. Apple 패턴 — 성공 상태도 별도 green 안 쓰고 accent로 일관. 아이콘 형태로 의미 차별화 (✉ vs ✓).
  • demo-hint는 wireframe 전용 helper — italic + opacity 0.75, "데모: 메일 링크 클릭한 척 → 다음 단계" 같은 시뮬레이션 진입. 실제 앱엔 없음. 메일 외부 트리거가 있는 흐름의 다음 화면 보여주기 위해.
  • 비밀번호 보기 toggle은 새 비밀번호 입력에만 — 재설정 입력 시 사용자가 *입력한 값을 의도적으로 확인*할 동기 큼 (오타 방지). 비밀번호 확인 필드엔 없음 — "확인"의 목적 자체가 같은지 검증.
  • 이메일 인증 (b) — 2 step 흐름 — verify-pending (이메일 회원가입 직후 도착, "메일로 마지막 단계를 보냈어요" + ✉ icon + 다시 보내기 + 로그인으로 돌아가기) → verify-done (메일 링크 클릭 후 도착, "가입이 완료되었어요" + ✓ icon + "write-note에 오신 것을 환영합니다" + 시작하기 ink pill → 홈으로 jump).
  • verify-pending 정서: "마지막 단계" — 가입이 *진행 중*이라는 인지가 핵심. reset-sent의 "메일을 확인해주세요"와 차별 — 비밀번호 재설정은 기존 사용자, 이메일 인증은 가입 자체가 미완성. 같은 ✉ icon이지만 텍스트로 단계 차이 표현.
  • verify-done에서 "시작하기" → 홈으로 jump — data-jump="H" 사용. 첫 진입은 빈 홈 (프로젝트 0개) 상태일 텐데, 이는 Tier 1 #5 빈 상태 디자인에서 다룰 예정. 지금은 일단 기존 홈으로 jump.
  • 폼 에러 상태 (e) — 3가지 컴포넌트 — (1) .alert-error 폼 상단 알림 박스 (전역 에러: 로그인 실패, 시도 횟수 제한 안내, ❗ 원형 아이콘 + bold 제목 + 작은 부제), (2) .form-input.error + .form-error 필드 인라인 에러 (! 마이크로 아이콘 + 빨간 텍스트 + 인라인 액션 링크 "로그인하기 →"), (3) .submit.is-disabled hairline 배경 + cursor not-allowed.
  • 에러 색상: Apple System Red 톤 — 라이트 #d70015 (텍스트·border), 다크 #ff453a. 6-8% 배경 opacity로 alert box의 부드러운 강조. 디자인 시스템 토큰화는 V2 (지금은 인라인 hex로 두기).
  • 두 데모 패널 (login-error, signup-error) — 진입은 정상 패널 하단 demo-hint로. login-error: alert-error 박스 + 비밀번호 필드만 error border (이메일은 정상). signup-error: 이메일·비밀번호 *둘 다* form-error + 약관 미체크로 submit disabled — *3가지 에러 상태가 동시 노출되는 worst case 시뮬레이션*.
  • 인라인 액션 링크 패턴 — "이미 가입된 이메일입니다. 로그인하기 →" 처럼 에러 메시지 안에 *해결 경로*를 같이 제공. 사용자가 막다른 골목에 안 빠지게.
  • 로그인 5회 실패 정책 (Supabase 정책 노출) — alert-error 부제에 "남은 시도 N회. 5회 실패 시 30분 제한" 안내. 보안 정책이 사용자에게 *예측 가능하게* 보임.
  • 인증 중 로딩 (a) — 인라인 spinner 패턴 — submit 버튼 안에 16px 도넛 spinner (rgba 30% border + 흰색 top-color, 0.7s linear infinite) + 라벨 변경 ("로그인 중…"). 폼 전체에 .panel.is-loading 적용 시 input·소셜 버튼·약관·링크는 opacity 0.45 + pointer-events none으로 명확히 *대기 상태* 시각화.
  • 카카오는 로딩 디자인 없음 — Supabase 기본 OAuth가 full redirect라 우리 화면 자체가 페이지를 떠남 (모바일 친화적, popup 차단 회피). popup 방식 채택 시에만 추가 dim+spinner 화면 필요 — V1.5에 결정.
  • 모든 폼이 같은 로딩 패턴 재사용 — login-loading 한 패널만 디자인했지만 회원가입·비번 재설정·이메일 인증 모든 submit이 동일 컴포넌트 (.submit.is-loading + .spinner) 적용. wireframe에선 한 사례만 시각화.
  • 카드 상단은 브랜드 정체성 (아이콘 + wordmark + 모드 라벨) — 펜촉 SVG chip(ink 배경) + "write-note" display 22px wordmark + accent blue 작은 모드 라벨. 시적 인사말은 모두 제거 ("다시 만나서 반가워요" 류는 모드 라벨과 의미 중복). 입구 화면의 1순위는 브랜드 인지 + 모드 인지.
  • 소셜은 Kakao 단독 — 한국어 작가 타깃, 사용자 결정. 카카오 공식 컬러 (#FEE500 노란색, #191919 검정 텍스트) + "K" 마크. Google/Apple은 V1 빼고 추후 사용 데이터 보고 결정.
  • 매직 링크 V1에서 제거 — *이메일로 일회용 로그인 링크를 받아 비밀번호 없이 로그인하는 방식 (Slack/Notion/Vercel 패턴)*. 한국 사용자에게 친숙하지 않다는 피드백 반영. Supabase Auth는 지원하니 V2 후보로 보존.
  • 이메일+비밀번호 메인 + Kakao 보조 — 카카오 버튼이 위, 구분선 아래 이메일 폼. 카카오가 더 빠른 경로지만, 비밀번호 폼이 *진짜 계정 소유 정서*라 둘 다 비중 있게.
  • 인사말은 italic prose 부제 — "지난번 멈춘 그 자리에서" / "메모를 잃지 않는 작업공간을 만들어요" — 데이터 입력 폼이 아닌 *책상 입구* 정서 유지 (홈의 "오늘은 어떤 글을 쓸까요?"와 일관).
  • 그림자 없음, hairline border만 — 디자인 시스템 일관. 폼 input·panel·구분선 모두 동일 규칙.
  • CTA는 ink pill (검정 알약) — accent blue 아님. Apple 패턴 — 색이 아니라 형태(pill)와 weight(600)로 강조.
