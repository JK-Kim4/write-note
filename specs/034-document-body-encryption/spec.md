# Feature Specification: 작품 본문 사용자별 봉투 암호화 (Document Body At-Rest Encryption)

**Feature Branch**: `034-document-body-encryption`

**Created**: 2026-06-23

**Status**: Draft

**Input**: User description: 작가의 창작 본문을 DB에 평문이 아닌 암호문으로 보관해 DB 유출 시에도 원문을 복호할 수 없게 한다. 사용자별 암호화 키를 발급하고, 집필 중 저장/로드 성능 저하가 없어야 한다.

## 배경 / 신뢰 경계 *(필수 컨텍스트)*

- **무엇을 보호하나**: 작가가 쓴 작품 본문(`documents.body`). 이는 작가 개인에게 민감한 미공개 창작물이다.
- **누구로부터 보호하나 (차단 대상)**: ① 작품 소유자가 아닌 다른 사용자, ② **데이터베이스 유출 공격자** — DB 덤프·백업·읽기 복제본을 통째로 탈취한 자.
- **누구를 신뢰하나 (범위 밖)**: 운영자 본인 — 애플리케이션 코드와 마스터 키와 DB에 모두 접근 가능한 자. 운영자는 정상 운영 중 본문을 복호할 수 있다(서버가 복호 가능한 봉투 암호화 방식). 운영자에게까지 본문을 숨기는 종단간 제로지식 방식은 본 기능 범위가 **아니다**.
- **핵심 불변식**: 본문을 복호하는 마스터 키는 **데이터베이스에 저장하지 않는다**. 따라서 데이터베이스 내용만으로는(키 없이는) 본문을 복호할 수 없다.

## Clarifications

### Session 2026-06-23

- Q: KEK(마스터키) 회전을 이번 범위에 포함? → A: 구조만 준비(키 버전 식별자 보유), 회전 흐름·재암호화는 후속 범위 밖.
- Q: 본문 복호 실패 시 관측 수준? → A: 에러 응답 + 서버 로그 + **디스코드 웹훅 알림**까지. 알림은 best-effort·비차단(웹훅 실패가 요청을 깨뜨리지 않음), 평문·키 미포함, 웹훅 URL은 환경변수 주입.
- Q: 성능 수용 기준? → A: 정량 목표 명시 — 암호화 도입으로 인한 본문 저장/로드의 서버 처리 추가 지연 p95 ≤ 5ms.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 투명한 본문 암호화 (작가 경험 무변화) (Priority: P1)

작가는 평소처럼 집필실에서 글을 쓰고 자동 저장되고 다시 불러온다. 작가가 체감하는 것은 아무것도 없다 — 저장도 불러오기도 글자 수 표시도 이전과 똑같이 동작한다. 하지만 그 본문은 데이터베이스에 **암호문 형태로만** 저장된다.

**Why this priority**: 본 기능의 본질. 작가 경험(특히 저장/로드 성능)을 한 치도 해치지 않으면서 본문을 암호화하는 것이 전체 목표다. 이 슬라이스만으로도 "본문이 DB에 평문으로 남지 않는다"는 핵심 가치가 달성된다.

**Independent Test**: 본문을 작성·저장한 뒤 (a) 에디터에서 다시 불러오면 작성한 내용이 그대로 보이고 글자 수가 맞고, (b) 데이터베이스의 해당 본문 레코드를 직접 조회하면 작성한 한국어 원문이 **읽히지 않는다**(암호문). (c) 저장/불러오기에 체감 가능한 지연이 추가되지 않는다.

**Acceptance Scenarios**:

1. **Given** 로그인한 작가가 집필실에서 본문을 작성, **When** 자동 저장이 일어남, **Then** 저장이 성공하고 글자 수가 정상 갱신되며, 데이터베이스에 저장된 본문은 원문이 보이지 않는 암호문이다.
2. **Given** 암호화되어 저장된 본문, **When** 작가가 같은 작품을 다시 열어 불러옴, **Then** 작성했던 원문이 손실 없이 그대로 표시된다.
3. **Given** 작가가 본문을 저장하는 중, **When** 동일 작품에 저장 충돌(버전 불일치)이 발생, **Then** 기존과 동일하게 충돌이 감지·처리된다(암호화 도입이 충돌 감지 동작을 바꾸지 않는다).
4. **Given** 빈 본문 또는 매우 긴 본문, **When** 저장·불러오기, **Then** 양쪽 모두 손실 없이 왕복한다.

---

### User Story 2 - DB 유출 시 본문 복호 불가 (Priority: P2)

데이터베이스(또는 그 백업/복제본)가 통째로 유출되어도, 공격자는 마스터 키가 없으므로 본문 원문을 복호할 수 없다.

**Why this priority**: 본 기능이 막으려는 위협 그 자체. P1으로 본문이 암호문화되면 본 보장은 키를 DB 밖에 두는 설계로 따라온다.

**Independent Test**: 데이터베이스 내용 전체(본문 암호문 + 저장된 사용자 키 자료)만 주어지고 마스터 키가 없는 상태에서 본문 복호를 시도하면 복호에 실패한다(원문 0건 노출).

**Acceptance Scenarios**:

1. **Given** 본문 암호문과 데이터베이스에 저장된 사용자 키 자료, **When** 마스터 키 없이 복호 시도, **Then** 복호가 불가능하다(원문을 얻을 수 없다).
2. **Given** 두 명의 서로 다른 작가의 본문, **When** 한 작가의 데이터 암호화 키가 노출, **Then** 다른 작가의 본문은 여전히 복호되지 않는다(사용자 간 키 격리).
3. **Given** 데이터베이스에서 누군가 본문 암호문을 변조, **When** 복호 시도, **Then** 변조가 감지되어 복호가 실패하고 손상된 본문을 작가에게 내보내지 않는다(fail-closed).

---

### User Story 3 - 기존 평문 본문 무중단 전환 (Priority: P3)

기능 배포 시점에 이미 평문으로 저장돼 있던 작품 본문도 깨지지 않고 정상적으로 열린다. 작가가 그 작품을 다음에 저장하는 순간 자동으로 암호문으로 전환된다. 일괄 변환(다운타임/부하 급증)은 발생하지 않는다.

**Why this priority**: 운영 중인 데이터를 안전하게 전환하기 위한 호환 장치. 신규 데이터 암호화(P1) 위에 얹히는 마이그레이션 안전망이다.

**Independent Test**: 평문 상태의 기존 본문 레코드를 (a) 불러오면 정상 표시되고, (b) 저장하면 그 이후로는 데이터베이스에 암호문으로 남는다. 배포 시 전체 본문을 한꺼번에 변환하는 작업이 없다.

**Acceptance Scenarios**:

1. **Given** 기능 배포 전부터 평문으로 저장돼 있던 본문, **When** 작가가 그 작품을 열어 불러옴, **Then** 원문이 정상 표시된다(레거시 평문 읽기 호환).
2. **Given** 평문 레거시 본문, **When** 작가가 수정 후 저장, **Then** 데이터베이스에는 암호문으로 저장된다.
3. **Given** 배포 직후, **When** 시스템이 정상 가동, **Then** 기존 본문을 일괄 변환하는 대량 작업이 트리거되지 않는다(작가가 건드리지 않은 작품은 다음 저장 전까지 평문으로 남아도 무방).

---

### Edge Cases

- **마스터 키 부재/오설정**: 서버에 마스터 키가 주입되지 않았거나 잘못된 경우 → 본문 복호/암호화가 실패하며, 평문을 우회 노출하거나 본문을 손상시키지 않는다(저장/로드는 명확한 오류로 실패, fail-closed).
- **복호 실패(변조·키 불일치)**: 본문을 평문으로 대체하거나 빈 본문으로 덮어쓰지 않고 오류로 처리하며, 서버 로그 + 디스코드 웹훅 알림(best-effort)을 남긴다(FR-014).
- **디스코드 웹훅 호출 실패/미설정**: 알림 전송이 실패하거나 웹훅 URL이 설정되지 않아도 본문 로드/저장 요청은 정상 처리된다(알림은 비차단·선택적).
- **데이터 암호화 키가 없는 사용자**: 기존 사용자(배포 전 가입자)는 키 자료가 없을 수 있음 → 본문 저장/암호화 시점까지 키가 없으면 자동 발급된다(작가 개입 없음).
- **빈 본문 / 초대형 본문**: 둘 다 손실 없이 암복호 왕복.
- **저장 충돌(409)과 암호화의 상호작용**: 버전 불일치 충돌 응답에 포함되는 "서버 현재 본문"도 복호되어 평문으로 전달된다(기존 충돌 UX 유지).
- **내보내기(export)**: 클라이언트가 구조화해 전송하므로 서버가 본문 평문을 별도로 다루지 않음 → 영향 없음(회귀 점검 대상).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 작품 본문(`documents.body`)을 데이터베이스에 **암호문 형태로** 저장해야 한다. 신규 저장되는 본문은 평문으로 데이터베이스에 남아서는 안 된다.
- **FR-002**: 시스템은 **사용자별로 고유한 데이터 암호화 키**를 사용해 본문을 암호화해야 한다(한 사용자의 키로 다른 사용자의 본문이 복호되지 않는다).
- **FR-003**: 본문을 복호하는 데 필요한 **마스터 키는 데이터베이스에 저장되지 않아야** 한다(런타임에 외부에서 주입). 데이터베이스 내용만으로는 본문을 복호할 수 없어야 한다.
- **FR-004**: 사용자별 데이터 암호화 키는 데이터베이스에 **마스터 키로 감싼(암호화된) 형태로만** 저장되어야 하며, 평문 키가 데이터베이스에 남아서는 안 된다.
- **FR-005**: 사용자별 데이터 암호화 키는 **가입 시 발급**된다. 키가 아직 없는 기존 사용자의 경우, 본문 저장 시점에 자동 발급되어 작가의 개입이 필요 없어야 한다.
- **FR-006**: 본문 불러오기(GET)와 저장(PUT)의 **클라이언트 API 계약(요청/응답 형태)은 변경되지 않아야** 한다. 암복호는 서버 내부에서만 일어난다.
- **FR-007**: 글자 수(word count) 계산은 **기존과 동일하게** 동작해야 한다(작가가 보는 글자 수가 달라지지 않는다).
- **FR-008**: 본문 저장 시 기존 **낙관적 동시성(버전 토큰) 및 저장 충돌 감지·응답 동작이 보존**되어야 한다.
- **FR-009**: 본문 복호가 실패하는 경우(키 부재/불일치, 암호문 변조 등) 시스템은 **fail-closed** — 평문을 우회 노출하거나 손상/빈 본문으로 덮어쓰지 않고 명확한 오류로 처리해야 한다.
- **FR-010**: 기능 배포 시점에 **이미 평문으로 저장된 본문도 정상적으로 읽혀야** 하며(레거시 호환), 해당 본문은 다음 저장 시 암호문으로 전환되어야 한다.
- **FR-011**: 배포 시 기존 본문을 **일괄 변환하는 대량 작업이 강제되지 않아야** 한다(무중단 점진 전환). 전수 전환은 선택적 후속 작업으로 분리한다.
- **FR-012**: 본 기능의 암호화 범위는 **작품 본문(`documents.body`)에 한정**된다. 제목·메모(곁쪽지)·인물 노트·줄거리·세계관 등 다른 텍스트는 본 기능에서 평문 유지하며 동작이 바뀌지 않아야 한다.
- **FR-013**: 사용자 데이터 암호화 키 레코드는 **키 버전 식별자**를 보유해 향후 마스터 키(KEK) 회전을 구조적으로 지원해야 한다. 단 **회전 실행 흐름(다중 KEK 버전 운용·DEK 재암호화·운영 명령)은 본 기능 범위 밖**(후속 작업)이며, 본 범위에서는 단일 활성 KEK만 사용한다.
- **FR-014**: 본문 복호 실패(FR-009) 발생 시 시스템은 ① 명확한 오류 응답, ② 원인 서버 로그(평문·키 자료 미포함), ③ **디스코드 웹훅을 통한 운영자 알림**을 수행해야 한다. 알림은 **best-effort·비차단** — 웹훅 호출 실패가 본문 로드/저장 요청 처리를 깨뜨려서는 안 되며, 알림 본문에 평문·키를 포함하지 않는다. 웹훅 대상 주소는 외부(환경변수) 주입이며 미설정 시 알림은 조용히 생략(요청은 정상 처리).

### Key Entities *(include if feature involves data)*

- **사용자 데이터 암호화 키 레코드(User Data Encryption Key)**: 사용자별 1개. 사용자가 작성한 본문을 암복호하는 데 쓰이는 키를, 마스터 키로 감싼(암호화된) 형태로 보관한다. 키 회전 추적용 버전 정보, 사용자 참조, 생성 시각을 가진다. 평문 키는 보관하지 않는다.
- **본문 암호문 봉투(Encrypted Body Envelope)**: 작품 본문의 저장 형태. 복호에 필요한 부속(형식 버전 식별자, 1회용 난수, 암호문 본체, 변조 감지 정보)을 하나로 묶은 묶음. 형식 버전 식별자로 "레거시 평문 / 암호화본"을 구분해 한 저장 위치에서 공존시킨다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 암호화 도입으로 인한 본문 저장/로드의 **서버 처리 추가 지연이 p95 ≤ 5ms** 이며(동일 본문 기준 도입 전후 비교), 작가는 저장·불러오기 속도 차이를 체감하지 못한다.
- **SC-002**: 신규로 저장된 본문을 데이터베이스에서 직접 조회했을 때 **작성 원문이 그대로 읽히는 경우가 0건**이다(전수 암호문).
- **SC-003**: 마스터 키 없이 데이터베이스 내용만으로 본문을 복호하려는 시도는 **항상 실패**한다(원문 노출 0).
- **SC-004**: 본문을 작성→저장→불러오기 했을 때 원문이 **100% 손실 없이 왕복**한다(암호화 도입으로 인한 데이터 손상 0건).
- **SC-005**: 기능 배포 후 기존(평문) 작품을 여는 작가의 **본문 로드 성공률 100%**(레거시 호환 회귀 0).
- **SC-006**: 글자 수 표시, 저장 충돌 감지, 내보내기 등 **기존 동작의 회귀가 0건**이다.
- **SC-007**: 본문 복호 실패가 발생하면 **원문/빈 본문이 작가에게 노출되는 일이 0건**이고(전부 오류 처리), 운영자는 디스코드 알림으로 해당 사건을 인지할 수 있다(웹훅 설정 시).

## Assumptions

- 본문은 데이터베이스에서 검색·정렬·인덱스 대상이 아니다(실측 확인) — 따라서 암호화로 인해 깨질 쿼리 기능이 본문에는 없다.
- 마스터 키는 기존 시스템 비밀값(예: 토큰 서명 키)과 동일한 경로(런타임 환경변수 주입)로 안전하게 공급될 수 있다.
- 마스터 키는 데이터베이스 백업과 **분리 보관**된다(둘이 함께 유출되면 보호가 무력화되므로 운영 수칙으로 관리). — 본 기능 외 운영 책임.
- 전송 구간 보호(평문이 작가↔서버로 오가는 구간)는 기존 HTTPS에 의존한다.
- 운영 중 애플리케이션 메모리 침해(실행 중 평문/키 노출)는 본 기능의 위협 모델 범위 밖이다(운영자 신뢰 모델).
- 기존 사용자(배포 전 가입자)의 데이터 암호화 키는 배포 후 첫 본문 저장 시점에 지연 발급되어도 무방하다.
- 복호 실패 운영 알림은 디스코드 Incoming Webhook으로 전송하며, 웹훅 URL은 환경변수로 주입된다(미설정 환경에서는 알림 없이 동작). 알림 호출은 요청 처리 경로를 차단하지 않는다.
- 본 기능은 작가에게 새로운 화면·설정·조작을 추가하지 않는다(투명한 at-rest 암호화).
