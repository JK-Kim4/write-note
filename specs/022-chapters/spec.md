# Feature Specification: 챕터(Chapter) — 작품 1:N 본문 구조

**Feature Branch**: `022-chapters`

**Created**: 2026-06-13

**Status**: Draft

**Input**: Round 2.5 챕터 기능 — 작품 1:N 본문 구조. 설계 SoT = `docs/superpowers/specs/2026-06-11-chapters-design.ko.md`, 계획 = `docs/plan/04-web-launch-v1-plan.md §Round 2.5`, GitHub 이슈 #58~#62.

## 배경

현재 한 작품(project)은 본문(document) 1개만 가진다. 장편을 쓰는 작가는 글이 길어지면 장(章) 단위로 나눠 관리하고 싶어 하지만, 지금은 모든 내용을 단일 본문에 욱여넣어야 한다. 본 기능은 작품을 **여러 챕터로 나눠 쓰고, 순서를 조절하고, 챕터를 지웠다 되돌릴 수 있게** 한다. (export 시 챕터를 골라 합치는 기능은 다음 라운드인 Round 3 export 의 선행 토대다.)

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 작품을 여러 챕터로 나눠 집필 (Priority: P1)

작가는 한 작품 안에서 새 챕터를 만들고, 챕터 사이를 오가며 각각을 독립된 원고처럼 쓴다. 기존에 쓰던 본문은 그대로 "1번 챕터"가 되어 한 글자도 잃지 않는다. 집필실을 떠났다 돌아와도(새로고침 포함) 마지막에 보던 챕터로 복귀한다.

**Why this priority**: 1:N 구조의 본질. 이것만 있어도 "장편을 장 단위로 나눠 쓴다"는 핵심 가치가 성립하는 MVP다. 나머지 스토리(순서·삭제·집계)는 이 위에 얹힌다.

**Independent Test**: 작품 하나를 열어 새 챕터 2~3개를 만들고, 챕터를 오가며 각각에 다른 글을 쓴 뒤, 집필실을 나갔다 다시 들어와 같은 챕터·같은 내용이 보이는지로 검증.

**Acceptance Scenarios**:

1. **Given** 본문이 있던 기존 작품, **When** 챕터 구조로 전환된 뒤 집필실을 연다, **Then** 기존 본문이 1번 챕터로 그대로 보이고 내용 손실이 없다.
2. **Given** 집필실에서 챕터 목록을 본다, **When** "새 챕터"를 추가한다, **Then** 목록 맨 뒤에 새 챕터가 생기고 거기서 바로 쓸 수 있다.
3. **Given** 여러 챕터가 있는 작품, **When** 다른 챕터를 선택한다, **Then** 에디터가 그 챕터 본문으로 바뀌고 현재 챕터 표시가 갱신된다.
4. **Given** 어떤 챕터에서 글을 쓰던 중, **When** 다른 챕터로 전환한다, **Then** 직전 챕터에 쓰던 내용이 유실 없이 저장된다(한국어 IME 조합 중 전환 포함).
5. **Given** 2번 챕터를 보다가 집필실을 떠났다, **When** 같은 작품으로 재진입(또는 새로고침)한다, **Then** 2번 챕터로 복귀한다.

---

### User Story 2 - 챕터 순서 조절 (Priority: P2)

작가는 챕터 목록에서 각 챕터를 위/아래로 옮겨 작품 안 순서를 바꾼다. 바뀐 순서는 영속되며 다음 진입 시에도 유지된다.

**Why this priority**: 장 순서는 집필 중 자주 바뀐다. 다만 챕터를 만들고 쓰는 것(US1)이 먼저 동작해야 의미가 있으므로 P2.

**Independent Test**: 챕터 3개를 만든 뒤 2번을 맨 위로 올리고, 재진입 후에도 그 순서가 유지되는지로 검증.

**Acceptance Scenarios**:

1. **Given** 챕터 3개가 순서대로 있다, **When** 가운데 챕터의 "위로"를 누른다, **Then** 그 챕터가 한 칸 위로 이동하고 목록이 즉시 갱신된다.
2. **Given** 순서를 바꾼 뒤, **When** 집필실을 나갔다 다시 들어온다, **Then** 바뀐 순서가 그대로 유지된다.
3. **Given** 맨 위 챕터, **When** "위로"가 불가능한 위치다, **Then** 위로 이동이 비활성(또는 무효)이다.

---

### User Story 3 - 챕터 삭제 · 되돌리기 (Priority: P2)

작가는 더 이상 필요 없는 챕터를 지운다. 실수로 지워도 되돌리기로 즉시 복구할 수 있다. 단, 작품에는 항상 챕터가 최소 하나 남아야 하므로 마지막 챕터는 지울 수 없다.

**Why this priority**: 삭제는 핵심 편집 동작이지만, 안전장치(되돌리기·마지막 챕터 보호)가 함께 가야 한다. US1 위에서 동작하므로 P2.

**Independent Test**: 챕터 2개 중 하나를 지우고 되돌리기로 복구, 그리고 챕터가 1개만 남았을 때 삭제가 막히는지로 검증.

**Acceptance Scenarios**:

1. **Given** 챕터 여러 개, **When** 한 챕터의 삭제를 누른다, **Then** 목록에서 즉시 사라지고 "되돌리기"를 제안하는 안내가 잠시 표시된다.
2. **Given** 방금 삭제한 직후, **When** "되돌리기"를 누른다, **Then** 챕터가 본문 그대로 복구된다.
3. **Given** 활성 챕터가 1개만 남은 작품, **When** 그 챕터를 삭제하려 한다, **Then** 삭제가 거부되고 삭제 수단이 비활성으로 보인다.
4. **Given** 지금 보고 있는 챕터를 삭제한다, **When** 삭제가 처리된다, **Then** 바로 앞 챕터(맨 앞이면 바로 다음)로 자동 전환되어 빈 화면이 남지 않는다.

---

### User Story 4 - 대시보드 작품 카드의 챕터 합산 반영 (Priority: P3)

작가가 홈(작품 벽/대시보드)에서 작품 카드를 볼 때, 글자수·마지막 수정 시각·마지막 문장이 여러 챕터를 합쳐 정확히 보인다.

**Why this priority**: 표시 정확성 문제. 챕터 도입으로 단일 본문 가정이 깨지면 카드 수치가 틀어지므로 반드시 함께 가야 하지만, 집필 자체(US1~3)보다는 후순위 P3.

**Independent Test**: 챕터 2개에 각각 글을 쓴 뒤 대시보드 카드의 글자수가 두 챕터 합과 일치하고, 마지막 문장이 가장 최근 수정한 챕터에서 오는지로 검증.

**Acceptance Scenarios**:

1. **Given** 활성 챕터 여러 개에 글이 있다, **When** 대시보드 작품 카드를 본다, **Then** 글자수가 모든 활성 챕터 글자수의 합과 같다.
2. **Given** 챕터마다 수정 시각이 다르다, **When** 카드를 본다, **Then** 표시 수정 시각이 활성 챕터 중 가장 최신이고, 마지막 문장이 그 최신 챕터에서 온다.
3. **Given** 삭제된(soft-delete) 챕터가 있다, **When** 카드 집계를 본다, **Then** 삭제된 챕터는 글자수·수정시각 집계에서 제외된다.

---

### Edge Cases

- **마지막 활성 챕터 삭제 시도**: 화면에서 삭제 수단이 비활성이어야 하고, 우회 요청(직접 호출)도 서버가 거부해야 한다(이중 방어).
- **현재 보는 챕터 삭제**: 빈 화면이 남지 않도록 인접 챕터로 자동 전환.
- **IME 조합 중 챕터 전환**: 한국어 조합 중 전환해도 작성 자모/문장이 유실되지 않아야 한다(016 자동저장 재설계의 회귀 영역).
- **삭제된 챕터로의 직접 접근/저장**: 삭제 표시된 챕터에 대한 본문 저장은 거부(찾을 수 없음 취급).
- **순서 변경의 잘못된 입력**: 누락·중복·다른 작품 소속 id 가 섞인 순서 요청은 거부.
- **기존 1:1 본문 이관**: 마이그레이션 시 기존 본문이 정확히 1개의 1번 챕터로 편입되고 중복 생성되지 않아야 한다.
- **A형/B형 집필실 정합**: 두 집필실 모두에서 챕터 목록·전환·삭제가 동일하게 동작해야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 한 작품은 여러 챕터를 가질 수 있다(작품 1 : 챕터 N).
- **FR-002**: 각 작품은 항상 **활성 챕터를 최소 1개** 가진다(불변식). 작품 생성 시 챕터 1개가 자동으로 함께 만들어진다.
- **FR-003**: 기존(챕터 도입 전) 작품의 본문은 **손실 없이** 그 작품의 1번 챕터가 된다.
- **FR-004**: 작가는 작품 안에 새 챕터를 추가할 수 있고, 새 챕터는 순서상 맨 뒤에 놓인다. 새 챕터의 기본 제목은 `새 챕터`(고정 문구)이며 작가가 언제든 수정할 수 있다.
- **FR-005**: 각 챕터는 제목을 가진다(기존 본문 제목 규칙 재사용).
- **FR-006**: 작가는 챕터 사이를 전환할 수 있으며, 재진입·새로고침 시 마지막으로 보던 챕터로 복귀한다. 명시적 선택이 없으면 가장 최근 수정한 활성 챕터로 진입한다.
- **FR-007**: 챕터 전환 시 직전 챕터에 작성 중이던 내용이 유실되지 않는다(한국어 IME 조합 중 전환 포함).
- **FR-008**: 챕터별로 자동저장·초안 복구·저장 충돌 방지가 서로 독립적으로 동작한다.
- **FR-009**: 작가는 챕터 순서를 조절할 수 있고(위/아래 이동), 변경된 순서는 영속된다.
- **FR-010**: 작가는 챕터를 삭제할 수 있다. 삭제는 **복구 가능(soft-delete)** 하며, 삭제 직후 되돌리기를 제공한다.
- **FR-011**: **마지막 활성 챕터는 삭제할 수 없다.** 화면 수단 비활성 + 서버 거부의 이중 방어.
- **FR-012**: 지금 보고 있는 챕터를 삭제하면 **바로 앞 챕터**(맨 앞 챕터였다면 **바로 다음 챕터**)로 자동 전환된다 — 빈 화면이 남지 않는다.
- **FR-013**: 삭제 표시된 챕터는 목록·집계에서 제외되며, 그 챕터에 대한 본문 저장은 거부된다.
- **FR-014**: 대시보드 작품 카드의 글자수는 활성 챕터 글자수의 합, 표시 수정 시각은 활성 챕터 중 최신, 마지막 문장은 가장 최근 수정한 활성 챕터에서 온다. 삭제된 챕터는 제외된다.
- **FR-015**: 챕터 목록·전환·순서 변경·삭제·되돌리기는 A형 집필실과 B형 집필실 **양쪽에서** 동일하게 동작한다.
- **FR-016**: 챕터는 자기 작품에만 속한다 — 다른 작품의 챕터를 가져오거나 공유할 수 없다.

### Key Entities *(include if feature involves data)*

- **챕터(Chapter)**: 작품에 속한 하나의 본문 단위. 속성 — 제목, 작품 안 순서, 본문, 글자수, 마지막 수정 시각, 삭제 표시(복구 가능 여부). 기존 "본문(document)" 개념을 1:N 으로 확장한 것 — 챕터 1개 = 본문 1개. 작품과 N:1.
- **작품(Project)**: 여러 챕터를 가진다. 작품 카드의 집계 값(글자수·수정시각·마지막 문장)은 그 작품의 활성 챕터들에서 파생된다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 작가는 집필실을 떠나지 않고(같은 화면 안에서) 새 챕터를 만들고 챕터 사이를 전환할 수 있다.
- **SC-002**: 챕터 도입 마이그레이션 후 기존 작품의 본문 손실은 **0건** — 모든 기존 본문이 정확히 1개의 1번 챕터로 보존된다.
- **SC-003**: 챕터 전환 시 작성 중 내용 유실 **0건**(한국어 IME 조합 중 전환 4케이스 포함).
- **SC-004**: 마지막 활성 챕터 삭제 시도는 **100% 차단**된다(화면·서버 양쪽).
- **SC-005**: 삭제한 챕터는 되돌리기로 본문 그대로 복구된다.
- **SC-006**: 대시보드 작품 카드의 글자수가 해당 작품 모든 활성 챕터 글자수의 합과 **정확히 일치**한다.
- **SC-007**: 위 모든 동작이 A형·B형 집필실에서 동일하게 검증된다.

## Assumptions

- **데이터 구조**: 신규 테이블을 만들지 않고 기존 본문(document) 테이블을 1:N 으로 확장한다(설계 안 A). 챕터 1개 = 본문 1행.
- **자동저장 세션 재사용**: 챕터별 자동저장·초안 복구·충돌 방지는 016 자동저장 재설계의 문서 id 단위 세션을 그대로 재사용한다(추가 설계 없음).
- **집필실 골격 재사용**: 017 3단 집필실 골격과 좌측 패널 접기 토글은 불변이며, 챕터 목록은 좌측 패널에 얹힌다.
- **운영 DB 적용 분리**: 본 작업의 스키마 변경은 **로컬 개발 DB 에만** 적용한다. 운영(Supabase) 적용은 Round 4 운영 인프라에서 일괄 처리한다(코드-우선 전략).
- **export 위임**: export 시 챕터를 골라 합치는 UI 는 본 범위가 아니며 Round 3 export 가 본 데이터 구조를 토대로 구현한다.
- **순서 변경 방식**: 드래그 앤 드롭이 아니라 위/아래 버튼으로 시작한다(드래그는 후속 후보).
- **계층 없음**: 부/권 같은 챕터 묶음 상위 계층은 범위 밖 — 단일 평면 목록.

## Dependencies

- **선행**: Round 2(집필실 기능, 020) 완료 — 현재 develop 에 merge 됨.
- **후행**: Round 3 export(#42~#44, #51)가 본 챕터 합본 구조에 의존 — 본 라운드가 export 의 선행 필수.
- **의존 순서(이슈)**: #58(데이터 구조) → #59(챕터 동작)·#60(카드 집계 재설계) → #61(집필실 목록·전환) → #62(삭제·되돌리기·마지막 챕터 가드).
