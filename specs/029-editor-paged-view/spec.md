# Feature Specification: 집필실 에디터 페이지 넘김 뷰

**Feature Branch**: `develop` (전용 브랜치 미생성 — 사용자 지시로 develop 직접 작업)

**Created**: 2026-06-20

**Status**: Draft

**Input**: 집필실 자체 에디터를 연속 세로 스크롤에서 "한 화면에 한 페이지씩" 페이지 넘김 뷰로 완전 대체. 좌/우 `< >` + 키보드로 페이지 이동, 캐럿이 흘러가면 자동 페이지 전환.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 한 화면에 한 페이지만 보며 글을 쓴다 (Priority: P1)

작가가 집필실에서 글을 쓸 때, 페이지가 화면 아래로 끝없이 늘어나 화면을 벗어나는 대신, **항상 한 화면에 한 페이지(원고지/종이 한 장)** 만 보인다. 글이 길어져 현재 페이지를 넘기면 자동으로 다음 페이지가 화면에 나타난다.

**Why this priority**: 현재 연속 스크롤은 페이지가 아래로 늘어나 "지금 몇 쪽째인지", "이 페이지에 얼마나 남았는지"를 가늠하기 어렵고 화면을 벗어난다는 불편이 dogfooding에서 보고됐다. 종이 한 장 단위로 보는 것이 본 도구(원고지 집필)의 핵심 경험이다.

**Independent Test**: 집필실에 들어가 한 페이지를 채울 만큼 글을 쓰고, 화면이 아래로 늘어나지 않고 다음 페이지로 넘어가는지(이전 페이지가 화면에서 사라지는지) 확인.

**Acceptance Scenarios**:

1. **Given** 작가가 집필실을 연다, **When** 화면을 본다, **Then** 한 화면에 한 페이지만 보이고 그 아래로 다른 페이지가 이어 붙어 늘어나지 않는다.
2. **Given** 현재 페이지를 거의 채웠다, **When** 계속 입력해 내용이 다음 페이지로 넘어간다, **Then** 자동으로 다음 페이지가 화면에 나타나고 캐럿(입력 위치)이 보이는 페이지 안에 있다.
3. **Given** 줌인해서 페이지가 화면보다 커졌다, **When** 스크롤한다, **Then** 그 페이지 안에서만 스크롤되고 옆/다음 페이지로는 스크롤로 넘어가지 않는다.

---

### User Story 2 - `< >` 와 키보드로 페이지를 넘긴다 (Priority: P1)

작가가 화면 좌/우 가장자리의 `<` `>` 버튼이나 키보드로 페이지를 앞뒤로 넘겨본다. 지금 몇 쪽 중 몇 쪽인지 표시된다.

**Why this priority**: P1(단일 페이지 표시)와 한 쌍 — 한 페이지만 보이면 페이지 간 이동 수단이 반드시 필요하다.

**Independent Test**: 여러 페이지 분량의 글에서 `< >` 와 PageUp/PageDown으로 앞뒤 페이지가 바뀌고, 첫/끝 페이지에서 더 못 넘어가며, 현재 위치 표시가 맞는지 확인.

**Acceptance Scenarios**:

1. **Given** 3쪽 분량의 글에서 1쪽을 보고 있다, **When** `>` 를 누른다(또는 PageDown), **Then** 2쪽이 보이고 위치 표시가 "2 / 3"이 된다.
2. **Given** 첫 페이지를 보고 있다, **When** `<` 를 본다, **Then** `<` 는 비활성(더 이전 없음)이다. 마지막 페이지에서 `>` 도 동일.
3. **Given** `>` 로 다음 페이지를 봤다(캐럿은 이전 페이지에 그대로), **When** 보이는 페이지의 한 지점을 클릭한다, **Then** 캐럿이 그 위치로 이동하고 이후 입력은 그 페이지에서 이어진다.

---

### User Story 3 - 목차에서 해당 페이지로 점프한다 (Priority: P2)

작가가 목차(아웃라인)의 제목을 클릭하면 그 제목이 있는 페이지로 전환되고 캐럿이 그 위치로 이동한다.

**Why this priority**: 페이지 넘김 모드에서도 기존 목차 점프가 깨지지 않아야 한다(무회귀). 단일 페이지 뷰에 맞춰 "스크롤"이 아닌 "페이지 전환"으로 동작해야 한다.

**Independent Test**: 여러 페이지에 걸친 제목들에서 목차 항목 클릭 시 해당 페이지로 바뀌고 캐럿이 그 제목으로 가는지 확인.

**Acceptance Scenarios**:

1. **Given** 3쪽에 있는 제목이 목차에 있다, **When** 그 항목을 클릭한다, **Then** 화면이 3쪽으로 전환되고 캐럿이 그 제목 위치로 간다.

---

### Edge Cases

- **페이지 경계를 넘는 드래그 선택**: 단일 페이지만 보이므로 드래그 선택은 **현재 페이지 안에서만** 가능하다(v1). 여러 페이지를 가로지르는 드래그 선택은 지원하지 않으며, 전체 선택은 ⌘A로 한다. 선택 하이라이트는 보이는 페이지 부분만 표시된다.
- **캐럿이 보이지 않는 페이지에 있을 때 입력**: `< >` 로 캐럿과 다른 페이지를 보던 중 타이핑하면, 화면이 캐럿이 있는 페이지로 자동 전환된다(작성분이 안 보이는 곳에 쌓이지 않게).
- **삭제로 페이지 수가 줄어 현재 페이지가 사라질 때**: 현재 페이지 번호가 마지막 페이지를 넘으면 마지막 페이지로 보정한다(빈 화면 금지).
- **한글 IME 조합 중 페이지 경계**: 조합 중 캐럿/페이지 전환이 조합을 깨지 않아야 한다(자체 에디터 한글 입력 회귀 4케이스 적용).
- **모바일(축소 렌더)·데스크탑(줌) 양쪽**에서 단일 페이지 표시·넘김이 동일하게 동작해야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 집필실 에디터는 한 화면에 **한 페이지만** 표시해야 하며, 페이지들이 세로로 이어 붙어 화면 아래로 늘어나서는 안 된다.
- **FR-002**: 입력으로 내용이 현재 페이지를 넘치면 시스템은 자동으로 다음 페이지를 표시하고 캐럿이 보이는 페이지 안에 있게 해야 한다.
- **FR-003**: 작가는 화면 좌/우의 `< >` 컨트롤과 키보드(PageUp/PageDown)로 이전/다음 페이지로 이동할 수 있어야 한다.
- **FR-004**: 첫 페이지에서 "이전", 마지막 페이지에서 "다음"은 비활성(더 이동 없음)이어야 한다.
- **FR-005**: 화면에 현재 페이지 위치("현재 / 전체")가 표시되어야 한다.
- **FR-006**: `< >` 로 페이지를 넘기는 것은 보기 이동이며 캐럿을 옮기지 않는다. 보이는 페이지의 한 지점을 클릭하면 캐럿이 그 위치로 이동해야 한다.
- **FR-007**: 작가가 캐럿이 없는 페이지를 보던 중 입력하면, 화면이 캐럿이 있는 페이지로 자동 전환되어야 한다.
- **FR-008**: 드래그 텍스트 선택은 현재 보이는 페이지 안에서만 동작해야 한다(v1). 전체 선택(⌘A)은 유지되며, 선택 하이라이트는 보이는 페이지 부분만 표시된다.
- **FR-009**: 목차(아웃라인) 항목 클릭 시 해당 제목이 있는 페이지로 전환되고 캐럿이 그 위치로 이동해야 한다.
- **FR-010**: 줌인으로 페이지가 화면보다 커지면 그 페이지 안에서만 스크롤되고, 스크롤로 다른 페이지로 넘어가지 않아야 한다.
- **FR-011**: 삭제 등으로 전체 페이지 수가 줄어 현재 페이지 인덱스가 범위를 벗어나면 마지막 유효 페이지로 보정되어야 한다.
- **FR-012**: 페이지 넘김 동작은 데스크탑·모바일 양쪽에서 동일하게 동작해야 하며, 한글 입력(IME 조합)을 깨뜨리지 않아야 한다.
- **FR-013**: 본 변경은 기존 페이지 분할·줄 측정·문서 모델·PDF 내보내기 결과를 바꾸지 않아야 한다(표시 방식만 변경).

### Key Entities *(include if feature involves data)*

- **현재 페이지(currentPage)**: 화면에 보이는 페이지 인덱스(0-기반). 저장하지 않는 일시적 뷰 상태. 캐럿 흐름·`< >`·목차 점프로 변한다. 전체 페이지 수 범위로 클램프.
- **페이지 목록(pages)**: 기존 페이지 분할 결과(분리된 페이지 배열). 본 기능은 이 배열을 새로 만들지 않고 **표시 대상만 1장으로 좁힌다**.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 한 페이지 분량 이상 입력해도 화면이 아래로 늘어나지 않고 한 화면에 한 페이지만 유지된다.
- **SC-002**: 작가가 `< >` 또는 키보드로 임의 페이지로 이동할 수 있고, 현재 위치가 항상 정확히 표시된다.
- **SC-003**: 입력이 다음 페이지로 넘어갈 때 작성 중인 글(캐럿)이 항상 화면에 보인다(보이지 않는 곳에 쌓이지 않는다).
- **SC-004**: 목차 점프·캐럿 이동·한글 입력·선택이 기존과 동일하게 동작한다(표시 방식 변경으로 인한 기능 회귀 0).
- **SC-005**: 페이지 분할 위치·줄바꿈·PDF 내보내기 결과가 본 변경 전후 동일하다.

## Assumptions

- 작업 브랜치는 신규 브랜치 없이 `develop`에서 직접 작업한다(사용자 명시 — speckit 브랜치 훅 건너뜀).
- 페이지 넘김 모드가 연속 스크롤을 **완전히 대체**한다(토글 미제공, v1).
- 페이지 경계를 넘는 드래그 선택은 v1 범위 밖(현재 페이지 내 선택 + ⌘A 전체선택만).
- 페이지 분할·측정·문서 모델·PDF 내보내기 로직은 변경하지 않으며, 표시(렌더) 계층만 단일 페이지로 바꾼다.
- 본 에디터는 전면 교체가 진행 중인 회귀 위험 영역이므로, 자동 테스트로 못 잡는 캐럿/선택/IME/모바일 정합은 dogfooding 게이트로 검증한다.
- "페이지를 화면에 맞춤"은 기본 동작이며, 작가가 줌을 키우면 페이지 내 스크롤이 허용된다.
