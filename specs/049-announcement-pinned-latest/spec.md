# Feature Specification: 공지사항 고정 슬롯 + 최신 슬롯 (Pinned + Latest Announcement)

**Feature Branch**: `049-announcement-pinned-latest`

**Created**: 2026-07-01

**Status**: Draft

**Input**: User description: "공지사항을 고도화 하는 작업을 진행할거야. 지금은 하나의 공지사항만 고객 메인에 노출되고있는데, 항상 고정할 공지 사항 1개, 최신 등록순으로 노출할 공지사항 1개를 구분해서 항상 고정할 공지사항은 보다 구분이 잘 식별되도록 추가하고싶어."

## 배경 (Context)

현재 고객 메인(홈) 상단에는 공지 배너가 **최신 공개 공지 1건**만 노출된다. 운영자 관리 도구에는 이미 "고정(pin)" 표시 기능과 데이터가 존재하지만, 그 고정 여부는 **메인 화면에 전혀 반영되지 않는다** — 즉 운영자가 어떤 공지를 고정해도 방문자 메인에서는 아무 변화가 없다.

이 기능은 메인에 **두 개의 구분된 슬롯**을 도입한다: (1) 항상 고정 공지 1건, (2) 최신 공지 1건. 그리고 고정 공지 슬롯을 최신 공지 슬롯과 **시각적으로 뚜렷이 구분**해 방문자가 "이건 고정 공지"임을 즉시 식별하게 한다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 고정 공지를 메인에서 구분해 본다 (Priority: P1)

방문자가 메인(홈)에 들어오면, 운영자가 "고정"으로 지정한 공지가 최신 공지와 **시각적으로 구분되는 별도 슬롯**에 노출되어, 그것이 상시 강조 공지임을 즉시 알아본다.

**Why this priority**: 사용자가 명시한 핵심 요구("항상 고정할 공지사항은 보다 구분이 잘 식별되도록"). 또한 현재 고정 지정이 메인에 아무 효과가 없는 상태를 실제로 동작하게 만드는, 이 기능의 본질 슬라이스다.

**Independent Test**: 운영 도구에서 한 공지를 공개+고정으로 설정 → 메인에서 그 공지가 고정 슬롯(구분되는 시각 처리)에 노출되는지 확인. 고정을 끄면 고정 슬롯에서 사라지는지 확인.

**Acceptance Scenarios**:

1. **Given** 공개+고정된 공지가 1건 있을 때, **When** 방문자가 메인에 진입하면, **Then** 그 공지가 최신 공지와 구분되는 고정 슬롯에 노출된다.
2. **Given** 공개+고정된 공지가 여러 건일 때, **When** 방문자가 메인에 진입하면, **Then** 그중 가장 최근에 공개된 1건만 고정 슬롯에 노출된다.
3. **Given** 공개+고정된 공지가 하나도 없을 때, **When** 방문자가 메인에 진입하면, **Then** 고정 슬롯은 표시되지 않는다.
4. **Given** 고정 공지가 있을 때, **When** 방문자가 그 공지를 클릭하면, **Then** 해당 공지 상세로 이동한다.

---

### User Story 2 - 고정과 별개로 최신 공지도 함께 본다 (Priority: P2)

방문자가 메인에서 고정 공지와 **별개로** 가장 최근 공개 공지 1건을 함께 본다. 단 고정 슬롯에 이미 노출된 공지와 같은 공지가 최신 슬롯에 **중복 노출되지는 않는다**.

**Why this priority**: 사용자가 요구한 "최신 등록순으로 노출할 공지사항 1개" 슬롯. 고정 슬롯(US1)만으로도 "구분 식별" 가치는 성립하므로 P2로 두되, 완전한 요구 충족을 위해 필요하다.

**Independent Test**: 공개 공지 2건(1건 고정, 1건 비고정) 상태에서 메인 진입 → 고정 슬롯엔 고정 공지, 최신 슬롯엔 비고정 최신 공지가 각각 노출되고 같은 공지가 두 번 나오지 않는지 확인.

**Acceptance Scenarios**:

1. **Given** 공개 공지가 여러 건(일부 고정)일 때, **When** 방문자가 메인에 진입하면, **Then** 최신 슬롯에는 고정 슬롯에 노출된 공지를 제외한 가장 최근 공개 공지 1건이 노출된다.
2. **Given** 가장 최근 공개 공지가 곧 고정 공지와 동일할 때, **When** 방문자가 메인에 진입하면, **Then** 최신 슬롯에는 그다음으로 최근인 비중복 공개 공지가 노출된다.
3. **Given** 공개 공지가 정확히 1건이고 그것이 고정일 때, **When** 방문자가 메인에 진입하면, **Then** 고정 슬롯만 노출되고 최신 슬롯은 표시되지 않는다.
4. **Given** 공개 공지가 하나도 없을 때, **When** 방문자가 메인에 진입하면, **Then** 두 슬롯 모두 표시되지 않는다.

---

### Edge Cases

- **고정 공지 = 최신 공개 공지**: 같은 공지를 두 슬롯에 중복 노출하지 않는다. 최신 슬롯은 고정 슬롯에 없는 그다음 최신 공개 공지를 노출하고, 없으면 최신 슬롯을 비운다.
- **고정 공지 다수**: 메인은 가장 최근 공개된 1건만 고정 슬롯에 노출한다(운영 도구의 다중 고정 자체는 막지 않는다).
- **고정 공지만 존재, 다른 공지 없음**: 고정 슬롯만 노출.
- **공개 공지 0건**: 두 슬롯 모두 미표시(빈 화면 깨짐 없이 정상).
- **고정 공지가 비공개로 전환됨**: 즉시(캐시 만료 후) 고정 슬롯에서 사라진다.
- **비공개(미게시) 공지**: 고정 여부와 무관하게 어느 슬롯에도 노출되지 않는다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 메인(홈)은 공개된 공지를 "고정 공지 슬롯" 1건과 "최신 공지 슬롯" 1건으로 구분해 노출해야 한다.
- **FR-002**: 고정 공지 슬롯에는 공개되고 고정된 공지 중 **가장 최근에 공개된 1건**을 노출해야 한다.
- **FR-003**: 고정된 공지가 여러 건 있어도 메인은 그중 1건만 고정 슬롯에 노출해야 한다. 시스템은 운영 도구에서의 다중 고정 지정 자체를 제한하지 않는다(표시 시점 선택 방식).
- **FR-004**: 최신 공지 슬롯에는 공개된 공지 중 가장 최근에 공개된 1건을 노출하되, 그 공지가 고정 공지 슬롯에 노출된 공지와 동일하면 **고정 슬롯에 없는 그다음 최근 공개 공지**를 노출해야 한다(같은 공지의 두 슬롯 중복 노출 금지).
- **FR-005**: 고정 공지 슬롯은 최신 공지 슬롯과 **시각적으로 뚜렷이 구분**되어, 방문자가 별도 안내 없이도 고정 공지임을 즉시 식별할 수 있어야 한다. 확정된 처리(2026-07-01 dogfooding 승인 — `docs/research/2026-07-01-announcement-pinned-latest-mockup.html` = 색·치수 SoT):
  - 세로 적층(고정 위, 최신 아래).
  - **최신 슬롯** = 기존 운영 배너와 동일한 옅은 **청록(teal) pill** + 인라인 「공지」 + 확성기 아이콘(색 유지).
  - **고정 슬롯(강조)** = **앰버/골드(amber)** 채운 배경 + 좌측 금빛 세로 바 + 채운 「고정」 배지(솔리드 amber·흰 글자·핀 아이콘) + 진한 제목. 최신(청록)과 **다른 색상(앰버)**이라 한눈에 갈린다.
  - 두 슬롯 모두 **고정 밝은 배경 + 고정 어두운 텍스트(teal-800 / amber-900)**라 라이트·다크 양쪽에서 가독성이 유지되어야 한다(테마 따라 뒤집히는 색 금지 — 다크에서 고정 제목이 안 보이던 버그의 근본 방지).
  - 배지 문구 = 고정 「고정」 / 최신 「공지」.
- **FR-011**: 최신(일반) 공지 배너 색은 **기존 운영 배너(teal)를 그대로 유지**한다. 신규 고정 슬롯만 대비되는 앰버/골드 강조를 추가한다(기존 사용자에게 최신 배너 색 변화 없음).
- **FR-006**: 공개되고 고정된 공지가 하나도 없으면 고정 공지 슬롯은 표시하지 않아야 한다.
- **FR-007**: 공개된 공지가 하나도 없으면 두 슬롯 모두 표시하지 않아야 한다.
- **FR-008**: 각 슬롯의 공지를 클릭하면 해당 공지 상세로 이동해야 한다(기존 공지 상세 경로 재사용).
- **FR-009**: 비공개(미게시) 공지는 고정 여부와 무관하게 어느 슬롯에도 노출되지 않아야 한다.
- **FR-010**: 메인이 고정 슬롯과 최신 슬롯을 구분 렌더할 수 있도록, 공개 조회는 각 공지가 고정 공지인지 식별할 수 있게 해야 한다. (노출 방식 — 요약 응답 필드 추가 vs 전용 조회 — 은 plan에서 결정한다.)

### Key Entities *(include if feature involves data)*

- **공지사항(Announcement)**: 운영자가 작성해 방문자에게 노출하는 안내 글. 주요 속성 = 제목, 본문, 공개 여부, **고정 여부**, 공개일, 등록일. 기존 도메인(030 운영 툴)을 재사용하며 **신규 필드는 추가하지 않는다** — 고정 여부·공개일·등록일이 이미 존재한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 운영자가 한 공지를 공개+고정으로 지정한 뒤, 방문자가 메인에서 그 공지를 고정 슬롯에서 확인할 수 있다(현재는 고정 지정이 메인에 반영되지 않음 — 반영률 0% → 100%).
- **SC-002**: 방문자는 메인에서 고정 공지와 최신 공지를 별도 안내 없이 시각적으로 구분해 인지할 수 있고, 같은 공지가 두 슬롯에 중복 노출되는 경우가 0건이다.
- **SC-003**: 고정 공지가 없거나 공지가 전혀 없는 경우에도 메인이 깨짐 없이 정상 표시된다(빈 슬롯으로 인한 레이아웃 결함 0건).
- **SC-004**: 두 슬롯을 채우기 위한 메인 진입 시 공지 노출이 기존과 동등하게 지체 없이(체감 1초 이내) 완료된다.

## Assumptions

- **"최신"의 기준 = 공개일**: 사용자 표현은 "최신 등록순"이나, 등록만 해두고 나중에 공개한 공지가 공개 시점에 "최신"으로 읽히는 것이 자연스럽고 현재 공개 목록 정렬(공개일 내림차순)과도 일치하므로, 최신 슬롯 및 고정 슬롯의 "가장 최근" 기준을 **공개일(공개로 전환된 시각)**로 해석한다. (등록일 기준을 원하면 plan 전에 정정.)
- **운영 도구는 변경하지 않는다**: 고정 토글과 다중 고정 허용은 기존 운영 도구 동작을 그대로 유지한다(표시 시점 1건 선택 방식으로 결정됨). 이번 작업은 운영자 관리 화면·백엔드 쓰기 로직을 변경하지 않는다.
- **시각 구분 디자인 확정(2026-07-01 목업 승인)**: 고정 슬롯의 구체 시각 처리(색·라벨·아이콘·배치)는 `docs/research/2026-07-01-announcement-pinned-latest-mockup.html`(사용자 승인)로 잠갔다 — FR-005 참조. plan은 이 목업을 색·치수 SoT로 삼아 앱 토큰(`frontend/src/styles/tokens.css`)에 매핑한다. "말풍선"은 확성기 아이콘 배너를 가리키며, 별도 버블(꼬리) 도형은 도입하지 않는다.
- **공개 조회/상세 경로 재사용**: 기존 공개 공지 목록·상세 및 공지 상세 페이지(/notice)를 재사용한다.
- **고정 여부 노출 방식은 plan에서 결정**: 메인이 고정/최신을 구분하려면 공개 조회가 고정 여부를 알 수 있어야 한다(현재 공개 요약 응답에는 고정 여부가 없음). 요약 응답에 필드를 추가할지, 두 슬롯을 한 번에 주는 전용 조회를 둘지는 plan에서 정한다.
- **데이터 모델 무변경**: 공지 도메인의 데이터 모델(제목/본문/공개/고정/공개일/등록일)은 이미 이 기능을 지원하므로 스키마 마이그레이션이나 신규 필드는 필요하지 않다.
