# Feature Specification: 운영 툴 (Admin Ops Tool) v1

**Feature Branch**: `030-admin-ops-tool`

**Created**: 2026-06-21

**Status**: Draft

**Input**: 설계 문서 `docs/superpowers/specs/2026-06-21-admin-ops-tool-design.md` 기반. 2026-06-21 웹 런칭 후 사용자 유입에 대비한 솔로 운영자용 경량 운영 툴. 별도 어드민 앱으로 공지 발행·회원 조회·사용 현황 통계를 제공하고, 문의는 외부 채널로 받는다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 공지사항 발행과 사용자 노출 (Priority: P1)

운영자는 서비스 업데이트·점검·안내를 사용자에게 알려야 한다. 운영자가 어드민에서 공지를 작성·발행하면, 사용자는 서비스 홈 상단 배너에서 최신 공지를 보고, 공지 목록 페이지에서 지난 공지 전체와 상세를 확인한다. 운영자는 공지를 수정·삭제하거나 비공개로 내릴 수 있다.

**Why this priority**: 런칭 직후 가장 먼저 필요한 운영 행위. 사용자와의 유일한 일방향 공식 소통 창구이며, 이것만 있어도 운영 툴로서 즉시 가치를 낸다.

**Independent Test**: 운영자가 공지 1건을 작성·발행 → 비로그인 사용자가 홈 배너와 공지 목록·상세에서 그 공지를 확인 → 운영자가 비공개 처리 → 사용자 화면에서 사라짐. 이 흐름만으로 독립 검증·배포 가능.

**Acceptance Scenarios**:

1. **Given** 발행된 공지가 없는 상태, **When** 사용자가 홈에 접속, **Then** 공지 배너가 표시되지 않는다.
2. **Given** 운영자가 공지를 작성·발행, **When** 사용자가 홈에 접속, **Then** 홈 상단 배너에 최신 공지가 보이고 클릭 시 상세로 이동한다.
3. **Given** 발행된 공지 여러 건, **When** 사용자가 공지 목록 페이지 진입, **Then** 최신순으로 공지 목록이 보이고 각 항목에서 상세를 열 수 있다.
4. **Given** 발행된 공지, **When** 운영자가 그 공지를 비공개로 변경, **Then** 사용자 배너·목록·상세에서 더 이상 보이지 않는다.
5. **Given** 비공개(미발행) 공지, **When** 사용자가 목록·배너를 확인, **Then** 해당 공지는 노출되지 않는다.

---

### User Story 2 - 회원 현황 조회 (Priority: P2)

운영자는 누가 가입했는지, 얼마나 활동하는지 파악해야 한다. 운영자는 어드민에서 가입자 목록을 최신순으로 보고, 이메일로 특정 회원을 검색하며, 한 회원의 상세(가입일, 마지막 로그인, 이메일 인증 여부, 카카오 연동 여부, 작품 수)를 확인한다. 조회 전용이며 회원 데이터를 수정·삭제하지 않는다.

**Why this priority**: 유입이 시작되면 사용자 파악이 필요하나, 공지처럼 즉시 필수는 아니다. 읽기 전용이라 위험이 낮다.

**Independent Test**: 운영자가 회원 목록 페이지에서 전체 가입자를 확인 → 특정 이메일로 검색 → 검색된 회원 상세를 열어 가입일·마지막 로그인 등을 확인. 공지 기능과 무관하게 독립 검증 가능.

**Acceptance Scenarios**:

1. **Given** 가입자가 여러 명, **When** 운영자가 회원 목록을 연다, **Then** 가입일 최신순으로 회원이 표시되고 페이지 단위로 넘길 수 있다.
2. **Given** 회원 목록, **When** 운영자가 이메일 일부로 검색, **Then** 일치하는 회원만 표시된다.
3. **Given** 특정 회원, **When** 운영자가 상세를 연다, **Then** 가입일·마지막 로그인·이메일 인증 여부·카카오 연동 여부·작품 수가 보인다.
4. **Given** 어떤 회원 화면이든, **When** 운영자가 상세를 확인, **Then** 비밀번호 등 인증 비밀값은 어디에도 표시되지 않는다.

---

### User Story 3 - 사용 현황 통계 대시보드 (Priority: P3)

운영자는 홍보 효과와 서비스 성장을 한눈에 보고 싶다. 운영자는 어드민 대시보드에서 핵심 카운트(총 가입자, 오늘·이번 주 신규 가입, 활성 사용자, 총 작품 수)와 최근 30일 일별 가입 추이 그래프를 본다.

**Why this priority**: 의사결정에 유용하나 운영 행위(공지·회원 파악) 다음 순위. 기존 데이터 집계라 추가 데이터 수집이 필요 없다.

**Independent Test**: 운영자가 대시보드 진입 → 카운트 카드 수치가 실제 데이터와 일치하는지 확인 → 30일 가입 추이 그래프가 일자별로 표시되는지 확인. 독립 검증 가능.

**Acceptance Scenarios**:

1. **Given** 가입자·작품 데이터가 존재, **When** 운영자가 대시보드를 연다, **Then** 총 가입자·오늘 신규·이번 주 신규·활성 사용자·총 작품 수 카운트가 표시된다.
2. **Given** 최근 30일간 가입 기록, **When** 운영자가 대시보드를 본다, **Then** 일별 가입 수가 그래프로 표시된다.
3. **Given** 데이터가 없는 신규 환경, **When** 운영자가 대시보드를 본다, **Then** 0 또는 빈 상태가 오류 없이 표시된다.

---

### User Story 4 - 문의 외부 채널 안내 (Priority: P3)

사용자는 문제나 질문이 생기면 연락할 곳이 필요하다. 서비스 화면에 "문의하기" 링크를 두어 외부 채널(이메일 등)로 연결한다. 인앱 문의 폼·접수함은 만들지 않는다.

**Why this priority**: 사용자 신뢰에 도움되나 가장 가벼운 작업(링크 1개). 인앱 처리 없이 외부로 위임한다.

**Independent Test**: 사용자가 서비스 화면에서 "문의하기"를 누르면 지정한 외부 채널이 열린다.

**Acceptance Scenarios**:

1. **Given** 서비스 화면, **When** 사용자가 "문의하기"를 누른다, **Then** 지정된 외부 문의 채널이 열린다.

---

### Edge Cases

- 비관리자(일반 로그인 사용자 또는 비로그인)가 운영 기능 또는 운영 데이터에 접근을 시도하면 차단되고 운영 데이터가 노출되지 않는다.
- 공지 제목/본문이 비어 있으면 발행되지 않는다(유효성 거부).
- 공지가 매우 길거나 다수일 때 배너는 최신 1건만, 목록은 페이지 단위로 처리한다.
- 회원 검색 결과가 없으면 빈 결과를 오류 없이 표시한다.
- 통계 대상 데이터가 0건일 때 0/빈 상태로 표시한다.

## Requirements *(mandatory)*

### Functional Requirements

**공지사항 (US1)**

- **FR-001**: 운영자는 공지를 작성(제목·본문)·수정·삭제할 수 있어야 한다.
- **FR-002**: 운영자는 공지의 공개/비공개 상태를 전환할 수 있어야 한다.
- **FR-003**: 시스템은 발행된(공개) 공지만 사용자에게 노출해야 한다.
- **FR-004**: 사용자는 서비스 홈 상단 배너에서 최신 공개 공지 1건을 볼 수 있어야 하며, 공개 공지가 없으면 배너는 표시되지 않아야 한다.
- **FR-005**: 사용자는 공지 목록 페이지에서 공개 공지를 최신순으로 보고 상세를 열 수 있어야 한다.
- **FR-006**: 시스템은 제목 또는 본문이 비어 있는 공지의 발행을 거부해야 한다.

**회원 조회 (US2)**

- **FR-007**: 운영자는 전체 가입자를 가입일 최신순으로 목록 조회하고 페이지 단위로 탐색할 수 있어야 한다.
- **FR-008**: 운영자는 이메일로 회원을 검색할 수 있어야 한다.
- **FR-009**: 운영자는 회원 상세(가입일, 마지막 로그인, 이메일 인증 여부, 카카오 연동 여부, 작품 수)를 조회할 수 있어야 한다.
- **FR-010**: 시스템은 비밀번호 등 인증 비밀값을 어떤 운영 화면·응답에도 포함하지 않아야 한다.
- **FR-011**: 회원 관리 기능은 조회 전용이어야 한다(쓰기·삭제 액션 없음).

**통계 (US3)**

- **FR-012**: 운영자는 총 가입자 수, 오늘 신규 가입 수, 이번 주 신규 가입 수, 활성 사용자 수, 총 작품 수를 한 화면에서 볼 수 있어야 한다.
- **FR-013**: 운영자는 최근 30일 일별 가입 추이를 그래프로 볼 수 있어야 한다.

**문의 (US4)**

- **FR-014**: 서비스 화면에 외부 문의 채널로 연결되는 "문의하기" 링크가 있어야 한다.

**접근 제어 (전 기능 공통)**

- **FR-015**: 운영 기능과 운영 데이터(회원·통계·공지 관리)는 단일 지정 관리자만 접근할 수 있어야 하며, 그 외 사용자는 차단되어야 한다.
- **FR-016**: 비관리자가 운영 기능 접근을 시도하면 운영 데이터가 노출되지 않아야 한다.

### Key Entities *(include if feature involves data)*

- **공지(Announcement)** — 신규: 사용자에게 노출되는 안내 글. 속성: 제목, 본문, 공개 여부, 배너 노출 대상 여부, 공개 시각, 생성/수정 시각.
- **회원(User)** — 기존 재사용(읽기 전용): 가입자. 운영 조회에 쓰는 속성: 이메일, 카카오 연동 여부, 이메일 인증 여부, 마지막 로그인, 가입일. 작품 수는 작품(Project)에서 집계.
- **작성 세션·작품(WorkSession / Project)** — 기존 재사용(읽기 전용): 통계 집계 원천(총 작품 수 등).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 운영자는 공지 1건을 5분 이내에 작성·발행할 수 있고, 발행 직후 사용자 홈에서 확인된다.
- **SC-002**: 사용자는 홈 접속 시 최신 공지를 별도 탐색 없이 한눈에 인지할 수 있다(상단 배너).
- **SC-003**: 운영자는 특정 회원을 이메일로 검색해 30초 이내에 그 회원의 가입일·마지막 로그인을 확인할 수 있다.
- **SC-004**: 운영자는 대시보드 한 화면에서 총 가입자·신규 가입·활성 사용자·30일 가입 추이를 확인할 수 있다.
- **SC-005**: 비관리자는 어떤 경로로도 운영 데이터(회원 목록·통계·공지 관리)에 접근할 수 없다.
- **SC-006**: 회원의 비밀번호 등 비밀값은 어떤 운영 화면에도 노출되지 않는다.

## Assumptions

- **단일 관리자**: 운영자는 1명(나)이며, 다중 사용자 역할 권한 체계는 도입하지 않는다(운영자 다인화 시점에 후속).
- **기존 인증 재사용**: 관리자 인증은 서비스의 기존 로그인 체계를 재사용하고, 지정된 관리자 계정만 운영 기능에 통과시킨다.
- **별도 어드민 앱**: 운영 화면은 사용자 서비스와 분리된 별도 앱으로 배포한다(코드·배포·보안 경계 분리).
- **문의는 외부 채널**: 인앱 문의 폼·접수함은 만들지 않는다. "문의하기"의 기본 채널은 이메일(`jongbell4@gmail.com`)이며, 구현 1단계 전 카카오 채널 등으로 변경 가능.
- **활성 사용자 정의**: "활성 사용자" 기본 기준은 최근 7일 이내 로그인. 필요 시 조정 가능.
- **회원 쓰기 액션 제외**: 계정 잠금·삭제 등은 v1 범위 밖. 필요 시 데이터 직접 대응 또는 후속 라운드.
- **어드민 주소**: 기본 `admin.soseolbi.com`. 초기에는 임시 배포 주소로 시작 가능.
- **호스팅 플랜 유지**: 현 무료 플랜을 유지하며, 어드민 추가가 비용·한도에 미치는 영향은 무시 가능(운영자 본인 트래픽). 본 서비스의 플랜 상향은 별도 트랙.
- **공지 본문 형식**: 일반 텍스트(필요 시 간단한 줄바꿈) 수준. 리치 에디터는 범위 밖.
