# Feature Specification: 집필실 에디터 — 한글 받침 입력 정확성 + 서식 툴바 개선

**Feature Branch**: `025-editor-ime-toolbar`

**Created**: 2026-06-17

**Status**: Draft

**Input**: User description: "집필실 자체 에디터(024 custom-editor) 개선 — ① 받침 있는 한글 입력 시 받침이 다음 글자로 분리되는 IME 재조합 버그 수정(예: \"노란달\"→\"놀란달\"), ② 현재 인라인 버튼 형태의 서식 도구를 일반 에디터(TipTap 등) 수준의 툴바 UI로 디자인 개선. 1→2 순서로 진행."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 받침 있는 한글이 끊김 없이 정확히 입력된다 (Priority: P1)

작가가 집필실 에디터에서 한국어 본문을 입력할 때, 받침(글자의 끝소리 자음)이 있는 글자 뒤에 새 글자를 이어 쳐도 받침이 다음 글자로 잘못 넘어가지 않고 친 그대로 입력되어야 한다.

배경: 한글은 한 음절이 초성(첫 자음) + 중성(모음) + 받침(끝 자음)으로 조합된다. 받침으로 끝나는 글자 다음에 자음으로 시작하는 글자를 칠 때, 입력기(IME, 글자 조합 엔진)는 일시적으로 그 자음을 앞 글자의 받침 자리에 붙였다가, 뒤이어 모음이 오면 받침을 떼어 새 글자의 초성으로 옮긴다(받침 재조합). 현재 에디터는 이 재조합이 깨져 자음이 앞 글자의 받침에 고착되고 분리되지 않는다.

대표 증상: "노란달"을 한 글자씩 입력하면 "놀란달"이 된다 — "란"의 첫소리 ㄹ이 앞 글자 "노"의 받침으로 들어가 "놀"이 되고, 다음 모음 입력 시 분리되어야 할 받침이 분리되지 않는다.

**Why this priority**: 한국어 우선 제품(DESIGN.md 전제)에서 본문 입력은 가장 근본적인 핵심 기능이다. 받침 입력이 깨지면 작가가 정상적으로 글을 쓸 수 없어 제품 가치 자체가 무너진다. 양보 불가 핵심 기능이므로 최우선.

**Independent Test**: 비어 있는 새 챕터에서 "노란달", "강물", "갈가마귀", "맑은하늘" 등 "받침 → 자음 초성" 연속 조합을 한 글자씩 입력하고, 화면 표시와 자동저장 후 재로드 결과가 모두 입력한 그대로인지 확인하면 단독 검증된다.

**Acceptance Scenarios**:

1. **Given** 비어 있는 챕터, **When** "노란달"을 한 글자씩 입력, **Then** 화면에 "노란달"이 표시된다 (받침 이동 없음)
2. **Given** 비어 있는 챕터, **When** 받침으로 끝나는 글자 + 자음 초성 글자를 연속 입력(예: "강물", "산길", "봄날"), **Then** 각 글자가 입력한 자모 순서대로 조합된다
3. **Given** 한글을 입력한 본문, **When** 자동저장 후 챕터를 재로드, **Then** 입력한 한글이 손실·변형 없이 복원된다
4. **Given** 비어 있는 챕터, **When** 조합 중인 글자 위에서 다음 자모를 빠르게 연타, **Then** 조합이 중간에 끊기거나 자모가 분리되지 않는다

### User Story 2 - 서식 도구가 일반 에디터 수준의 툴바로 제공된다 (Priority: P2)

작가가 본문에 서식(굵게/기울임/밑줄/취소선, 제목, 인용, 목록, 구분선)을 적용할 때, 현재의 낱개 버튼 나열 대신 일반 글쓰기 에디터에서 익숙한 정돈된 툴바로 조작한다.

배경: 현재 서식 도구는 동일한 모양의 인라인 버튼이 한 줄로 나열돼 있어 시각적으로 정돈되지 않고, 기능 그룹 구분이 약하며, 어떤 서식이 지금 적용 중인지 직관적으로 드러나지 않는다. 널리 쓰이는 글쓰기 에디터가 제공하는 툴바(아이콘, 기능 그룹 구분선, 활성 상태 강조)로 개선해 집필 경험의 완성도를 높인다.

**Why this priority**: 입력 정확성(P1) 다음으로 집필 경험의 질을 좌우한다. 서식 기능 자체는 이미 동작하므로 본 스토리의 가치는 "사용성·시각적 완성도" 향상이다. P1이 깨진 상태에서는 의미가 없어 후순위.

**Independent Test**: 집필실에서 툴바의 각 서식 도구를 눌러 선택한 텍스트/블록에 서식이 적용되고, 커서가 놓인 위치의 현재 서식이 툴바에 강조 표시되는지 확인하면 단독 검증된다.

**Acceptance Scenarios**:

1. **Given** 본문 일부를 선택한 상태, **When** 툴바의 굵게 도구를 누름, **Then** 선택 부분이 굵게 표시되고 툴바의 굵게 도구가 활성으로 강조된다
2. **Given** 커서가 제목 블록 안에 있음, **When** 툴바를 봄, **Then** 현재 블록 유형(제목 레벨)이 활성 상태로 표시된다
3. **Given** 여러 서식 도구, **When** 툴바를 봄, **Then** 관련 기능(글자 서식 / 블록 서식 등)이 그룹으로 구분되어 한눈에 들어온다
4. **Given** A형 집필실과 B형 집필실, **When** 각각 진입, **Then** 동일한 형태의 툴바가 일관되게 제공된다

---

### Edge Cases

**받침 입력 (P1)**

- 겹받침(ㄳ, ㄼ, ㄵ 등)으로 끝나는 글자 뒤에 자음/모음을 입력할 때의 재조합
- 받침 글자 뒤에 모음으로 시작하는 글자(받침 유지) vs 자음으로 시작하는 글자(받침 후보 결합)의 분기
- 한자 변환, 자모 백스페이스 분해 등 기존 한글 입력기 4케이스(`docs/poc/0-1-tiptap-korean.md`)와의 정합
- 줄 끝 / 블록 끝 / 소프트 줄바꿈 경계에서 받침이 있는 글자를 입력할 때
- 선택 영역을 한글로 덮어쓰며 입력할 때

**툴바 (P2)**

- 좁은 화면 또는 원고지 폭에서 툴바 도구가 한 줄에 다 들어가지 않을 때
- 커서가 구분선(캐럿 진입 불가한 원자 블록) 근처에 있을 때 블록 서식 도구의 표시 상태
- 커서가 접혀 있는(선택 없는) 상태에서 다음 타이핑에 적용될 보류 서식의 표시

## Requirements *(mandatory)*

### Functional Requirements — 받침 입력 (P1)

- **FR-001**: 받침이 있는 한글 글자 뒤에 새 글자를 입력할 때, 입력기의 받침 재조합이 정상 동작하여 받침이 앞 글자에 잘못 고착되지 않아야 한다.
- **FR-002**: 한글을 연속 입력할 때 각 음절이 사용자가 입력한 자모 순서 그대로 조합되어야 한다.
- **FR-003**: 한글 입력 결과가 화면 표시 · 자동저장 · 재로드 전 구간에서 손실·변형 없이 일관되어야 한다.
- **FR-004**: 기존 한글 입력기 4케이스(빠른 타자, 조합 중 서식 토글, 한자 변환, 백스페이스 분해)가 회귀 없이 유지되어야 한다.

### Functional Requirements — 툴바 (P2)

- **FR-010**: 기존 서식 기능 전체(굵게/기울임/밑줄/취소선, 제목 H1~H3, 인용, 글머리표 목록, 번호 목록, 구분선)가 새 툴바에서 빠짐없이 제공되어야 한다.
- **FR-011**: 현재 커서/선택 위치에 적용된 서식이 툴바에 활성 상태로 표시되어야 한다.
- **FR-012**: 관련 서식 도구가 의미 단위로 그룹화되어 시각적으로 구분되어야 한다.
- **FR-013**: A형 집필실과 B형 집필실 양쪽에서 동일한 형태의 툴바가 일관되게 제공되어야 한다.
- **FR-014**: 툴바 디자인 변경이 기존 서식 적용 동작과 저장 결과를 바꾸지 않아야 한다 (표현 계층만 교체).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: "받침 → 자음 초성" 연속 한글 입력에서 받침 분리/고착 오류 0건 (대표 케이스 "노란달" 포함 다수 케이스 검증).
- **SC-002**: 작가가 한국어 본문 한 문단을 입력 중단·재시도 없이 끝까지 입력 완료할 수 있다.
- **SC-003**: 기존 한글 입력기 4케이스 회귀 0건.
- **SC-004**: 변경 전 제공되던 모든 서식 기능이 새 툴바에서 빠짐없이 접근 가능하다 (기능 누락 0).
- **SC-005**: 작가가 현재 적용 중인 서식을 툴바만 보고 식별할 수 있다.

## Assumptions

- 받침 버그의 근본 원인은 본 명세 시점에 **미확정**이다. 추측으로 수정 방법을 단정하지 않고, 구현 단계에서 실제 브라우저 재현과 입력 이벤트 관찰로 끊긴 지점을 먼저 확정한 뒤 고친다 (HOW는 plan/implement 단계에서 결정).
- 대상은 024 라운드에서 도입한 자체 입력 엔진 기반 집필실 에디터이며, 이전 TipTap 에디터는 이미 폐기된 상태다.
- 백엔드 변경 없음. 본문 저장 포맷도 바꾸지 않는다 (순수 프론트엔드 작업).
- 툴바 개선은 서식을 적용하는 내부 명령 로직을 바꾸지 않고, 표현 계층(버튼 나열 → 툴바 디자인)만 교체한다.
- A형 집필실과 B형 집필실은 동일한 에디터 컴포넌트를 공유하므로, 한 번의 변경이 양쪽에 함께 반영된다.
- 대상 환경은 자체 입력 엔진이 지원되는 최신 브라우저(Chromium 계열)이며, Safari 대응은 별도 후속 라운드 영역이다.
