## User Scenarios & Testing *(mandatory)*

### User Story 1 - 이메일·비밀번호 회원가입 및 로그인 (Priority: P1)

작가 본인(1차 사용자)이 이메일과 비밀번호로 계정을 만들고, 이메일 인증을 통과한 뒤 로그인하여 자신의 작업공간에 접근할 수 있어야 한다. 또한 로그아웃하면 발급된 갱신 토큰이 즉시 무효화되어 다음 갱신 요청이 거부되어야 한다.

**Why this priority**: 본 도구의 1차 사용자는 본인 1명. 카카오 등 다른 진입 흐름이 없어도 본인 dogfooding(주 1회 글쓰기 세션) 진입이 가능한 최소 본질 흐름이다. 이 흐름 단독으로 V1 MVP 의 인증 영역 가치를 만든다.

**Independent Test**: 새 이메일로 회원가입 → 발송된 인증 토큰 검증 → 비밀번호 로그인 → 발급된 access token 으로 보호 endpoint(`GET /api/auth/me`) 호출 GREEN → 갱신 토큰으로 새 access token 발급 → 로그아웃 후 같은 갱신 토큰으로 다시 시도 시 거부.

**Acceptance Scenarios**:

1. **Given** 미가입 이메일, **When** 이메일·비밀번호 회원가입 요청, **Then** 계정이 생성되고 24시간 유효한 이메일 인증 토큰이 메일로 발송된다.
2. **Given** 인증 토큰을 받은 사용자, **When** 인증 토큰 제출, **Then** 계정의 이메일 인증 상태가 완료로 표시되고 같은 토큰의 재사용은 거부된다.
3. **Given** 이메일 인증이 완료된 계정, **When** 올바른 비밀번호로 로그인, **Then** 1시간 유효 access token 과 30일 유효 갱신 토큰이 발급된다.
4. **Given** 유효한 갱신 토큰, **When** access token 갱신 요청, **Then** 새 access token 이 발급된다.
5. **Given** 로그인 상태, **When** 로그아웃 요청, **Then** 해당 갱신 토큰이 즉시 무효화되어 이후 갱신 요청이 거부된다.
6. **Given** 사용자 A 의 access token, **When** 보호 endpoint 호출, **Then** A 본인의 데이터만 반환되며 타 사용자의 데이터는 존재 자체가 노출되지 않는다(404 응답).

### User Story 2 - 카카오 한 번 동의로 즉시 로그인 (Priority: P2)

한국어 작가 사용자가 별도 비밀번호 없이 카카오 한 번의 동의로 계정이 만들어지고 즉시 작업공간에 진입할 수 있어야 한다.

**Why this priority**: 본 도구 1차 사용자는 한국어 작가. 카카오로 진입할 수 있으면 비밀번호 만들기/기억하기 마찰이 0 이 되어 dogfooding 시작 마찰이 사라진다. 본인이 이 경로로 진입할 확률이 가장 높다.

**Independent Test**: 신규 카카오 사용자가 동의 → 콜백 통과 → 계정 자동 생성 + 이메일 인증 즉시 완료 + 토큰 발급 → 보호 endpoint 호출 GREEN.

**Acceptance Scenarios**:

1. **Given** 카카오 계정으로 처음 진입한 사용자, **When** 카카오 동의 후 콜백 도달, **Then** 신규 계정이 생성되고 이메일 인증 상태가 즉시 완료로 표시된다.
2. **Given** 이미 카카오가 연결된 계정, **When** 카카오로 로그인, **Then** 기존 계정으로 토큰이 발급된다.
3. **Given** 콜백 성공, **When** 프론트로 복귀, **Then** access token 과 갱신 토큰이 프론트가 안전하게 받을 수 있는 형식으로 전달된다.
4. **Given** 동일 이메일이 이미 비밀번호로 가입돼 있고 카카오 연결이 없는 상태, **When** 비로그인 사용자가 그 이메일로 카카오 진입 시도, **Then** 충돌 에러와 함께 "로그인 후 카카오 연결" 안내가 표준 envelope 으로 반환된다.

### User Story 3 - 비밀번호 재설정 (Priority: P3)

비밀번호를 잊은 사용자가 이메일 인증을 통해 새 비밀번호를 설정할 수 있어야 한다.

**Why this priority**: 본질적인 신규 가치는 아니지만, 본인 dogfooding 중 비밀번호 분실 시 복구 경로가 없으면 dogfooding 자체가 멈춘다(주 1회 사용 기준 1~2회 잊을 위험).

**Independent Test**: 가입 계정으로 재설정 요청 → 메일 토큰 수신 → 토큰 + 새 비밀번호 제출 → 변경된 비밀번호로 로그인 GREEN. 만료/재사용/약한 비밀번호 시도는 모두 거부.

**Acceptance Scenarios**:

1. **Given** 가입 계정, **When** 비밀번호 재설정 요청, **Then** 30분 만료의 일회용 토큰이 메일로 발송된다.
2. **Given** 유효한 재설정 토큰 + 비밀번호 정책을 만족하는 새 비밀번호, **When** 변경 제출, **Then** 비밀번호가 갱신되고 토큰은 사용 완료로 표시되어 재사용이 거부된다.
3. **Given** 만료된 재설정 토큰, **When** 변경 시도, **Then** 거부된다.
4. **Given** 이미 사용된 재설정 토큰, **When** 재사용 시도, **Then** 거부된다.
5. **Given** 약한 새 비밀번호(12자 미만 또는 영문/숫자/특수 조합 미달), **When** 변경 시도, **Then** 동일한 비밀번호 정책 위반으로 거부된다.

### User Story 4 - 5회 실패 + 30분 잠금으로 무작위 대입 차단 (Priority: P4)

한 계정에 비밀번호 시도가 5회 연속 실패하면 다음 30분 동안 로그인을 거부하여 무작위 대입 공격을 차단한다.

**Why this priority**: 본인 dogfooding 단계에서는 위협이 낮지만, V1 외부 노출 후 즉시 발생할 수 있는 보안 회귀를 본 인증 도입 시점에 박는다(이후 추가가 더 비싸다).

**Independent Test**: 같은 이메일에 5회 연속 잘못된 비밀번호 → 6번째 시도부터는 잠금으로 거부 → 잠금 만료 시각 이후 정상 비밀번호로 다시 통과.

**Acceptance Scenarios**:

1. **Given** 가입 계정, **When** 5회 연속 잘못된 비밀번호로 로그인 시도, **Then** 6번째 시도부터는 잠금 상태(`LOGIN_LOCKED`) 로 거부된다.
2. **Given** 잠금 상태, **When** 잠금 만료 시각 이후 올바른 비밀번호로 시도, **Then** 정상 로그인된다.
3. **Given** 4회 실패 후 5번째 시도에서 올바른 비밀번호로 성공, **When** 이후 시도, **Then** 실패 누적 횟수가 초기화되어 새 시도가 가능하다.
4. **Given** 잠금 응답, **When** 클라이언트가 사용자에게 표시, **Then** "30분 후 다시 시도" 안내가 가능한 사용자 메시지가 응답에 포함된다.

### User Story 5 - 이메일과 카카오를 한 계정에 둘 다 연결 (Priority: P5)

같은 사용자가 이메일·비밀번호로 가입했더라도 로그인 상태에서 카카오를 추가 연결할 수 있어야 하고, 카카오로 가입한 사용자가 이메일·비밀번호를 추가 등록할 수도 있어야 한다.

**Why this priority**: SoT §5-1 #2 결정(Q②-1 B). 본인이 환경별로 다른 진입 방법을 쓸 수 있을 때(데스크탑은 카카오, 모바일은 비밀번호) dogfooding 흐름이 끊기지 않는다.

**Independent Test**: 이메일 가입 → 로그인 → 카카오 추가 연결 → 로그아웃 → 같은 계정에 카카오만으로 로그인 GREEN. 역방향(카카오 가입 → 비밀번호 추가)도 동일.

**Acceptance Scenarios**:

1. **Given** 이메일 로그인 상태 + 카카오 미연결, **When** 카카오 추가 연결 시작, **Then** 카카오 동의 후 같은 계정에 카카오 식별자가 추가된다.
2. **Given** 카카오 가입 계정 + 비밀번호 미설정 로그인 상태, **When** 비밀번호 추가 등록 요청, **Then** 같은 계정에 비밀번호가 등록되어 다음번 이메일·비밀번호 로그인이 가능하다.
3. **Given** 이미 다른 사용자가 사용 중인 카카오 식별자, **When** 본인 계정에 연결 시도, **Then** 충돌로 거부되고 계정 상태는 변경되지 않는다.

### User Story 6 - 본인 정보 조회 + 임시 owner 헤더 정리 (Priority: P6)

로그인 사용자가 본인의 식별자·이메일·카카오 연결 여부·이메일 인증 시각·활성 모바일 캡처 토큰 수를 조회할 수 있어야 한다. 또한 기존 임시 owner 식별 방식(특정 헤더로 owner 를 클라이언트가 직접 지정하던 방식)은 본 spec 종료 시점에 인증된 사용자 식별 기반으로 교체되어야 한다.

**Why this priority**: 홈/설정 화면이 본인 정보 조회에 의존한다. 또한 001 spec(`specs/001-phase-1a-backend-scaffold`) 의 임시 owner 헤더를 인증 도입 시점에 함께 정리하지 않으면 인증 도입 후에도 임시 헤더가 살아남아 회귀 위험이 누적된다(`docs/plan/02-progress.md §3 트랙 B 1B-5` 명시).

**Independent Test**: 토큰 로그인 상태에서 `/api/auth/me` 호출 → 본인 정보(이메일·카카오 연결 여부·이메일 인증 시각·활성 모바일 캡처 토큰 수) 반환. 기존 프로젝트 CRUD endpoint 가 임시 owner 헤더 없이 access token 만으로 동작. 임시 헤더 강제 지정 시도는 인증된 식별자가 우선되어 무시.

**Acceptance Scenarios**:

1. **Given** 로그인 상태, **When** 본인 정보 조회 호출, **Then** 본인 식별자, 이메일, 카카오 연결 여부, 이메일 인증 시각, 활성 모바일 캡처 토큰 수가 표준 envelope 으로 반환된다.
2. **Given** 로그인 상태, **When** 기존 프로젝트 CRUD endpoint 호출, **Then** 임시 owner 헤더 없이 토큰만으로 owner 식별이 일어난다.
3. **Given** 비로그인 호출, **When** 보호 endpoint 호출, **Then** `AUTH_TOKEN_MISSING` 응답이 표준 envelope 으로 반환된다.
4. **Given** 클라이언트가 owner 식별자를 임의 헤더로 조작 시도, **When** 보호 endpoint 호출, **Then** 인증된 사용자 식별이 우선되어 조작이 무시된다.

### Edge Cases

- 약한 비밀번호(12자 미만 또는 영문/숫자/특수 조합 미달) 가입·재설정 시도 → 400 + `PASSWORD_TOO_WEAK` 또는 동등 코드.
- 이메일 형식 오류 가입 → 400 + `EMAIL_INVALID_FORMAT` 또는 `VALIDATION_FAILED`.
- 이미 가입된 이메일 재가입 → 409 + `EMAIL_ALREADY_REGISTERED`.
- 만료된 이메일 인증/재설정 토큰 사용 → 거부.
- 이미 사용된 일회용 토큰 재사용 → 거부.
- 만료된 access token 으로 보호 endpoint 호출 → 401 + `AUTH_TOKEN_EXPIRED` (클라이언트는 자동 갱신 시도 가능).
- 로그아웃으로 무효화된 갱신 토큰 재사용 → 401 + `AUTH_TOKEN_REVOKED`.
- 형식이 아예 잘못된 토큰 → 401 + `AUTH_TOKEN_INVALID`.
- Authorization 헤더 자체 부재 → 401 + `AUTH_TOKEN_MISSING`.
- 카카오 콜백 도중 동의 거부 또는 외부 인증 상태 만료.
- 동일 이메일이 비로그인 상태에서 카카오 신규 가입을 시도(이미 이메일 가입자 존재 + 카카오 연결 없음) → 409 + `KAKAO_EMAIL_ALREADY_REGISTERED` + "로그인 후 카카오 연결" 안내.
- 이미 다른 사용자에 묶인 카카오 식별자에 본인 계정을 연결 시도 → 409 + `KAKAO_ALREADY_LINKED`.
- CORS preflight(OPTIONS) 요청이 본 spec 의 어떤 endpoint 로 도달하든 V1 정책에 따라 통과.
- 모바일 캡처용 장기 토큰(접두사가 `wnt_` 인 토큰)이 일반 보호 endpoint 에 잘못 사용된 경우 → 본 spec 의 모바일 토큰 필터는 `POST /api/capture` 한정이므로 다른 경로에서는 적용되지 않고 거부.
- 같은 계정에 동시에 다수의 로그인·갱신·재설정 시도가 들어오는 경우 → 잠금 카운트와 토큰 무효화가 결정적으로 처리.
- 카카오 OAuth 동의 화면에서 사용자가 권한을 일부만 동의 → 필수 권한(`account_email`) 누락 시 거부.

## Requirements *(mandatory)*

### Functional Requirements

본 요구사항은 모두 백엔드 SoT(`docs/plan/03-backend-requirements.md`) 의 결정을 인용한다. 각 항목 끝의 (SoT §x) 는 출처.

#### 가. 계정 생성 / 이메일 인증

- **FR-001**: 시스템은 이메일 + 비밀번호로 신규 계정을 생성할 수 있어야 한다 (SoT §3-2 #1).
- **FR-002**: 비밀번호는 최소 12자 + 영문/숫자/특수문자 조합을 강제해야 한다. 위반 시 표준 검증 실패로 거부 (SoT §4-3).
- **FR-003**: 비밀번호는 단방향 해시(BCrypt cost 12) 로 저장되어야 한다 (SoT §4-3).
- **FR-004**: 이메일은 시스템 내 유일성 제약을 가져야 한다. 재가입 시도는 409 충돌 (SoT §2-2 Users + §3-1 에러 매트릭스).
- **FR-005**: 회원가입 직후 24시간 만료의 이메일 인증 토큰이 생성되고 인증 메일이 발송되어야 한다 (SoT §2-2 AuthToken).
- **FR-006**: 이메일 인증 토큰 검증 endpoint 는 일회용으로 동작해야 한다(검증 후 같은 토큰 재사용 거부) (SoT §2-2 AuthToken).

#### 나. 로그인 / 토큰 발급·갱신·로그아웃

- **FR-007**: 이메일 인증이 완료된 계정만 이메일·비밀번호 로그인을 통과해야 한다.
- **FR-008**: 로그인 성공 시 1시간 만료 access token 과 30일 만료 갱신 토큰이 함께 발급되어야 한다 (SoT §4-1).
- **FR-009**: access token 은 표준 알고리즘(HS256) 으로 서명되고 payload 는 사용자 식별자·이메일·발급/만료 시각을 포함해야 한다 (SoT §4-1).
- **FR-010**: 갱신 토큰은 시스템 내부 저장소(통합 토큰 보조 테이블)에 보관되어 즉시 무효화 가능해야 한다 (SoT §4-1).
- **FR-011**: 유효한 갱신 토큰으로 새 access token 을 발급할 수 있어야 하며, 무효/미존재/만료/사용 완료 갱신 토큰은 거부되어야 한다.
- **FR-012**: 로그아웃 요청 시 해당 갱신 토큰이 즉시 무효화되어 이후 갱신 요청을 거부해야 한다 (SoT §3-2 #9).

#### 다. 로그인 실패 잠금

- **FR-013**: 같은 계정에 비밀번호 실패가 5회 연속 누적되면 그 시점부터 30분 동안 잠금 상태가 되어 새 로그인 시도가 거부되어야 한다 (SoT §4-4 + §2-2 Users).
- **FR-014**: 로그인 성공 시 실패 누적 횟수가 초기화되어야 한다.
- **FR-015**: 잠금 상태의 응답은 `LOGIN_LOCKED` 코드와 사용자에게 노출 가능한 안내 메시지(예: "30분 후 다시 시도") 를 포함해야 한다 (SoT §4-4).

#### 라. 비밀번호 재설정

- **FR-016**: 비밀번호 재설정 요청 시 30분 만료의 일회용 토큰이 생성되고 메일이 발송되어야 한다 (SoT §2-2 AuthToken).
- **FR-017**: 재설정 토큰 + 새 비밀번호로 변경 시 비밀번호 정책(FR-002, FR-003) 이 동일 적용되어야 한다.
- **FR-018**: 만료된 또는 이미 사용된 재설정 토큰은 거부되어야 한다 (SoT §2-2 AuthToken).

#### 마. 카카오 OAuth

- **FR-019**: 카카오 OAuth 진입 endpoint 와 콜백 endpoint 가 제공되어야 하며, 콜백은 카카오에서 받은 사용자 정보로 기존 계정을 조회하거나 신규 계정을 생성해야 한다 (SoT §3-2 #4·#5, §4-2).
- **FR-020**: 카카오로 신규 생성된 계정의 이메일 인증 상태는 즉시 완료로 표시되어야 한다(카카오가 이메일을 보증) (SoT §4-2).
- **FR-021**: 카카오 콜백 성공 시 access token 과 갱신 토큰이 프론트에 안전하게 전달되어야 한다(SoT §4-2 의 결정대로 URL fragment 형식). 결정된 형식이 일관되게 사용되어야 한다.
- **FR-022**: 비로그인 사용자가 카카오로 진입했는데 동일 이메일이 이미 비밀번호 가입자로 존재하고 카카오 연결이 없는 경우, 충돌 응답과 함께 "로그인 후 카카오 연결" 안내가 표준 envelope 으로 반환되어야 한다 (SoT §4-2 충돌 처리).

#### 바. 이메일 ↔ 카카오 추가 연결

- **FR-023**: 로그인 상태 사용자가 카카오를 추가 연결할 수 있어야 한다 (SoT §3-2 #11).
- **FR-024**: 카카오로 가입한 로그인 사용자가 이메일·비밀번호를 추가 등록할 수 있어야 한다. 이때 이메일은 카카오에서 받은 값을 그대로 사용하고 비밀번호 정책은 FR-002, FR-003 을 따른다 (SoT §3-2 #12).
- **FR-025**: 이미 다른 사용자에게 연결된 카카오 식별자는 본인 계정에 추가 연결할 수 없으며, 충돌로 거부되고 계정 상태는 변경되지 않아야 한다.

#### 사. 본인 정보 조회 + Owner Context 정리

- **FR-026**: 로그인 사용자가 본인 식별자·이메일·카카오 연결 여부·이메일 인증 시각·활성 모바일 캡처 토큰 수를 조회할 수 있어야 한다 (SoT §3-2 #10).
- **FR-027**: 기존 임시 owner 식별 방식(특정 헤더로 owner 를 클라이언트가 직접 지정하던 방식, 001 spec 도입분) 은 본 spec 완료 시점에 인증된 사용자 식별 기반으로 교체되어 런타임에서 제거되어야 한다 (`docs/plan/02-progress.md §3 트랙 B 1B-5`).
- **FR-028**: 클라이언트는 어떤 헤더로도 owner 식별자를 변조할 수 없어야 한다. owner 식별은 인증 결과에서만 도출되어야 한다.

#### 아. 보안 / 필터 체인 / 인프라

- **FR-029**: 인증 필터 체인은 세 개의 분리된 책임으로 구성되어야 한다 — (a) 로그인 시도 잠금 검증(로그인 endpoint 한정), (b) access token 검증(브라우저 토큰, 접두사 `eyJ`), (c) 모바일 캡처 장기 토큰 검증(접두사 `wnt_`, `POST /api/capture` 한정) (SoT §4-4).
- **FR-030**: 인증 실패는 5종 코드 (`AUTH_TOKEN_MISSING`, `AUTH_TOKEN_INVALID`, `AUTH_TOKEN_EXPIRED`, `AUTH_TOKEN_REVOKED`, `LOGIN_LOCKED`) 중 하나로 매핑되어 표준 envelope 으로 반환되어야 한다 (SoT §4-4 401 응답 매트릭스).
- **FR-031**: 공개 endpoint(회원가입·이메일 인증·로그인·카카오 OAuth·비밀번호 재설정·토큰 갱신) 외 모든 보호 endpoint 는 access token 없이는 401 응답을 반환해야 한다 (SoT §4-4).
- **FR-032**: CORS 는 V1 정책 — 모든 origin 허용, credentials=false, 메서드 GET/POST/PUT/PATCH/DELETE/OPTIONS, 허용 헤더 `Authorization`/`Content-Type`/`Idempotency-Key`/`Accept`, 노출 헤더 `Location`, preflight 캐시 1시간 — 으로 구성되어야 한다 (SoT §4-6).
- **FR-033**: 토큰 보조 테이블은 (이메일 인증 / 비밀번호 재설정 / 갱신) 3종을 하나의 통합 테이블 + 타입 컬럼으로 관리해야 한다. 모든 토큰은 시스템 내부에 SHA-256 해시 형태로만 보관되어야 한다 (SoT §2-2 AuthToken).
- **FR-034**: 만료된 토큰 / 사용 완료된 비-갱신 토큰의 일일 청소 작업이 제공되어야 한다 (SoT §2-2 AuthToken).
- **FR-035**: 외부 카카오 API 호출과 메일 발송은 데이터베이스 트랜잭션 외부에서 일어나야 하며, 외부 호출 실패 시에도 시스템 데이터의 일관성이 유지되어야 한다 (시스템 일반 원칙).

#### 자. 응답 / 검증

- **FR-036**: 모든 인증 endpoint 의 성공·실패 응답은 기존 표준 envelope (`success`/`data` 또는 `success`/`error.code`/`error.message`) 을 준수해야 한다 (SoT §3-1).
- **FR-037**: 인증 endpoint 의 입력 검증 실패는 400 + 도메인 코드 (`PASSWORD_TOO_WEAK`, `EMAIL_INVALID_FORMAT`) 또는 일반 코드 (`VALIDATION_FAILED`) 로 반환되어야 한다 (SoT §3-1 에러 코드 매트릭스).
- **FR-038**: 같은 계정에 동시에 다수의 인증 시도가 들어오는 경우(동시 로그인 시도 등) 잠금 카운트, 토큰 발급·무효화가 결정적으로 처리되어야 한다.

### Key Entities

본 spec 의 영속 모델 (SoT §2-2 인용):

- **Users (사용자 계정)**: 식별자, 이메일(유일), 카카오 식별자(선택, 유일), 비밀번호 해시(선택), 이메일 인증 시각(선택), 마지막 로그인 시각(선택), 누적 실패 횟수, 잠금 만료 시각(선택), 생성/수정 시각. 비밀번호 해시 또는 카카오 식별자 중 최소 하나가 채워져야 한다.
- **AuthToken (인증 보조 토큰)**: 식별자, 사용자 참조, 타입(이메일 인증 / 비밀번호 재설정 / 갱신), 토큰 해시(유일), 만료 시각, 사용 완료 시각(선택), 생성 시각. 만료/사용 완료 비-갱신 토큰은 일일 청소 작업 대상.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 신규 사용자가 이메일·비밀번호 회원가입 → 이메일 인증 → 첫 로그인 → 본인 정보 조회 까지를 5분 이내(이메일 도착 대기 포함)에 완료할 수 있다.
- **SC-002**: 카카오로 처음 진입한 사용자가 카카오 동의 화면 → 본인 정보 조회 까지를 1분 이내에 완료할 수 있다.
- **SC-003**: 본 spec 의 인증 endpoint 응답 100% 가 표준 envelope 을 따른다(자동 회귀 테스트로 검증).
- **SC-004**: 본 spec 의 인증 실패 응답 100% 가 5종 표준 코드 중 하나로 매핑된다(자동 회귀 테스트로 검증).
- **SC-005**: 5회 실패 + 30분 잠금 정책의 자동 회귀 테스트(잠금 진입 / 잠금 유지 / 잠금 해제 / 실패 카운트 초기화) 4 케이스가 모두 GREEN.
- **SC-006**: 만료/사용 완료/무효 토큰의 100% 가 거부된다(만료 / 사용 완료 / 형식 오류 / 다른 사용자 토큰 4 케이스 자동 회귀 테스트 GREEN).
- **SC-007**: 한 사용자의 보호 데이터에 다른 사용자가 접근하는 시도 100% 가 not-found 로 거부되며 데이터 존재 자체가 노출되지 않는다(자동 회귀 테스트).
- **SC-008**: 본 spec 완료 시점에 임시 owner 헤더(`X-User-Id`) 의 런타임 사용처가 코드 검색으로 0건이다.
- **SC-009**: 자동 검증 흐름(포맷 / 정적 분석 / 자동 테스트 / 빌드) 이 단일 명령(`./gradlew ktlintMainSourceSetCheck ktlintTestSourceSetCheck checkstyleMain test build`) 으로 GREEN 완료한다.
- **SC-010**: 본인 정보 조회 응답 시간이 단일 사용자 환경에서 일관되게 1초 이내(p95).
- **SC-011**: 본 spec 으로 추가된 코드의 자동 회귀 테스트 라인 커버리지가 60% 이상.
- **SC-012**: 사용자가 비밀번호를 잊어 재설정을 요청한 경우, 메일 수신 → 새 비밀번호 설정 → 로그인 까지를 5분 이내에 완료할 수 있다.

## Assumptions

- **백엔드 한정**: 본 spec 은 백엔드 구현 spec 이다. 인증 화면(`/auth/<panel>`) 의 프론트 외관은 002 spec(`specs/002-frontend-route-scaffold`) 에서 빈 골격으로 박혀 있으며, 본 spec 의 endpoint 와 결선하는 프론트 작업은 본 spec 의 보호 endpoint dogfooding GREEN 이후 별도 spec 또는 본 spec 완료 후 작업으로 진행한다.
- **메일 발송 인프라**: 메일 발송의 구체 인프라(SMTP 호스트 / 외부 메일 서비스) 선택은 본 spec 범위 외다. 메일 발송 추상화 인터페이스 + 로컬·테스트 단계 단순 구현(로그 기반) + 본인 dogfooding 단계 외부 서비스 결선은 plan 단계에서 결정한다. 로컬/테스트 환경의 자동 검증은 외부 메일 서비스 의존 없이 GREEN 이 되어야 한다.
- **카카오 외부 설정**: 카카오 OAuth redirect URI 등록, 클라이언트 ID/시크릿 발급은 외부 콘솔에서 사전 준비되어야 한다. 본 spec 은 시크릿을 환경 변수로 로딩하는 것까지만 가정하며, 시크릿 값 자체는 소스에 포함되지 않는다.
- **사용 부하**: 1차 사용자 = 본인 1명. 동시 사용자 부하 가정은 V1 범위 외다. FR-038 의 동시성 안전은 같은 사용자가 다른 디바이스로 동시에 로그인·갱신·재설정 시도하는 경우에 한정한다.
- **모바일 캡처 토큰 발급 API**: 모바일 캡처용 장기 토큰의 발급·조회·해지 endpoint 자체(SoT §3-6 의 #43~#46) 는 Week 4 영역으로 본 spec 범위 외다. 본 spec 은 (a) 인증 필터 체인에서 해당 토큰 분기의 골격 + (b) 본인 정보 조회 시 활성 토큰 수 노출 만 다룬다.
- **갱신 토큰 회전**: V1 = 회전(rotation) 미적용. 갱신 토큰은 만료까지 재사용 가능하며 로그아웃 시에만 즉시 무효화. V2 에서 회전 도입 검토 (SoT §4-1).
- **권한 분리(role)**: V1 = 단일 role (모든 사용자 = user). 관리자 등 추가 role 은 V2 영역 (SoT §4-5).
- **카카오 추가 연결 시 이메일 불일치**: 이메일·비밀번호 가입자가 카카오를 추가 연결할 때 카카오 이메일이 기존 계정 이메일과 다를 수 있다. 본 spec 은 기존 계정 이메일을 변경하지 않고 카카오 식별자만 추가한다. 향후 사용자 흐름에 따라 변경 필요 시 별도 결정.
- **001 spec 의존**: 본 spec 은 001-phase-1a-backend-scaffold 의 모든 산출물(표준 응답 envelope, 전역 예외 처리, CORS 골격, 데이터베이스 마이그레이션 파이프라인, `Users`/`Project` 1차 스키마) 위에 빌드된다.
- **002 spec 비결선**: 002 frontend route scaffold 의 인증 화면 골격은 본 spec endpoint 와 아직 결선되어 있지 않다. 본 spec 완료가 002 dogfooding 5 영역의 진행을 차단하지는 않는다.
