# Feature Specification: 관리자 문의·의견 보내기 (Desktop)

**Feature Branch**: `011-desktop-contact-feedback`

**Created**: 2026-06-08

**Status**: Draft

**Input**: 설계 문서 `docs/superpowers/specs/2026-06-08-desktop-contact-feedback-design.ko.md` — Rail 진입점 + 전용 화면 + 외부 폼-투-이메일 서비스 인앱 메일 전송 + 카카오 오픈채팅 링크

## User Scenarios & Testing *(mandatory)*

> 맥락: Desktop(Electron) 완전 로컬 앱을 패키징·배포한 뒤, 초기 소수 사용자로부터 의견을 받아 정식 배포판 개발에 반영하기 위한 피드백 수집 채널이다. 로컬 앱에는 서버가 없으므로 인앱 전송은 외부 폼-투-이메일 서비스를 경유한다.

### User Story 1 - 인앱으로 의견 보내기 (Priority: P1)

사용자는 앱을 쓰다가 불편한 점이나 바라는 점이 생기면, 좌측 메뉴(Rail) 최하단의 상시 노출된 진입점을 눌러 문의 전용 화면으로 이동한다. 화면의 메일 폼에 의견 본문을 적고(회신받을 이메일은 선택), [보내기]를 누르면 의견이 관리자에게 전달되고 "보내주셔서 감사합니다" 안내를 본다.

**Why this priority**: 이 기능의 존재 이유 그 자체 — 앱을 떠나지 않고 의견을 보낼 수 있어야 피드백이 실제로 모인다. 이것만 구현해도 독립적으로 가치를 전달하는 MVP다.

**Independent Test**: 진입점 클릭 → 본문 입력 → 보내기 → 감사 안내 노출까지를 단독으로 검증할 수 있고, 그 자체로 "의견이 전달된다"는 가치를 완결한다.

**Acceptance Scenarios**:

1. **Given** 앱이 실행된 상태, **When** Rail 최하단 문의 진입점을 클릭, **Then** 문의 전용 화면(메일 폼 + 카카오 버튼)이 메인 영역에 표시된다
2. **Given** 문의 화면에서 본문이 비어 있는 상태, **When** 폼을 본다, **Then** [보내기] 버튼이 비활성 상태다
3. **Given** 본문을 입력하고 회신 이메일은 비워 둔 상태, **When** [보내기] 클릭, **Then** 의견이 익명으로 전송되고 "보내주셔서 감사합니다" 안내가 노출되며 폼이 초기화된다
4. **Given** 본문과 올바른 형식의 회신 이메일을 입력한 상태, **When** [보내기] 클릭, **Then** 의견이 회신 이메일과 함께 전송되어 관리자가 답장할 수 있다
5. **Given** 의견 전송 시, **When** 관리자가 받은 메일을 확인, **Then** 본문에 앱 버전·운영체제·전송 시각이 자동으로 첨부되어 있다

---

### User Story 2 - 카카오 오픈채팅으로 문의 (Priority: P2)

사용자는 메일 대신 실시간 대화로 문의하고 싶을 때, 문의 화면의 "카카오톡 오픈채팅으로 문의" 버튼을 눌러 기본 브라우저로 열리는 오픈채팅방에서 직접 대화한다.

**Why this priority**: 메일이 부담스럽거나 즉답을 원하는 사용자를 위한 대체 채널. 핵심(P1)과 독립적으로 동작하며 폼과 무관하게 가치를 전달한다.

**Independent Test**: 문의 화면에서 카카오 버튼만 클릭해 기본 브라우저로 오픈채팅 URL이 열리는지 단독 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 문의 화면이 열린 상태, **When** "카카오톡 오픈채팅으로 문의" 버튼 클릭, **Then** 기본 브라우저로 오픈채팅방 URL이 열린다
2. **Given** 카카오 버튼 동작, **When** 클릭, **Then** 메일 폼의 입력 내용·상태에 영향을 주지 않는다(폼과 독립)

---

### User Story 3 - 전송 실패 시 의견 보존·재시도 (Priority: P3)

사용자가 오프라인이거나 외부 서비스 장애로 전송이 실패하면, 작성한 의견이 사라지지 않고 보존되며 "전송 실패, 잠시 후 다시 시도" 안내를 받아 다시 시도할 수 있다.

**Why this priority**: 핵심 흐름의 신뢰성 보강. 실패 시 작성 내용이 날아가면 사용자가 의견 제출을 포기하므로, 피드백 수집률을 지키는 안전망이다.

**Independent Test**: 전송 실패 상황을 유도해 안내 노출 + 폼 내용 보존을 단독 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 본문을 입력한 상태, **When** [보내기]를 눌렀으나 네트워크·서비스 실패, **Then** "전송 실패, 잠시 후 다시 시도" 안내가 노출되고 작성한 본문·회신 이메일이 폼에 그대로 남는다
2. **Given** [보내기]를 누른 직후 전송이 진행 중인 상태, **When** 응답을 기다리는 동안, **Then** [보내기] 버튼이 로딩·비활성 상태가 되어 연타·중복 전송이 방지된다
3. **Given** 전송 실패 후 폼 내용이 보존된 상태, **When** 사용자가 [보내기]를 다시 클릭, **Then** 재전송이 시도된다

---

### Edge Cases

- **본문 빈 값**: [보내기] 비활성으로 빈 의견 전송을 원천 차단한다.
- **잘못된 형식의 회신 이메일**: 형식 검증으로 안내한다. 단, 비어 있으면 선택 필드이므로 통과(익명 제출).
- **전송 중 창 전환·재클릭**: 전송 진행 중에는 버튼 비활성으로 중복 전송을 막는다.
- **오프라인 / 외부 서비스 장애**: 실패 안내 + 폼 내용 보존(US3).
- **카카오 URL 미설정/오류**: 진입점 동작은 메일 폼과 독립이므로 폼 사용에는 영향이 없어야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 좌측 메뉴(Rail) 최하단에 문의 진입점을 **상시 노출**해야 한다(숨김·조건부 노출 금지).
- **FR-002**: 사용자가 문의 진입점을 선택하면 시스템은 작품/집필/메모/기록과 동급의 **전용 화면(5번째 화면)으로 전환**해야 한다.
- **FR-003**: 문의 화면은 상단 안내 문구, 메일 폼(회신 이메일 입력 + 본문 입력 + 보내기 버튼), 구분선, 카카오 오픈채팅 버튼을 표시해야 한다.
- **FR-004**: 사용자는 의견 **본문을 필수로 입력**할 수 있어야 하며, 본문이 비어 있으면 [보내기]는 비활성이어야 한다.
- **FR-005**: 회신 이메일은 **선택 입력**이어야 하며, 비워 두면 **익명으로 제출**되어야 한다.
- **FR-006**: 회신 이메일이 입력된 경우 시스템은 **간단한 이메일 형식 검증**을 수행해야 한다. 비어 있으면 통과한다.
- **FR-007**: 사용자가 [보내기]를 선택하면 시스템은 의견을 외부 폼-투-이메일 서비스를 통해 **관리자에게 전달**해야 한다.
- **FR-008**: 전송되는 의견 본문에는 **앱 버전·운영체제·전송 시각**이 자동으로 첨부되어야 한다.
- **FR-009**: 회신 이메일이 있으면 관리자가 **답장 가능한 형태**(회신 주소)로 전달되어야 한다.
- **FR-010**: 전송 성공 시 시스템은 "보내주셔서 감사합니다"류 **성공 안내**를 노출하고 폼을 초기화해야 한다.
- **FR-011**: 전송 진행 중에는 [보내기] 버튼을 **로딩·비활성**으로 두어 연타·중복 전송을 방지해야 한다.
- **FR-012**: 전송이 실패(오프라인·서비스 장애 등)하면 시스템은 **실패 안내를 노출하고 작성 내용(본문·회신 이메일)을 보존**해 재시도를 허용해야 한다.
- **FR-013**: 사용자가 카카오 오픈채팅 버튼을 선택하면 시스템은 **기본 브라우저로 오픈채팅방 URL을 열어야** 한다.
- **FR-014**: 카카오 버튼 동작은 메일 폼의 입력·상태와 **독립**이어야 한다(서로 영향 없음).
- **FR-015**: 의견 전송 채널 연동 값(외부 서비스 식별자, 카카오 오픈채팅 URL)은 **앱 내 상수로 관리**되며, 사용자가 별도 입력하지 않아도 동작해야 한다.
- **FR-016**: 기존 기능(작품/집필/메모/기록 화면 및 기존 동작)에 **회귀가 없어야** 한다.

### Key Entities *(include if feature involves data)*

- **문의(Contact Message)**: 사용자가 전송하는 1건의 의견. 속성 — 본문(필수), 회신 이메일(선택), 자동 첨부 메타(앱 버전·운영체제·전송 시각). 로컬에 영속 저장하지 않고 외부 서비스로 전달만 한다(히스토리는 범위 밖).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자는 어느 화면에서든 진입점 클릭 → 본문 입력 → 전송까지 **3단계 이내**로 의견을 보낼 수 있다.
- **SC-002**: 본문이 비어 있는 상태에서는 의견 전송이 **불가능**하다(빈 의견 전송 0건).
- **SC-003**: 회신 이메일을 비운 제출은 **100% 익명**으로 처리되고, 입력한 제출은 관리자가 **답장할 수 있는** 형태로 도착한다.
- **SC-004**: 관리자가 수신하는 모든 의견에는 앱 버전·운영체제·전송 시각이 **빠짐없이** 첨부된다.
- **SC-005**: 전송 실패 시 작성한 의견이 **유실되지 않고** 재시도할 수 있다(실패로 인한 내용 손실 0건).
- **SC-006**: 카카오 버튼 클릭 시 기본 브라우저로 오픈채팅방이 **열린다**(폼 상태 변화 없음).
- **SC-007**: 기능 추가 후에도 기존 화면·동작의 자동화 검증이 **전부 통과**(회귀 0)한다.

## Assumptions

- 외부 폼-투-이메일 서비스로 **Formsubmit**을 사용한다(2026-06-08 실측 확정). 무가입·API 키 없음 — 엔드포인트 경로의 수신 이메일/랜덤 해시가 곧 수신처이므로 별도 `.env`·시크릿 관리가 필요 없다. (당초 Web3Forms 였으나 **Web3Forms 는 main 프로세스 호출을 무료 플랜에서 차단**함이 실측되어 교체)
- 모든 외부 호출은 Electron main 프로세스를 경유한다. Formsubmit 은 main 호출 시 **`Referer` 헤더가 필요**함이 실측으로 확인됐다(헤더 1줄로 충족, 구조 변경 없음).
- 회신 주소 전달 필드명 등 외부 서비스의 세부 payload 규격은 **구현 시 Formsubmit 공식 문서 1회 확인 + dogfooding 답장 테스트로 확정**한다(추측 금지). 핵심 전송 동작은 실측 확정.
- 구현 착수 전 사용자(관리자)가 (1) Formsubmit 수신 엔드포인트(이메일 첫 제출 → activation, 이메일 비노출용 해시 권장), (2) 카카오 오픈채팅방 생성 및 공유 URL 확보를 준비한다. 이 값들이 없으면 실제 전송·연결은 동작하지 않는다. (activation 은 2026-06-08 완료)
- 다음은 이번 범위 밖(YAGNI)이다: 보낸 문의의 로컬 저장·조회(히스토리), 문의 분류(버그/제안/기타) 셀렉트, 첨부파일·스크린샷 업로드, 백엔드(Spring Boot) 경유 전송. 외부 서비스 전송 모듈은 추후 백엔드로 교체할 수 있도록 격리하되 지금은 미구현이다.
- 안내·라벨 등 사용자 노출 문구는 한국어 우선(프로젝트 전제)이다.
