## User Scenarios & Testing *(mandatory)*

사용자는 **web 브라우저로 접속한 작가**다. desktop 앱에서 쓰던 화면·집필 경험이 **설치 없이 브라우저에서 1:1로** 동작하고, 데이터는 계정에 묶여 서버에 영속·기기 간 동기화된다(하위 작업 1 backend). 인증·로그인은 기존 web 자산(frontend 005)을 그대로 쓴다.

핵심 제약: desktop 의 **"진짜 페이지 분할"(원고지처럼 본문이 페이지로 나뉘어 보임)** 과 **한글 입력(IME)** 이 브라우저(Next.js SSR/hydration 환경)에서 깨지지 않아야 한다 — 이것이 본 작업의 양보 불가 핵심이자 최대 리스크다.

User Story 는 검증된 풀스택 관통 전략(설계 §5)에 따라 우선순위화한다: **projects 도메인을 먼저 끝까지 관통하며 핵심 리스크를 증명**한 뒤 나머지 도메인으로 복제한다.

### User Story 1 - 작품 벽 → 집필실: 진짜 페이지 분할 + 한글 집필 (Priority: P1) 🎯 MVP

작가가 브라우저에서 로그인하면 **작품 벽**(작품 카드 목록)을 보고, 새 작품을 만들거나 기존 작품을 열어 **서랍형 집필실**로 들어간다. 집필실에서 본문을 쓰면 **원고지처럼 페이지가 실시간으로 나뉘어** 보이고, **한글이 자모 분리·깨짐 없이** 입력되며, 변경은 자동 저장되어 다시 접속해도 보존된다. 화면 전환은 브라우저 URL 로 표현되어 새로고침·뒤로가기·딥링크가 동작한다.

**Why this priority**: desktop 의 페이지 분할(CSS `column-wrap` + 기하 계산 + ResizeObserver)과 TipTap 한글 IME 가 Next.js 환경에서 동작하는지가 전체 포팅의 성패를 가른다(설계 §10). 이 한 도메인(projects/documents)을 backend(014)→데이터계층→화면→연동까지 풀스택으로 먼저 관통해 패턴과 핵심 동작을 증명해야, 나머지(memos/logs/sessions)를 안전하게 복제할 수 있다. 주변 기능을 먼저 쌓고 핵심을 미루면 "이도저도 아닌" 산출물이 된다(작업 규율 §10).

**Independent Test**: 로그인한 작가가 작품을 생성→집필실 진입→한 페이지 분량 이상 한글 본문 입력 시, (1) 본문이 페이지 단위로 나뉘어 렌더되고 (2) 빠른 한글 타자·조합 중 서식 토글·한자 변환·Backspace 분해 4케이스가 깨지지 않고 (3) 새로고침/재로그인 후 본문이 보존되며 (4) URL 로 해당 집필실에 딥링크/뒤로가기가 동작함을 브라우저에서 확인.

**Acceptance Scenarios**:

1. **Given** 로그인한 작가가 작품이 없을 때, **When** 작품 벽에 진입하면, **Then** 빈 상태 안내와 "새 작품" 진입점이 보인다.
2. **Given** 작품 벽, **When** 새 작품을 만들면, **Then** 작품 카드가 목록에 나타나고(서버 영속) 카드에 "다음 장면" 등 메타가 표시된다.
3. **Given** 작품 카드, **When** 작품을 열면, **Then** 그 작품의 집필실 URL 로 이동하고 저장된 본문이 로드된다.
4. **Given** 집필실, **When** 작가가 한 페이지를 넘는 분량의 본문을 입력하면, **Then** 본문이 실시간으로 페이지(원고지) 단위로 나뉘어 보인다.
5. **Given** 집필실에서 한글 입력 중, **When** 빠른 타자/조합 중 서식 토글/한자 변환/Backspace 분해를 수행하면, **Then** 자모 분리·중복·유실 없이 정상 입력된다(PoC 0-1 4케이스).
6. **Given** 본문을 수정한 상태, **When** 자동 저장이 동작하고 새로고침하면, **Then** 마지막 저장 내용이 보존된다.
7. **Given** 집필실 URL, **When** 새로고침 또는 브라우저 뒤로/앞으로를 누르면, **Then** 화면 상태가 URL 과 일관되게 복원된다.
8. **Given** 작가 A 가 로그인하지 않았을 때, **When** 보호된 화면 URL 로 직접 접근하면, **Then** 로그인으로 리다이렉트된다(005 가드 재사용).

### User Story 2 - 곁쪽지(메모): 집필실 서랍 + 곁쪽지 책상 + 고정 (Priority: P2)

작가가 집필 중 떠오른 생각을 **곁쪽지**로 빠르게 적고, 집필실의 **서랍**에서 그 작품의 곁쪽지를 보며, 하나를 **고정**해 항상 위에 둔다. 별도 **곁쪽지 책상**(메모 인박스)에서 흩어진 곁쪽지를 작품·태그로 정리·연결한다. 고정·연결 상태는 서버에 영속된다(014 pin/curation).

**Why this priority**: 곁쪽지는 작가의 재진입 맥락 유지에 핵심이나, US1 의 데이터계층·라우팅 패턴이 확립된 뒤 복제하는 것이 안전하다. 014 의 pin endpoint·작품별 메모 조회를 소비한다.

**Independent Test**: 집필실 서랍에서 그 작품의 곁쪽지를 보고, 하나를 고정하면 서버 반영 후 목록 상단/표식에 반영되며, 다른 곁쪽지를 고정하면 이전 고정이 자동 해제됨(작품당 1개)을 확인. 곁쪽지 책상에서 미분류 메모를 작품에 연결하면 집필실 서랍에 나타남을 확인.

**Acceptance Scenarios**:

1. **Given** 집필실, **When** 빠른 캡처로 곁쪽지를 적으면, **Then** 그 곁쪽지가 서버에 저장되고 서랍/책상에서 보인다.
2. **Given** 작품에 연결된 곁쪽지 2개, **When** 하나를 고정하면, **Then** 그 곁쪽지가 고정 표시되고, 다른 하나를 고정하면 이전 고정이 해제된다(작품당 1개).
3. **Given** 곁쪽지 책상의 미분류 메모, **When** 작품에 연결하면, **Then** 그 작품 집필실 서랍에 나타난다.
4. **Given** 곁쪽지 책상, **When** 작품/태그로 필터하면, **Then** 해당 곁쪽지만 표시된다.

### User Story 3 - 집필 기록 + 작업 세션 (Priority: P3)

작가가 집필실에 들어가 있는 동안 **작업 세션**이 추적되고, 작업을 마치며 **집필 기록**을 남기면 기록과 세션이 함께 저장된다. **집필 기록 화면**에서 작품별 진척(최근 기록·누적 작업 시간)을 한눈에 본다(014 logs/sessions 소비).

**Why this priority**: 기록·세션은 표시·추적 성격이 강해 핵심 집필 경험(US1) 이후. desktop 의 화면 진입/이탈을 web 이벤트(라우트 이동·탭 가시성)로 매핑하는 결정이 필요하다.

**Independent Test**: 집필실 진입 시 세션 시작, 이탈/탭 종료 시 종료(짧은 세션 폐기), "작업 종료+기록" 시 기록 저장 + 세션 보존을 확인. 기록 화면에서 작품별 최근 기록·누적 시간이 표시됨을 확인.

**Acceptance Scenarios**:

1. **Given** 작가가 집필실에 진입하면, **When** 작업을 시작하면, **Then** 작업 세션이 시작된다.
2. **Given** 집필 중, **When** 작가가 "작업 종료+기록"으로 한 줄을 남기면, **Then** 집필 기록이 저장되고 세션이 보존된다.
3. **Given** 집필실 URL 을 벗어나거나 탭/창을 닫으면, **When** 종료 신호가 전달되면, **Then** 세션이 종료되고 너무 짧은 세션은 통계에 포함되지 않는다. (백그라운드 탭 전환은 종료하지 않고 계속 작업 중으로 간주)
4. **Given** 집필 기록 화면, **When** 진입하면, **Then** 작품별 최근 기록과 누적 작업 시간이 카드로 표시된다.

### User Story 4 - 문의 (Priority: P4)

작가가 **문의 화면**에서 이메일로 문의를 보내거나 카카오톡 채널 링크로 이동한다. 데스크탑의 외부 링크 열기(`shell.openExternal`)는 브라우저의 새 탭 열기로, 문의에 첨부되던 앱 버전·OS 메타는 web 컨텍스트에서 재구성된다.

**Why this priority**: 보조 기능이며 데이터 의존이 적어 마지막. electron 전용 호출 교체의 대표 사례.

**Acceptance Scenarios**:

1. **Given** 문의 화면, **When** 이메일 폼을 제출하면, **Then** 문의가 전송된다(메타는 web 컨텍스트에서 생성).
2. **Given** 문의 화면, **When** 카카오톡 채널 링크를 누르면, **Then** 새 탭/창으로 외부 링크가 열린다.

### Edge Cases

- **페이지 분할 SSR/hydration**: 서버 렌더 시점엔 본문 높이를 알 수 없어 페이지 수 계산이 클라이언트에서만 가능 — 첫 페인트에 깜빡임/레이아웃 점프가 없어야 한다(브라우저 레이아웃 의존 컴포넌트는 client 경계).
- **브라우저 버전 차이**: desktop 은 Electron Chromium 고정, web 은 다양한 브라우저 — `column-wrap` 등 CSS 동작 차이 검증 필요.
- **한글 IME 회귀**: TipTap 이식 후 PoC 0-1 4케이스(빠른 타자/조합 중 서식/한자 변환/Backspace 분해) 재검증.
- **다중 탭·기기 동시 편집**: 같은 문서를 두 곳에서 열어 저장 충돌 시(409 DOCUMENT_VERSION_CONFLICT) 사용자에게 충돌이 안내된다(client.ts 기존 처리 재사용).
- **탭 종료 시 세션**: 브라우저 비정상 종료로 종료 신호 누락 시, 서버 측 dangling 정리(014)가 통계를 보정한다.
- **폰트/한글 렌더**: `next/font` 한국어 subset 미지원 → 시스템 fallback chain 으로 한글 본문이 라이트/다크·iOS/Android 에서 정상 표시.
- **미로그인 접근**: 보호 화면 직접 URL 접근 시 로그인 리다이렉트(005 가드).

## Requirements *(mandatory)*

### Functional Requirements

**데이터 계층·아키텍처 (전 도메인 공통)**

- **FR-001**: 화면은 데이터 접근을 desktop 과 **동일한 인터페이스 경계**(`electronAPI` 형태)를 통해 수행하되, 그 구현은 IPC 가 아닌 서버 REST API(014) 호출로 교체되어야 한다(설계 §3, 화면은 구현을 모른다).
- **FR-002**: 데이터 계층은 기존 web HTTP 배관(`client.ts` — credentials 포함·401 자동 갱신)과 서버 상태 캐시(React Query) 위에 구현되어야 한다.
- **FR-003**: desktop 의 27 IPC 채널 중 본 범위 도메인(projects·documents·memos·logs·sessions·contact·settings·shell)은 014 의 `contracts/ipc-rest-mapping.md` 에 따라 web 호출로 매핑되어야 한다.

**라우팅 (전 도메인 공통)**

- **FR-004**: desktop 의 화면 전환(screen-state)은 브라우저 **URL 라우팅**으로 변환되어야 하며, 각 주요 화면(작품 벽·집필실·곁쪽지 책상·기록·문의)이 딥링크 가능한 URL 을 가져야 한다.
- **FR-005**: 새로고침·뒤로/앞으로 가기 시 화면 상태가 URL 과 일관되게 복원되어야 한다.
- **FR-006**: 기존 frontend 006 화면(작품/문서/메모 화면)은 이식된 desktop 화면으로 폐기·교체되어야 한다.

**인증 (전 도메인 공통)**

- **FR-007**: 로그인·인증 가드·인증 쿠키·세션 갱신은 기존 frontend 005 자산을 재사용해야 하며, 보호 화면은 미로그인 시 로그인으로 리다이렉트되어야 한다.

**US1 — 작품 벽 + 집필실 + 페이지 분할**

- **FR-008**: 작가는 작품 벽에서 본인 작품 목록(카드)을 보고, 새 작품을 만들고, 작품을 열 수 있어야 한다. 카드에는 desktop 과 동등한 표시값(다음 장면 등)이 보여야 한다.
- **FR-009**: 집필실은 저장된 본문을 로드해 표시하고, 본문 입력 시 **실시간으로 페이지(원고지) 단위로 나뉘어** 렌더되어야 한다(desktop 페이지 분할 동작 동등).
- **FR-010**: 한글 입력이 자모 분리·중복·유실 없이 동작해야 한다(PoC 0-1 4케이스 통과).
- **FR-011**: 본문 변경은 자동 저장되어 재접속·새로고침 후 보존되어야 하며, 저장 충돌 시 사용자에게 안내되어야 한다.
- **FR-012**: 보기 설정(줌·줄노트·테마 등)은 desktop 과 동등하게 제공되어야 한다.

**US2 — 곁쪽지**

- **FR-013**: 작가는 집필 중 곁쪽지를 빠르게 캡처할 수 있어야 하고, 집필실 서랍에서 그 작품의 곁쪽지를 볼 수 있어야 한다.
- **FR-014**: 작가는 작품당 곁쪽지 하나를 고정할 수 있어야 하며, 새 고정 시 이전 고정이 자동 해제되어야 한다(014 불변식 소비).
- **FR-015**: 곁쪽지 책상에서 곁쪽지를 작품/태그로 정리·연결·필터할 수 있어야 한다.

**US3 — 집필 기록 + 작업 세션**

- **FR-016**: 작가가 집필실에서 작업하는 동안 작업 세션이 추적되어야 한다(시작/종료, 014 소비).
- **FR-017**: 작가는 "작업 종료+기록"으로 집필 기록을 남길 수 있고, 기록과 세션이 함께 저장되어야 한다.
- **FR-018**: 집필 기록 화면에서 작품별 최근 기록·누적 작업 시간이 표시되어야 한다.
- **FR-019**: 작업 세션 종료는 **(1) 집필실 라우트 이탈 시 + (2) 탭/창 닫기 시(best-effort)** 트리거되어야 한다. 백그라운드 탭 전환(가시성 변화)은 종료 트리거가 아니다(계속 작업 중 간주). 탭 닫기로 종료 신호가 유실되면 서버 측 dangling 정리(014)가 통계를 보정한다.

**US4 — 문의 + electron 전용 교체**

- **FR-020**: 작가는 문의 화면에서 이메일 문의를 보낼 수 있어야 하며, 첨부 메타(앱/환경 정보)는 web 컨텍스트에서 생성되어야 한다.
- **FR-021**: 외부 링크(카카오 채널 등)는 브라우저 새 탭/창 열기로 동작해야 한다(`shell.openExternal` 대체).

**비기능 (한글·폰트)**

- **FR-022**: 한글 본문이 라이트/다크 테마와 주요 모바일/데스크톱 브라우저에서 폰트 fallback chain 으로 정상 표시되어야 한다.

### Key Entities *(데이터는 014 backend 가 소유 — 본 작업은 소비)*

- **작품 카드 / 문서 / 곁쪽지 / 집필 기록 / 작업 세션**: 모두 하위 작업 1(014)의 서버 데이터를 조회·표시·갱신하는 **클라이언트 뷰**다. 본 작업은 새 영속 엔티티를 만들지 않으며, 014 REST 계약을 통해 읽고 쓴다. 클라이언트 로컬 상태(보기 설정·UI 토글)는 기기 로컬에 보관한다(가정 참조).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 작가가 **설치 없이 브라우저**에서 desktop 의 핵심 집필 흐름(작품 생성→집필실→페이지 분할 본문 작성→자동 저장→재접속 보존)을 끝까지 완수할 수 있다.
- **SC-002**: 진짜 페이지 분할이 브라우저에서 동작한다 — 한 페이지를 넘는 본문이 페이지 단위로 나뉘어 보이고, 주요 브라우저 라이트/다크에서 한글 본문이 정상 렌더된다.
- **SC-003**: 한글 입력 4케이스(빠른 타자/조합 중 서식/한자 변환/Backspace 분해)가 회귀 없이 통과한다.
- **SC-004**: desktop 의 대상 화면(작품 벽·집필실·곁쪽지(서랍+책상)·기록·문의)이 web 에서 기능 동등하게 동작하고, 모든 데이터가 계정 단위로 서버에 영속되어 다른 기기에서 동일하게 보인다.
- **SC-005**: 모든 주요 화면이 딥링크 가능한 URL 을 가지며 새로고침·뒤로가기에서 상태가 깨지지 않는다.
- **SC-006**: 미로그인 사용자가 보호 화면에 접근하면 100% 로그인으로 리다이렉트된다.

## Assumptions

- **범위**: 본 spec 은 하위 작업 2(front 이식)만. 등장인물 UI·모바일 캡처·로컬 `.db` 가져오기·배포(Vercel/Render)는 범위 밖(하위 작업 3·4). 백엔드는 014 계약을 소비만 하며 변경하지 않는다(필요 시 발견은 별도 트랙으로 surfacing).
- **베타 = desktop 1:1**: 화면·UX 는 desktop 최신본(009 재디자인·010 페이지 분할)을 정본으로 1:1 이식하되, web 고유 요소(딥링크 URL·뒤로가기·다중 탭)는 표준 web 관례로 보강한다.
- **데이터 계층 청사진**: desktop `preload.ts`/`ipc/contract.ts` 의 `electronAPI` 인터페이스를 web 구현체의 청사진으로 삼고, 014 `contracts/ipc-rest-mapping.md` 의 ✅/🧩 매핑을 계약으로 사용한다. "마지막 문장"·LogCard/ProjectCard 집계는 014 R6 대로 front 가 조립한다.
- **재사용 자산(005)**: `client.ts`(401 reactive refresh·credentials)·`guard`(useAuthGuard)·`QueryProvider`·`next.config` same-origin 프록시·로그인/회원가입 화면을 그대로 재사용.
- **로컬 설정 보관**: 보기 설정(테마·줌·줄노트·집필 모드)은 기존 zustand preferences(localStorage persist)에 보관하고 서버 동기화하지 않는다(실제 집필 데이터만 서버). 서버 동기화가 필요하면 후속에서.
- **검증 전략(설계 §5)**: projects 도메인을 풀스택으로 먼저 관통(US1)해 페이지 분할·IME·라우팅·데이터계층 패턴을 증명한 뒤 memos/logs/sessions/contact 로 복제. 핵심(페이지 분할+한글) 미증명 상태에서 광범위 화면 이식·다수 커밋 누적 금지(작업 규율 §10).
- **페이지 분할 동작 가정**: desktop 의 CSS `column-wrap` + 기하 계산 방식을 우선 그대로 이식해 검증하며, Next SSR/hydration 환경에서 부적합 시 대안은 plan/PoC 에서 결정(별도 트랙 surfacing).
