## User Scenarios & Testing *(mandatory)*

### User Story 1 - 전역 허브에서 모든 보드 찾기 (Priority: P1)

작가가 상단 `보드` 메뉴를 열면, 작품·시리즈·아이디어를 가로질러 **내 모든 보드**가 **출처(소속) 라벨**과 함께 최근 연 순으로 보인다. 보드가 어디에 매달렸는지(어느 작품·시리즈인지, 또는 "아이디어"인지)를 라벨로 식별하고, 검색어로 작품명·시리즈명·보드명을 가로질러 좁힌다.

**Why this priority**: PRD §5.4 전역 진입의 의도 = "그 보드 어디 있더라". 보드가 작품·시리즈·아이디어에 흩어지면 이름만으로 출처를 못 찾는다 → 소속 라벨이 핵심 가치. 다른 모든 스토리(생성·매핑)의 결과를 보는 화면이기도 하다.

**Independent Test**: 작품 보드·시리즈 보드·아이디어 보드를 각 1개씩 둔 상태에서 `보드` 메뉴를 열어, 셋이 각자 라벨("○○작품"/"○○시리즈"/"아이디어")과 함께 최근순으로 보이고, 검색어로 셋을 가로질러 필터됨을 확인.

**Acceptance Scenarios**:

1. **Given** 작품·시리즈·아이디어 보드가 섞여 있을 때, **When** 전역 보드 허브를 열면, **Then** 모든 보드가 소속 라벨과 함께 최근 연 순으로 나열된다.
2. **Given** 허브에 여러 보드가 있을 때, **When** 작품명/시리즈명/보드명 일부를 검색하면, **Then** 그 키워드에 걸리는 보드만 남는다(아이디어 보드는 보드명으로 매칭, 소속이 비어도 누락되지 않음).

---

### User Story 2 - 한 작품/시리즈에 보드 여러 개 + 단일 소유 (Priority: P1)

작가가 한 작품 또는 시리즈에 **여러 개의 보드**(예: "인물 관계 보드", "1부 사건 흐름")를 둘 수 있다. 한 보드는 작품·시리즈 **둘 중 한 곳에만** 소속되거나, 아무 데도 소속되지 않는다(아이디어).

**Why this priority**: PRD §4·§5 확정 = Board ↔ (시리즈/작품) 1:N. 현재는 대상당 보드 1개로 제한돼 관점별 보드 분리가 불가능하다. 단일 소유는 "이 보드는 어디에 쓸 건가요?"의 단일 선택과 정합한다.

**Independent Test**: 한 작품에 보드 2개를 생성해 둘 다 그 작품 라벨로 보이고(충돌 안내 없음), 한 보드의 소속을 작품→시리즈로 바꾸면 작품 소속이 사라지고 시리즈 소속만 남음을 확인.

**Acceptance Scenarios**:

1. **Given** 한 작품에 이미 보드 1개가 있을 때, **When** 같은 작품을 대상으로 보드를 하나 더 만들면, **Then** 두 보드가 모두 그 작품 소속으로 생성된다(매핑 충돌 안내 없음).
2. **Given** 작품에 소속된 보드가 있을 때, **When** 소속을 시리즈로 변경하면, **Then** 그 보드는 시리즈 소속이 되고 작품 소속은 동시에 유지되지 않는다.
3. **Given** 보드 생성/연결 시, **When** 본인 소유가 아니거나 존재하지 않는 작품·시리즈를 대상으로 지정하면, **Then** 거부되고 보드는 변경되지 않는다.

---

### User Story 3 - 아이디어 보드 & 나중에 붙이기 (Priority: P2)

작가가 아직 어느 작품이 될지 모르는 구상을, 소속 없이 **아이디어 보드**로 먼저 펼친다. 나중에 작품/시리즈가 정해지면 그 아이디어 보드를 작품/시리즈에 **연결**한다. 전역 메뉴에서 보드를 만들 때는 "이 보드는 어디에 쓸 건가요? — 이 작품 / 시리즈 전체 / 아이디어"를 작가의 언어로 고른다.

**Why this priority**: PRD §5.3 = "먼저 끄적이고 나중에 정리". 작품이 정해져야만 보드를 만들 수 있으면 초기 구상을 막는다. "나중에 붙이기"는 미아 보드 방지의 필수 짝 기능.

**Independent Test**: "아이디어"로 보드를 만들어 "아이디어" 라벨로 보이고, 그 보드의 "작품/시리즈에 연결"로 한 작품을 고르면 라벨이 그 작품명으로 바뀜을 확인.

**Acceptance Scenarios**:

1. **Given** 전역 메뉴에서 보드 만들기를 누르면, **When** "아이디어"를 고르고 이름을 입력하면, **Then** 소속 없는 아이디어 보드가 만들어지고 허브에 "아이디어" 라벨로 보인다.
2. **Given** 아이디어 보드가 있을 때, **When** "작품/시리즈에 연결"로 대상을 고르면, **Then** 그 보드의 소속 라벨이 고른 대상으로 갱신된다.
3. **Given** 전역 메뉴에서 보드 만들기를 누르면, **When** "이 작품" 또는 "시리즈 전체"를 고르고 대상·이름을 입력하면, **Then** 그 대상 소속 보드가 만들어진다.

---

### User Story 4 - 대상 삭제 시 보드 보존 (Priority: P3)

작가가 작품 또는 시리즈를 삭제해도, 거기 매달려 있던 보드는 **사라지지 않고 아이디어 보드로 남는다**(소속만 비워짐).

**Why this priority**: PRD §5.3 "대상 삭제 시 보드 보존". 보드 안의 카드·연결(설계 산출물)이 작품 정리로 함께 증발하면 안 된다. 데이터 안전.

**Independent Test**: 작품에 소속된 보드를 둔 뒤 그 작품을 (하드)삭제하고, 보드가 "아이디어" 라벨로 허브에 남아 카드·연결이 보존됨을 확인.

**Acceptance Scenarios**:

1. **Given** 작품/시리즈에 소속된 보드가 있을 때, **When** 그 작품/시리즈가 삭제되면, **Then** 보드는 아이디어 보드(소속 없음)로 남고 카드·연결은 보존된다.

---

### Edge Cases

- 같은 작품에 보드 N개가 있을 때 허브에서 모두 같은 작품 라벨로 구분 없이 보인다(이름으로 식별) — 1:N 정상.
- 검색어가 비면 전체 목록(최근순)으로 복귀.
- 아이디어 보드는 소속이 없어 검색에서 보드명으로만 잡힌다 — 소속 null이어도 목록·검색에서 누락 금지.
- 보드 소속을 "아이디어"로 되돌리기(연결 해제)도 허용(소속 지정/해제는 같은 경로).
- owner 종류·대상 짝이 불완전(종류만 있고 대상 없음 등)하면 거부.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 보드는 **작품(1개) · 시리즈(1개) · 소속 없음(아이디어)** 중 정확히 하나의 소속 상태를 가진다(작품과 시리즈에 동시 소속 불가).
- **FR-002**: 한 작품 또는 한 시리즈는 **보드를 여러 개**(1:N) 가질 수 있다(대상당 보드 1개 제한 폐지).
- **FR-003**: 보드 생성 시 소속은 **선택**이며, 생략하면 아이디어 보드(소속 없음)로 만들어진다.
- **FR-004**: 작가는 아이디어 보드를 나중에 작품 또는 시리즈에 **연결**(소속 지정)하거나 소속을 **해제**(아이디어로 되돌리기)할 수 있다.
- **FR-005**: 보드 소속 대상은 **본인 소유 작품/시리즈**여야 한다. 본인 것이 아니거나 존재하지 않으면 거부하고 보드를 변경하지 않는다.
- **FR-006**: 전역 보드 허브는 **내 모든 보드**(작품·시리즈·아이디어 가로질러)를 **소속 라벨**(작품명 · 시리즈명 · "아이디어")과 함께 **최근 연 순**으로 보여준다.
- **FR-007**: 전역 허브 검색은 **작품명 / 시리즈명 / 보드명**을 OR로 매칭한다. 아이디어 보드는 보드명으로 매칭되며, 소속이 없어도 목록·검색에서 누락되지 않는다.
- **FR-008**: 전역 메뉴에서 보드를 만들 때 작가는 **"이 작품 / 시리즈 전체 / 아이디어"**를 작가의 언어로 고른다("작품" 또는 "시리즈"를 고르면 대상을 선택). 내부 용어(예: owner_type)는 화면에 노출하지 않는다.
- **FR-009**: 작품 또는 시리즈가 삭제되면, 그 대상에 소속된 보드는 **소속만 비워져 아이디어 보드로 보존**되고 보드·카드·연결은 삭제되지 않는다.
- **FR-010**: 보드의 카드·연결·뷰포트(열기/배치/연결/화면상태)와 캡처 메모·인물 도메인은 본 변경의 영향을 받지 않는다(Track A/B 동작 보존).
- **FR-011**: 화면 문구는 `board-ux-worksheet.md` §5를 따른다. 폐기·내부 용어(`node`/`edge`/`메모`/`인물`(구 메뉴)/`viewport`/`owner_type`)는 화면에 노출되지 않는다.

### Key Entities *(include if feature involves data)*

- **보드(Board)**: 작가 소유의 플롯 설계 캔버스. **소속(owner)** = 작품 · 시리즈 · 없음(아이디어) 중 하나. 한 소속 대상은 여러 보드를 가질 수 있다(1:N). 카드·연결을 담는다(Track A/B).
- **소속(owner)**: 보드가 매달리는 대상. 종류(작품/시리즈/없음)와 대상 식별자가 **항상 짝으로** 채워지거나 함께 빈다. 진짜 외래키가 아니므로 대상 존재·소유는 검증으로 보장하고, 대상 삭제 시 보존을 위해 소속을 비운다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 작가가 전역 보드 허브에서 작품·시리즈·아이디어 보드를 **소속 라벨과 함께** 한 화면에서 식별할 수 있다(라벨 누락 0건).
- **SC-002**: 한 작품/시리즈에 보드를 **2개 이상** 만들 수 있다(충돌·거부 0건).
- **SC-003**: 검색으로 **작품명·시리즈명·보드명**을 가로질러 보드를 좁힐 수 있고, 아이디어 보드도 보드명으로 검색된다(누락 0건).
- **SC-004**: 아이디어 보드를 만든 뒤 작품/시리즈에 **연결**하면 라벨이 즉시 갱신된다.
- **SC-005**: 작품/시리즈를 삭제해도 그 보드와 카드·연결이 **100% 보존**되고 아이디어 보드로 남는다.
- **SC-006**: 기존 보드 열기·카드 배치·연결·화면 상태 복원이 **회귀 없이** 동작한다(Track A/B 보존).

## Assumptions

- **단일 사용자** 기준(동시 편집·사용자별 충돌은 비목표, PRD §11).
- 보드 도메인(테이블·API)은 **develop·main 미배포**(Track B 실증) → API 계약·마이그레이션 변경에도 운영 위험 0, 보드 트랙 누적분과 원자적 동반 merge.
- **내부 보드 탭**(작품/시리즈 상세 안 생성·관리)과 **집필 중 보드 참조**(분할 뷰·마지막 본 보드 기억)는 **본 트랙 범위 밖**(후속 증분, 호스트 UI 결정 필요).
- 검색은 **클라이언트 필터**(허브가 라벨 포함 전체 목록을 받아 즉시 필터, 단일 사용자·소량) — 별도 검색 질의 없음.
- 작품/시리즈 "삭제"의 의미(하드 삭제 vs 보관)는 구현 단계에서 기존 동작 실측 후, 대상이 실제로 사라지는 경우에 한해 보드 소속을 비운다.
