# Feature Specification: 작품 카테고리 분류

**Feature Branch**: `032-project-categories`

**Created**: 2026-06-22

**Status**: Draft

**Input**: User description: "작품 메뉴에 작품을 카테고리별로 분류할 수 있는 기능을 추가하고싶어. 지금은 작품 페이지에 들어가면 생성한 작품이 내림차순으로 정렬되어서 나오는데, 카테고리를 추가해서 하나의 디렉토리 안에 작품들을 모아둘 수 있는 기능이 추가되었으면 좋겠어. 카테고리 뎁스는 우선은 1뎁스로 한정하고 설계만 N뎁스가 가능하도록 설계를 진행하도록."

## User Scenarios & Testing *(mandatory)*

작가가 작품 수가 늘면서 작품 페이지(`/library`)에 작품들이 수정시각 내림차순 한 줄로만 쌓여 원하는 작품을 찾기 어렵다. 작가가 직접 만든 "카테고리(디렉토리)"로 작품을 묶어, 작품 페이지를 카테고리별로 정돈해서 보고자 한다.

### User Story 1 - 카테고리를 만들고 작품을 분류해서 본다 (Priority: P1)

작가가 자신의 작품을 묶을 카테고리(예: "장편 판타지", "단편 모음")를 만들고, 작품을 그 카테고리에 넣은 뒤, 작품 페이지에서 작품들이 카테고리별로 묶여 표시되는 것을 본다. 카테고리를 정하지 않은 작품(미분류)은 사라지지 않고 그대로 보인다.

**Why this priority**: 본 기능의 본질 — "디렉토리에 작품을 모아둔다". 카테고리 생성·배정·카테고리별 표시 세 가지만 있어도 작가는 즉시 정돈된 작품 페이지라는 가치를 얻는다. 이 슬라이스 하나만 구현해도 최소 동작 제품(MVP)이 성립한다.

**Independent Test**: 카테고리 1개를 만들고 → 기존 작품 1개를 그 카테고리에 배정하고 → 작품 페이지를 새로고침했을 때, 그 작품이 카테고리 묶음 아래에 보이고 카테고리 없는 다른 작품들은 미분류(루트)에 그대로 보이는지로 단독 검증 가능.

**Acceptance Scenarios**:

1. **Given** 카테고리가 하나도 없는 작가가 작품 페이지에 있을 때, **When** "새 카테고리"를 만들고 이름을 입력하면, **Then** 빈 카테고리가 작품 페이지에 나타난다.
2. **Given** 미분류 작품 3개와 카테고리 1개가 있을 때, **When** 작품 1개를 그 카테고리에 배정하면, **Then** 그 작품은 카테고리 묶음 아래로 이동하고 나머지 2개는 미분류(최상위)에 남는다.
3. **Given** 카테고리 도입 이전에 만든 작품들이 있을 때, **When** 작품 페이지에 진입하면, **Then** 기존 작품들이 누락 없이 미분류로 그대로 표시된다.
4. **Given** 한 카테고리 안에 작품 여러 개가 있을 때, **When** 작품 페이지를 볼 때, **Then** 그 카테고리 안의 작품들은 기존과 동일하게 수정시각 내림차순으로 정렬된다.

---

### User Story 2 - 카테고리를 관리한다 (이름 변경 · 삭제 · 작품 이동) (Priority: P2)

작가가 만든 카테고리의 이름을 바꾸고, 더 이상 필요 없는 카테고리를 삭제하고, 작품을 다른 카테고리로 옮기거나 카테고리에서 빼서 미분류로 되돌린다. 카테고리를 삭제해도 그 안의 작품은 사라지지 않는다.

**Why this priority**: 분류는 한 번 만들고 끝이 아니라 계속 손보는 작업이다. 다만 P1(생성·배정·표시) 없이는 의미가 없으므로 후순위. 작품 무손실(카테고리 삭제 ≠ 작품 삭제)이 핵심 안전 요건.

**Independent Test**: 작품 2개가 든 카테고리를 삭제했을 때, 그 작품 2개가 삭제되지 않고 미분류로 전환되어 작품 페이지에 그대로 보이는지로 단독 검증 가능.

**Acceptance Scenarios**:

1. **Given** 이름이 "임시"인 카테고리가 있을 때, **When** 이름을 "장편 판타지"로 변경하면, **Then** 작품 페이지의 카테고리 이름이 즉시 바뀐다.
2. **Given** 작품 2개가 든 카테고리를 삭제하려 할 때, **When** 삭제를 확정하면, **Then** 카테고리는 사라지지만 그 안에 있던 작품 2개는 미분류로 전환되어 그대로 남는다.
3. **Given** 카테고리 A에 든 작품을 카테고리 B로 옮길 때, **When** 이동을 수행하면, **Then** 그 작품은 A에서 빠지고 B에만 속한다(동시에 두 카테고리에 속하지 않는다).
4. **Given** 카테고리에 속한 작품을 카테고리에서 빼면, **When** "분류 없음"으로 되돌리면, **Then** 그 작품은 미분류(루트)로 이동한다.

### User Story 3 - 작품 페이지에서 카테고리를 탐색한다 (Priority: P3)

작가가 작품 페이지에서 카테고리 단위로 작품 묶음을 훑어보고, 필요하면 카테고리를 접거나 펼쳐 화면을 정돈한다. 미분류 작품 영역도 명확히 구분되어 보인다.

**Why this priority**: 표시·탐색 편의 향상. P1의 기본 표시가 있으면 동작은 하므로 후순위지만, 카테고리·작품 수가 많을 때 체감 가치가 크다.

**Independent Test**: 카테고리 2개와 미분류 작품이 있을 때, 한 카테고리를 접으면 그 안 작품이 화면에서 숨겨지고 다른 카테고리·미분류는 그대로 보이는지로 단독 검증 가능.

**Acceptance Scenarios**:

1. **Given** 작품이 든 카테고리가 펼쳐져 있을 때, **When** 카테고리를 접으면, **Then** 그 안의 작품 카드들이 숨겨지고 카테고리 머리(이름·작품 수)만 남는다.
2. **Given** 미분류 작품과 카테고리 작품이 섞여 있을 때, **When** 작품 페이지를 볼 때, **Then** 미분류 영역과 각 카테고리 영역이 시각적으로 구분되어 보인다.

---

### Edge Cases

- **카테고리 삭제 시 작품 보존**: 작품이 든 카테고리를 삭제하면 작품은 미분류로 전환된다(작품 영구 삭제 아님). → FR-007
- **작품 영구 삭제(기존 동작)**: 카테고리에 속한 작품을 영구 삭제하면 그 작품만 사라지고 카테고리 자체는 유지된다.
- **빈 카테고리**: 작품이 0개인 카테고리도 작품 페이지에 빈 채로 표시된다(작가가 미리 만들어 둘 수 있게).
- **미분류 작품 0개**: 모든 작품이 카테고리에 배정되어 미분류가 비면, 미분류 영역은 표시하지 않거나 빈 안내로 처리한다.
- **카테고리 이름 검증**: 빈 이름·공백만·길이 초과는 거부한다.
- **1뎁스 제한 강제**: 카테고리 안에 하위 카테고리를 만들려는 시도는 현재 차단된다(v1). → FR-010
- **보관(archive)된 작품과 카테고리**: 보관된 작품도 카테고리 연결을 유지하되, v1의 카테고리별 묶음 표시는 활성 작품 화면에만 적용한다. 보관 목록은 기존 동작을 유지한다. → Assumptions
- **다른 작가 격리**: 한 작가는 다른 작가의 카테고리·작품을 보거나 그 안에 배정할 수 없다. → FR-002

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 작가는 자신의 작품을 묶을 카테고리를 이름을 지정해 생성할 수 있어야 한다.
- **FR-002**: 카테고리와 그 안의 작품은 작가 단위로 격리되어야 한다 — 한 작가는 다른 작가의 카테고리·작품을 조회하거나 변경할 수 없다.
- **FR-003**: 작가는 작품을 카테고리에 배정하고, 다른 카테고리로 옮기고, 카테고리에서 빼서 미분류로 되돌릴 수 있어야 한다. **작품은 동시에 최대 1개 카테고리에만 속한다**(폴더형).
- **FR-004**: 작품 페이지는 작품을 카테고리별로 묶어 표시해야 하며, 어느 카테고리에도 속하지 않은 작품(미분류)은 최상위(루트)에 함께 표시해야 한다.
- **FR-005**: 각 카테고리(및 미분류) 묶음 안의 작품은 기존과 동일하게 수정시각 내림차순으로 정렬되어야 한다(기존 정렬 동작 보존).
- **FR-006**: 작가는 카테고리 이름을 변경할 수 있어야 한다.
- **FR-007**: 작가는 카테고리를 삭제할 수 있어야 한다. 카테고리 삭제 시 그 안의 작품은 **삭제되지 않고 미분류 상태로 전환**되어야 한다(작품 무손실).
- **FR-008**: 새 작품 생성 시 카테고리 지정은 **선택**이어야 한다 — 카테고리를 고르지 않으면 미분류로 생성된다.
- **FR-009**: 카테고리 도입 이후에도, 도입 이전에 만든 모든 작품은 손실 없이 미분류로 표시되어야 한다.
- **FR-010**: 현재 카테고리는 **1뎁스(최상위)만 허용**한다 — 카테고리 안에 하위 카테고리를 만들 수 없다. 단, 데이터 구조와 설계는 향후 **N뎁스(카테고리 중첩)로 확장 가능**해야 하며, 1뎁스 제한 해제가 데이터 구조 변경 없이 가능해야 한다.
- **FR-011**: 카테고리는 작품 페이지에서 안정적이고 예측 가능한 순서로 표시되어야 한다(작가가 의도한 순서 또는 명확한 기본 순서). 구체적 정렬 방식은 Assumptions 참조.
- **FR-012**: 카테고리 이름은 빈 값·공백만·과도한 길이를 거부해야 한다.
- **FR-013**: 작품 페이지는 카테고리·미분류 묶음과 함께, 각 카테고리에 속한 작품 수를 작가가 인지할 수 있어야 한다(예: 카테고리 머리에 작품 수 표시).

### Key Entities *(include if feature involves data)*

- **카테고리(Category)**: 작가가 작품을 묶기 위해 만든 디렉토리. 소유 작가, 이름, 표시 순서, 생성 시각을 가진다. **상위 카테고리 참조(parent)** 속성을 설계상 보유하되, v1에서는 항상 최상위(상위 없음)로 제한된다(N뎁스 확장 토대). 한 작가는 카테고리를 여러 개 가질 수 있다(작가 1:N 카테고리).
- **작품(Project, 기존 엔티티 확장)**: 기존 작품에 **소속 카테고리 참조**가 추가된다. 이 참조는 **비어 있을 수 있다(미분류)**. 작품은 최대 1개 카테고리에만 속한다(카테고리 1:N 작품). 기존 작품·정렬·보관 동작은 유지된다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 작가는 카테고리를 만들고 작품 하나를 그 안에 넣는 전체 과정을 처음 시도에 성공적으로 완료할 수 있다(별도 안내 없이).
- **SC-002**: 작품 페이지에서 모든 작품이 누락 없이 보인다 — 카테고리에 속한 작품은 해당 카테고리 아래, 속하지 않은 작품은 미분류에 100% 표시된다.
- **SC-003**: 작품이 든 카테고리를 삭제한 뒤에도 그 안에 있던 작품은 100% 보존되어 미분류로 표시된다(작품 유실 0건).
- **SC-004**: 카테고리 도입(데이터 변경) 후 기존 작품 100%가 그대로 미분류로 표시되며, 작품 본문·메타데이터·정렬에 회귀가 없다.
- **SC-005**: v1에서 하위 카테고리 생성 시도는 차단되며, 향후 N뎁스 허용으로의 전환이 데이터 구조 변경 없이(제한 규칙만 완화하여) 가능함이 설계로 확인된다.

## Assumptions

- **분류 방식 = 폴더형(확정, 사용자 컨펌 2026-06-22)**: 작품은 최대 1개 카테고리에만 속한다(태그형 다중 분류 아님). "하나의 디렉토리에 모아둔다"는 요청과 일치하며, N뎁스 트리 확장과 정합.
- **미분류 처리 = 허용 + 루트 표시 + 생성 시 선택(확정, 사용자 컨펌 2026-06-22)**: 카테고리 없는 작품을 허용하고 최상위에 표시하며, 새 작품 생성 시 카테고리 선택은 선택사항이다. 기존 작품에 기본 카테고리를 자동 배정하지 않는다(기존 작품 무손실).
- **카테고리 안 작품 정렬**: 기존 작품 페이지의 수정시각 내림차순 정렬을 카테고리 묶음 안에서 그대로 유지한다(FR-005).
- **카테고리 자체의 표시 순서(FR-011)**: 데이터에 순서 속성을 두되, v1에서 작가의 수동 재정렬(드래그) UI 포함 여부는 설계(plan) 단계에서 결정한다. 미정 시 기본은 생성 순서(또는 이름 오름차순)의 안정 정렬.
- **동일 레벨 카테고리 이름 중복**: v1에서는 같은 작가의 같은 레벨 내 카테고리 이름 중복을 막지 않는다(단순화). 필요 시 후속에서 강화.
- **보관(archive) 작품**: 카테고리별 묶음 표시는 활성 작품 화면에만 적용한다. 보관 목록 화면은 기존 동작을 유지한다(v1 범위 밖).
- **N뎁스 설계 방식(FR-010)**: 데이터는 상위 카테고리 참조를 보유하고, 1뎁스 제한은 애플리케이션 규칙으로 강제한다. 향후 N뎁스 허용은 데이터 구조 변경 없이 그 규칙만 완화하여 달성한다.
- **소유·인증 모델**: 카테고리는 기존 작품과 동일한 작가 소유·인증 모델(로그인 사용자 단위 격리)을 재사용한다.
- **모바일/데스크톱**: 본 명세는 작품 페이지(`/library`)의 분류 기능을 다루며, 화면 형태(반응형·모바일)는 기존 작품 페이지 동작 범위를 따른다.
- **인터랙션 방식 = 폴더 타일 드릴인(확정, 사용자 컨펌 2026-06-22)**: 작품 페이지 루트에 분류 묶음을 타일로 보이고, 묶음을 열고 들어가/나오는 디렉토리 메타포로 탐색한다(섹션 아코디언·사이드바형 대신). 작품을 묶음으로 옮기는 1차 동작은 카드를 묶음 타일로 끌어다 놓기(드래그 앤 드롭)이며, 터치·키보드용으로 카드 메뉴의 "이동"이 1급 대안으로 함께 제공된다. N뎁스(묶음 안 묶음) 확장에 가장 자연스러운 방식. 비교 프로토타입: `docs/research/2026-06-22-project-categories-interactions.html`.
- **사용자 노출 용어 = "시리즈"(확정, 사용자 컨펌 2026-06-22 dogfooding 후 "모음"→"시리즈" 변경)**: 본 명세는 개념 엔티티를 "카테고리"로 부르지만, **UI 표면에 노출되는 명칭은 "시리즈"**로 통일한다 — 버튼 "새 시리즈", 경로 "내 작품 / 〈시리즈명〉", 빈 상태 "이 시리즈에 작품을 끌어다 놓으세요", "아직 만든 시리즈가 없어요". "모음"은 복수 전제라 작품 1개일 때 어색 + "한 시리즈가 사실상 한 작품(여러 권)일 수 있음"에 부합. 코드 식별자(엔티티/컬럼/엔드포인트)는 영문 기술 명명(category)을 유지하되, 화면 문구는 "시리즈".
- **타일 = 책등 스택(장식) + 빨려 들어가는 드롭(확정, dogfooding 후 2026-06-22)**: 시리즈 타일은 안의 작품을 "책등(spine)"으로 쌓아 보인다 — **높이는 작품 id 시드 결정적 장식**(길이 의미 없음; 소설은 대부분 길어 clamp 비교 무의미 + 달성률 게이지는 PRODUCT.md 경계). 정확한 권수는 "작품 N편" 배지, 8권 초과는 "+N". 드롭 시 카드가 타일 중심으로 **빨려 들어가는 애니메이션**(@dnd-kit 기본 snap-back 제거, `dropAnimation={null}` + 커스텀 fly-in), 취소 시에만 제자리 복귀. 생성 UX=타일 형태 안 인라인 폼(이름+만들기/취소+예시). 목업: `docs/research/2026-06-22-series-tile-and-drop.html`.
