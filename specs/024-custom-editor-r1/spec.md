# Feature Specification: 자체 에디터 엔진 1라운드 — B형 집필실 수직 슬라이스(구조)

**Feature Branch**: `024-custom-editor`

**Created**: 2026-06-15

**Status**: Draft

**Input**: 브레인스토밍 산출 — 집필 에디터를 TipTap(CSS column-wrap 페이지분할)에서 자체 EditContext 엔진으로 교체하는 다라운드 작업의 첫 라운드. PoC 4기준(줄단위 분할·규격/폰트 리플로우·가변높이 이미지·한글 IME) 전부 검증 완료. 본 라운드는 "진짜 페이지 분할"을 실제 집필 경험(쓰기→저장→재로드) 안에서 동작시키는 가장 얇은 수직 슬라이스다.

## 배경 (왜 이 작업인가)

현행 집필 에디터의 페이지 분할은 CSS `column-wrap`(비표준·Chrome 전용)이 나누고 JS는 장수만 추정하는 **두 시계** 구조다. 35px 격자를 벗어나는 순간(제목 여백·인용·이미지) 어긋나, 괘선 밖 입력·문단 통째 점프·규격변경 깨짐·이미지 첨부 불가 같은 증상을 낳는다. 이는 버그가 아니라 아키텍처 한계다. 제어를 뒤집어 **JS가 측정→배치→렌더**하는 자체 엔진으로 교체하면 줄단위 분할·규격 리플로우·가변높이 블록이 한 메커니즘으로 풀린다. PoC가 이를 격리 라우트에서 실증했고, 본 라운드는 그 엔진을 **실제 B형 집필실에 꽂아** 실제 글쓰기·저장 흐름 안에서 처음으로 검증한다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 자체 엔진으로 글을 쓰고 저장·재로드한다 (Priority: P1)

작가가 자체 엔진 전용 신규 라우트에서 **새로 만든 테스트 챕터**를 열어 문단과 제목을 입력한다. 입력을 멈추면 잠시 후 자동으로 저장되고, 브라우저를 새로고침해도 방금 쓴 내용이 그대로 다시 뜬다. 글이 한 페이지를 넘기면 다음 페이지로 **줄 단위로 자연스럽게 이어지고**, 문단이 통째로 점프하지 않는다.

**Why this priority**: 이 작업의 양보 불가 핵심("진짜 페이지 분할이 실제 집필 경험 안에서 동작")이 바로 이 흐름이다. 이 한 스토리만 구현해도 자체 엔진이 실제 저장 인프라와 결선돼 동작함을 증명하는 최소 산출물(MVP)이 된다.

**Independent Test**: 신규 라우트에서 프레시 테스트 챕터에 여러 문단을 페이지 높이 이상으로 입력 → 1.5초 후 자동저장 발생 확인 → 새로고침 → 동일 내용·동일 분할 재현. 페이지 경계에서 한 문단이 두 페이지에 줄 단위로 걸쳐 이어지는지 확인.

**Acceptance Scenarios**:

1. **Given** 프레시 테스트 챕터가 열린 신규 라우트, **When** 작가가 페이지 본문 높이를 초과하는 분량의 문단을 입력, **Then** 초과분이 다음 페이지로 줄 단위로 이어지며 첫 페이지의 마지막 문단이 통째로 밀리지 않는다.
2. **Given** 작가가 입력 중, **When** 타자를 멈추고 1.5초가 지나거나 마지막 저장 후 10초가 지남, **Then** 본문이 서버에 저장되고 새 버전 토큰을 받는다.
3. **Given** 방금 저장된 챕터, **When** 브라우저를 새로고침, **Then** 저장된 본문과 분할이 손실 없이 동일하게 복원된다.
4. **Given** 입력 중 네트워크 저장이 아직 안 끝난 상태, **When** 작가가 페이지를 떠났다 돌아옴, **Then** 로컬 보존분(draft)으로 작성 내용이 복원돼 유실되지 않는다.

---

### User Story 2 - 제목으로 구조를 잡고 목차로 이동한다 (Priority: P2)

작가가 본문 중 한 줄을 제목(1~3수준)으로 지정한다. 제목은 본문보다 큰 글자로 렌더되고, 좌측 아웃라인(목차)에 그 제목이 순서대로 나타난다. 목차 항목을 누르면 해당 제목 위치로 스크롤되고, 스크롤을 내리면 현재 화면 상단의 제목이 목차에서 강조된다.

**Why this priority**: 제목은 집필실의 구조 뼈대이자 아웃라인 패널의 입력원이다. P1의 저장·분할이 동작하는 위에 "구조 잡기 + 탐색"을 얹어, 자체 엔진이 주변 패널과도 결선됨을 검증한다.

**Independent Test**: 제목 여러 개를 섞은 본문 작성 → 좌측 목차에 제목들이 순서대로 표시 → 목차 항목 클릭 시 해당 제목으로 스크롤 → 스크롤 시 상단 제목 강조 전환 확인. 저장·재로드 후에도 제목 수준이 보존되는지 확인.

**Acceptance Scenarios**:

1. **Given** 본문에 커서가 있을 때, **When** 작가가 해당 줄을 제목 수준 1~3 중 하나로 토글, **Then** 그 줄이 해당 수준의 큰 글자로 렌더되고 분할이 즉시 재계산된다.
2. **Given** 제목이 여럿 있는 본문, **When** 작가가 좌측 목차의 한 항목을 클릭, **Then** 해당 제목 위치로 스크롤된다.
3. **Given** 제목이 포함된 챕터를 저장·재로드, **When** 다시 열림, **Then** 제목 수준이 손실 없이 복원되고 목차도 동일하게 표시된다.

---

### User Story 3 - 실수를 되돌리고 평문을 붙여넣는다 (Priority: P3)

작가가 입력 도중 실수하면 실행취소(Cmd+Z)로 직전 상태로 되돌리고, 다시실행(Cmd+Shift+Z)으로 복원한다. 외부에서 복사한 텍스트를 붙여넣으면 **서식 없는 평문**으로 현재 커서 위치에 삽입된다.

**Why this priority**: 실제 글쓰기에 필수인 편집 보조 기능. 없으면 dogfooding 자체가 고통스럽지만, P1·P2의 핵심 흐름과 독립적이라 우선순위는 낮다.

**Independent Test**: 텍스트 입력 → Cmd+Z로 되돌림 → Cmd+Shift+Z로 복원 확인. 외부 서식 텍스트 복사 → 붙여넣기 → 평문으로만 삽입되는지 확인.

**Acceptance Scenarios**:

1. **Given** 작가가 텍스트를 입력한 직후, **When** Cmd+Z, **Then** 직전 편집 단위가 취소된다.
2. **Given** 실행취소 직후, **When** Cmd+Shift+Z, **Then** 취소된 내용이 복원된다.
3. **Given** 외부 앱에서 서식 있는 텍스트를 복사, **When** 에디터에 붙여넣기, **Then** 서식이 제거된 평문만 커서 위치에 삽입된다.

---

### Edge Cases

- **페이지보다 큰 단일 줄/이미지**: 빈 페이지에 그대로 놓아 무한 분할을 방지한다(렌더가 폭에 맞춰 처리).
- **빈 챕터**: 본문이 비어 있어도 첫 페이지(빈 종이)가 정상 표시되고 입력 가능해야 한다.
- **규격/폰트 변경 중 입력**: 용지·폰트를 바꾸는 순간 race 없이 1회 재배치되어야 하며, 진행 중 입력과 충돌하지 않는다.
- **저장 충돌**: 동일 챕터를 다른 세션이 먼저 저장해 버전 토큰이 어긋나면 현행 충돌 감지 흐름이 그대로 동작해야 한다(자체 엔진이 이를 무력화하지 않는다).
- **마크/리스트가 포함된 기존 챕터를 신규 라우트로 여는 경우**: 본 라운드 변환기는 미지원 서식을 평문으로 평탄화(lossy)하므로, 신규 라우트는 **프레시 테스트 챕터로만** 진입하도록 안내/가드해 기존 본문 손실을 막는다.
- **한글 IME 조합 중 페이지 경계**: 조합 중 자모가 깨지지 않고, 조합 완료 시점에 분할이 반영되어야 한다.

## Requirements *(mandatory)*

### Functional Requirements

**입력·편집**

- **FR-001**: 시스템은 작가가 신규 라우트의 에디터에서 한글 IME 조합을 포함한 텍스트를 입력·삭제할 수 있게 해야 하며, 조합 중 자모가 깨지지 않아야 한다.
- **FR-002**: 시스템은 Enter로 블록(문단/제목)을 분할하고, 블록 시작에서 Backspace로 이전 블록과 병합할 수 있어야 한다.
- **FR-003**: 시스템은 작가가 현재 블록을 본문(문단) 또는 제목 수준 1~3으로 토글할 수 있게 해야 한다.
- **FR-004**: 시스템은 실행취소(직전 편집 단위)와 다시실행을 제공해야 한다.
- **FR-005**: 시스템은 붙여넣기 시 서식을 제거한 평문을 커서 위치에 삽입해야 한다.
- **FR-006**: 시스템은 클릭·드래그·Shift/Cmd/Option+화살표·Cmd+A로 캐럿 이동과 선택을 지원하고, 선택 위 입력은 교체·Backspace는 삭제로 처리해야 한다.

**페이지 분할·렌더**

- **FR-007**: 시스템은 본문을 실제 용지 비율(A5/A4/B4/A3)의 페이지로 나누되, 문단이 페이지 경계를 넘으면 **줄 단위로 다음 페이지에 이어** 놓아야 한다(문단 통째 점프 금지).
- **FR-008**: 시스템은 제목 블록을 본문보다 큰 글자/줄높이로 측정·배치하고, 그로 인한 분할 변화를 정확히 반영해야 한다.
- **FR-009**: 시스템은 **용지 규격이 바뀌면** 1초 이내에 즉시 재배치(리플로우)해야 한다. (본문 폰트 크기 변경은 엔진이 동일 메커니즘으로 처리 가능하나, 1라운드에서는 사용자에게 노출하지 않고 고정 기본값을 쓴다 — 폰트 크기 사용자 변경 UI는 후속 라운드.)
- **FR-010**: 시스템은 캐럿·클릭 hit-test 위치를 렌더와 동일한 측정 방식으로 계산해 누적 드리프트가 없어야 한다.

**저장·로드(경계)**

- **FR-011**: 시스템은 챕터 본문을 현행 디스크 포맷(ProseMirror JSON 문자열, `documents.bodyJson`)으로 저장·로드하며, 자체 엔진의 내부 모델은 로드 시 변환해 들이고 저장 시 변환해 내보내야 한다.
- **FR-012**: 시스템은 본문 변경 후 타자 멈춤 기준 약 1.5초, 마지막 저장 후 최대 약 10초 이내에 자동저장하고, 저장 시 서버가 발급하는 새 버전 토큰을 갱신해야 한다.
- **FR-013**: 시스템은 입력 시 로컬 보존분(draft)을 즉시 기록해, 네트워크 저장 전에 페이지를 떠나도 작성 내용이 복원되어야 한다.
- **FR-014**: 시스템은 paragraph·heading(1~3) 블록을 변환 왕복(로드↔저장)에서 손실 없이 보존해야 한다.
- **FR-015**: 시스템은 현행 저장 API 계약(`{body, version}` PUT)·버전 토큰 의미·공동집필 충돌 감지를 변경 없이 그대로 사용해야 한다.

**통합·탐색**

- **FR-016**: 시스템은 신규 라우트에서 현행 B형 집필실의 셸·곁쪽지 패널·인물 패널을 재사용하고, 에디터 코어만 자체 엔진으로 대체해야 한다.
- **FR-017**: 시스템은 본문의 제목(1~3수준)에서 아웃라인(목차)을 파생하고, 목차 항목 클릭 시 해당 제목으로 스크롤하며, 스크롤 시 화면 상단 제목을 강조해야 한다.

**범위 가드**

- **FR-018**: 기본 B형 집필실(TipTap)은 본 라운드에서 변경 없이 그대로 동작해야 한다.
- **FR-019**: 신규 라우트는 마크·리스트 등 미지원 서식이 평문으로 평탄화될 수 있음을 전제로, 기존 서식 본문의 손실을 막기 위해 프레시 테스트 챕터 진입을 전제로 동작해야 한다.

### Key Entities *(include if feature involves data)*

- **챕터 본문(Chapter body)**: 디스크에는 ProseMirror JSON 문자열(`bodyJson`)로 저장되는 글의 내용. 자동저장과 버전 토큰의 대상.
- **내부 문서 모델(Internal document model)**: 자체 엔진이 메모리에서 다루는 표현 — 블록(문단/제목/이미지) 시퀀스와 블록별 속성(종류·제목 수준). 디스크 포맷과 경계에서 변환되는 휘발성 표현.
- **블록(Block)**: 문단·제목(1~3수준)·이미지 등 한 덩어리. 제목은 수준 속성을 가진다.
- **페이지 분할 결과(Layout pages)**: 측정된 블록을 본문 높이에 맞춰 나눈 페이지·조각 배치. 화면 렌더와 분할의 단일 진실원.
- **버전 토큰(Version token)**: 저장마다 서버가 발급하는 불투명 토큰. 공동집필 충돌 감지의 기준(제거·무력화 금지).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 작가가 신규 라우트의 프레시 테스트 챕터에 페이지 높이를 넘는 분량을 입력하면, 초과 문단이 100% 줄 단위로 다음 페이지에 이어진다(문단 통째 점프 0건).
- **SC-002**: 입력 멈춤 후 약 1.5초 이내에 자동저장이 일어나고, 새로고침 시 작성 내용이 손실 없이 100% 복원된다.
- **SC-003**: paragraph·heading(1~3) 본문을 저장·재로드하는 왕복에서 내용·제목 수준 손실이 0건이다(왕복 무손실).
- **SC-004**: 용지 규격 또는 폰트 크기를 바꾸면 1초 이내에 화면이 깨짐 없이 재배치된다.
- **SC-005**: 캐럿/클릭 위치가 브라우저 기준 위치(`caretRangeFromPoint`)와 측정 다수 지점에서 어긋남 없이 일치한다(드리프트 없음).
- **SC-006**: 좌측 목차가 본문 제목과 100% 일치하는 순서로 표시되고, 항목 클릭 시 해당 제목으로 이동한다.
- **SC-007**: 기본 B형 집필실(TipTap)의 기존 동작이 본 라운드 전후로 회귀 없이 동일하다.

## Assumptions

- **대상 브라우저**: Chromium(Chrome/Edge 121+) 전용 우선. Safari/Firefox 지원은 본 라운드 범위 밖(후반 라운드).
- **서식 범위**: 본 라운드는 paragraph·heading(1~3)만 렌더·편집한다. 마크(bold/italic/underline/strike)·리스트·인용·코드블록·구분선은 후속 라운드. 따라서 임의의 기존 챕터에 대한 완전 무손실 로드는 보장 대상이 아니며, dogfooding은 프레시 테스트 챕터에서 수행한다.
- **저장 인프라 재사용**: 현행 자동저장(016)·버전 토큰·로컬 draft·저장 API·공동집필 충돌 감지를 무수정으로 재사용한다. 본 라운드는 본문 표현의 경계 변환만 추가한다.
- **PoC 자산 재사용**: 페이지 기하·레이아웃 엔진·줄 측정·입력/캐럿/선택 로직(PoC 코드)을 토대로 승격·확장하며, 처음부터 다시 만들지 않는다.
- **이미지**: 엔진의 가변높이 이미지 처리 능력은 유지하되, 본 라운드에서 이미지 삽입 UI는 제공하지 않는다(현행 B형 저장 포맷에 이미지 노드가 없어 상호운용을 깨지 않기 위함).
- **전환 모델**: 본 라운드는 신규 전용 라우트에서 dogfooding하고, 기본 B형(TipTap)은 그대로 둔다. BEditor 완전 대체는 마크·서식 패리티를 갖춘 후속 라운드에서 수행한다.
- **격리 제약**: 023-export 브랜치의 미커밋 패치는 다른 세션 소유이므로 본 작업에서 절대 건드리지 않는다. 미래 머지 충돌(`BEditor.tsx`·`paper-editor.css`·`pageLayout.ts`)은 완전 대체 라운드에서 정리한다.
