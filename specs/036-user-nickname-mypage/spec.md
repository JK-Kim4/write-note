# Feature Specification: 사용자 닉네임 + 마이페이지

**Feature Branch**: `036-user-nickname-mypage`

**Created**: 2026-06-24

**Status**: Draft

**Input**: User description: "사용자 계정에 닉네임으로 사용될 컬럼을 추가한다. (1) 닉네임은 고유한 값으로 중복 불가, (2) 변경 가능, (3) 마이페이지(설정과 별개)에서 변경, (4) 추후 도입 예정인 공유 기능에서 공유자/첨삭자 식별자로 사용. 가입 시 시스템이 한글 단어 조합 랜덤 닉네임을 자동 부여하고, 기존 회원은 단순값으로 백필한다. 단어 큐레이션 시 비속어·사회적으로 문제될 단어는 배제한다."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 모든 사용자가 고유 닉네임을 보유 (Priority: P1)

모든 사용자 계정은 다른 사람과 겹치지 않는 고유한 닉네임을 가진다. 신규로 가입하는 사용자는 가입 직후 별도 입력 없이 읽기 좋은 한글 닉네임을 자동으로 부여받고, 이미 가입해 있던 기존 사용자도 빠짐없이 닉네임을 갖게 된다. 이 닉네임은 추후 도입될 공유·첨삭 기능에서 "누가 공유했는지 / 누가 첨삭했는지"를 가리키는 식별 이름의 토대가 된다.

**Why this priority**: 닉네임의 존재 자체가 본 기능의 양보 불가 핵심이다. 닉네임이 있어야 변경(US2)도, 표시(US3)도, 향후 공유 식별자도 성립한다. 닉네임 없는 계정이 하나라도 있으면 후속 기능이 깨지므로 가장 먼저 보장되어야 한다.

**Independent Test**: 신규 계정을 만들면 닉네임이 자동으로 채워지는지, 그리고 기존 계정에도 닉네임이 채워져 있는지(닉네임 없는 계정 0건) 본인 정보 조회로 확인하면 단독 검증된다. 마이페이지 화면 없이도 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 닉네임 기능이 도입된 상태, **When** 새 사용자가 이메일로 가입하면, **Then** 사용자는 별도 입력 없이 한글 단어 조합 형태의 고유 닉네임을 부여받는다.
2. **Given** 닉네임 기능이 도입된 상태, **When** 새 사용자가 카카오로 가입하면, **Then** 마찬가지로 고유 닉네임이 자동 부여된다.
3. **Given** 기능 도입 이전부터 존재하던 기존 사용자, **When** 기능이 도입되면, **Then** 그 사용자도 고유 닉네임을 보유하게 된다(미보유 계정 없음).
4. **Given** 두 사용자가 동시에 가입, **When** 자동 생성 닉네임 후보가 우연히 겹치면, **Then** 시스템이 자동으로 다른 값을 만들어 두 사용자 모두 서로 다른 고유 닉네임을 갖는다.

### User Story 2 - 마이페이지에서 닉네임 변경 (Priority: P2)

사용자는 자동 부여된 닉네임이 마음에 들지 않으면 원하는 이름으로 직접 바꿀 수 있다. 변경은 앱 동작 설정 화면이 아니라 "계정·신원" 영역인 마이페이지에서 이뤄진다.

**Why this priority**: 사용자가 자신을 표현하는 이름을 통제할 수 있어야 닉네임이 실질적 의미를 갖는다. P1(닉네임 존재)이 전제되어야 변경 대상이 생기므로 그다음 우선순위다.

**Independent Test**: 마이페이지에 진입해 닉네임을 새 값으로 바꾸고, 다시 조회했을 때 바뀐 값이 유지되는지 확인하면 단독 검증된다.

**Acceptance Scenarios**:

1. **Given** 로그인한 사용자가 마이페이지에 있을 때, **When** 현재 닉네임을 확인하고 유효한 새 닉네임으로 변경하면, **Then** 닉네임이 새 값으로 저장되고 화면에 반영된다.
2. **Given** 마이페이지에서 닉네임을 변경하려 할 때, **When** 이미 다른 사용자가 쓰고 있는 닉네임을 입력하면, **Then** 변경이 거부되고 "이미 사용 중"이라는 안내를 받는다.
3. **Given** 마이페이지에서 닉네임을 변경하려 할 때, **When** 길이/허용 문자 규칙에 어긋난 값을 입력하면, **Then** 변경이 거부되고 어떤 규칙을 어겼는지 안내를 받는다.
4. **Given** 한 사용자가 닉네임을 여러 번 바꾼 뒤, **When** 또 변경을 시도하면, **Then** 변경 횟수 제한 없이 다시 변경할 수 있다.
5. **Given** 마이페이지에서 닉네임을 변경하려 할 때, **When** 비속어·금칙어가 포함된 값을 입력하면, **Then** 변경이 거부되고 사용할 수 없는 단어가 포함되었다는 안내를 받는다.

### User Story 3 - 마이페이지에서 계정 정보 확인 (Priority: P3)

사용자는 마이페이지에서 자신의 계정 기본 정보(이메일, 가입 방식, 가입일)를 읽기 전용으로 확인할 수 있다.

**Why this priority**: 닉네임 변경(P2)이 들어가는 마이페이지에 계정 정보를 함께 보여주면 "내 계정" 화면으로서 완결성이 생긴다. 다만 닉네임 핵심 흐름과 독립적이라 우선순위는 가장 낮다.

**Independent Test**: 마이페이지에 진입해 이메일·가입 방식·가입일이 정확히 표시되는지 확인하면 단독 검증된다.

**Acceptance Scenarios**:

1. **Given** 이메일로 가입한 사용자가 마이페이지에 진입, **When** 화면을 보면, **Then** 자신의 이메일·"이메일 가입"·가입일이 읽기 전용으로 표시된다.
2. **Given** 카카오로 가입한 사용자가 마이페이지에 진입, **When** 화면을 보면, **Then** 가입 방식이 "카카오"로 표시된다.

---

### Edge Cases

- **자동 생성 닉네임 고유성 충돌**: 자동 생성 후보가 기존 닉네임과 겹치면 시스템이 다른 값으로 재시도해 반드시 고유성을 보장한다.
- **동시 변경 경쟁**: 두 사용자가 같은 닉네임으로 동시에 변경을 시도하면 한 명만 성공하고 다른 한 명은 "이미 사용 중" 안내를 받는다.
- **변경값이 현재 닉네임과 동일**: 사용자가 지금 닉네임과 같은 값으로 변경을 시도하면 충돌로 처리하지 않고 자연스럽게 수용한다(자기 자신과의 충돌 오인 방지).
- **앞뒤 공백 입력**: 닉네임 입력의 앞뒤 공백은 제거 후 규칙 검증한다.
- **기존 회원 백필값 충돌**: 백필 초기값은 계정마다 고유하게 식별되는 단순값을 쓰므로 백필 시점에 서로 충돌하지 않는다.
- **비로그인 접근**: 로그인하지 않은 사용자는 마이페이지·닉네임 변경에 접근할 수 없다.
- **부적절 단어 자동 생성**: 자동 생성에 쓰이는 단어 사전은 비속어·차별·혐오·성적 표현 등 사회적으로 문제될 단어를 배제하도록 큐레이션되어, 자동 부여 닉네임에 그런 단어가 나타나지 않는다.
- **부적절 단어 직접 입력**: 사용자가 금칙어를 포함한 닉네임으로 직접 변경을 시도하면 금칙어 검증에서 거부되어 저장되지 않는다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 모든 사용자 계정이 고유한 닉네임을 보유하도록 보장해야 한다(두 계정이 같은 닉네임을 가질 수 없음).
- **FR-002**: 시스템은 신규 가입(이메일·카카오 모두) 시 사용자에게 추가 입력을 요구하지 않고 닉네임을 자동으로 부여해야 한다.
- **FR-003**: 자동 부여 닉네임은 사람이 읽기 쉬운 한글 단어 조합(수식어 + 명사 + 숫자) 형태여야 한다.
- **FR-004**: 자동 부여에 사용되는 단어 사전은 비속어·욕설·차별·혐오·성적 표현·정치/종교 갈등 등 사회적으로 문제될 수 있는 단어를 배제하도록 큐레이션되어야 한다.
- **FR-005**: 시스템은 기능 도입 시점에 이미 존재하던 모든 기존 사용자에게도 닉네임을 부여해, 닉네임 미보유 계정이 남지 않도록 해야 한다.
- **FR-006**: 사용자는 마이페이지에서 자신의 현재 닉네임을 확인하고 다른 값으로 변경할 수 있어야 한다.
- **FR-007**: 닉네임 길이는 2자 이상 16자 이하여야 한다.
- **FR-008**: 닉네임 허용 문자는 한글·영문·숫자·밑줄(`_`)이며, 공백 및 그 외 특수문자는 허용하지 않는다.
- **FR-009**: 사용자가 이미 사용 중인 닉네임으로 변경을 시도하면 시스템은 변경을 거부하고 중복임을 안내해야 한다.
- **FR-010**: 사용자가 형식 규칙(길이·허용 문자)에 어긋난 닉네임을 입력하면 시스템은 변경을 거부하고 위반 사유를 안내해야 한다.
- **FR-011**: 닉네임 변경 횟수에는 제한을 두지 않는다.
- **FR-012**: 자동 부여 닉네임 생성 시 고유성 충돌이 발생하면 시스템은 자동으로 다른 값을 생성해 고유성을 보장해야 한다.
- **FR-013**: 본인 정보 조회 결과에 닉네임이 포함되어 화면에 표시될 수 있어야 한다.
- **FR-014**: 마이페이지에서 사용자는 자신의 이메일·가입 방식(이메일/카카오)·가입일을 읽기 전용으로 확인할 수 있어야 한다.
- **FR-015**: 마이페이지는 기존 앱 동작 설정 화면과 별개의 진입점으로 제공되어, 계정·신원 정보 영역과 앱 설정 영역을 분리해야 한다.
- **FR-016**: 닉네임 변경 및 마이페이지 접근은 로그인한 사용자에게만 허용되어야 한다.
- **FR-017**: 사용자가 마이페이지에서 직접 입력·변경하는 닉네임은 금칙어 사전(비속어·욕설·차별·혐오·성적 표현 등)에 대해 검증되어야 하며, 금칙어를 포함하면 변경을 거부하고 사용할 수 없는 단어가 포함되었음을 안내해야 한다.

### Key Entities *(include if feature involves data)*

- **사용자(User)**: 서비스 이용 계정. 본 기능으로 추가되는 핵심 속성은 **닉네임**(고유·필수·변경 가능). 기존 속성 중 마이페이지에 표시되는 것 = 이메일, 가입 방식(이메일/카카오), 가입일.
- **닉네임 단어 사전**: 자동 닉네임 생성에 쓰이는 큐레이션된 한글 수식어 목록과 명사 목록. 사회적으로 문제될 단어를 배제한 안전한 어휘만 포함한다.
- **금칙어 사전**: 사용자가 직접 입력하는 닉네임을 검증하는 데 쓰이는 차단 단어 목록(비속어·욕설·차별·혐오·성적 표현 등). 자동 생성용 안전 어휘 사전과는 목적이 반대인 별도 목록이다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 기능 도입 후 닉네임을 보유하지 않은 사용자 계정이 0건이다(기존·신규 전부 포함).
- **SC-002**: 신규 가입 흐름에 닉네임 입력을 위한 추가 단계가 0개다(사용자는 가입 직후 자동으로 닉네임을 받는다).
- **SC-003**: 사용자는 마이페이지 진입 후 3단계(진입 → 입력 → 저장) 이내로 닉네임을 변경할 수 있다.
- **SC-004**: 동일 닉네임을 가진 서로 다른 계정이 0건이다(중복 100% 차단).
- **SC-005**: 자동 생성 닉네임 표본 검수에서 비속어·사회적 문제 단어 출현 0건이다.
- **SC-006**: 사용자는 마이페이지에서 자신의 이메일·가입 방식·가입일을 정확히 확인할 수 있다.
- **SC-007**: 금칙어를 포함한 직접 입력 닉네임 변경 시도는 100% 거부된다(금칙어 포함 닉네임 저장 0건).

## Assumptions

- **공유/첨삭 기능 자체는 본 작업 범위 밖**이다. 본 기능은 그 기능이 사용할 식별자(닉네임) 토대만 마련한다.
- **닉네임 중복 판정은 입력값 정확 일치 기준**(대소문자 구분)이다. 영문 대소문자 정규화·유사 문자 정규화는 추후 과제이며 v1 범위 밖이다.
- **기존 회원 백필 초기값**은 계정마다 고유하게 식별되는 단순값(계정 식별번호 기반)이며, 한글 단어 조합과 달리 예쁘지 않을 수 있으나 사용자가 마이페이지에서 직접 변경 가능하다.
- **마이페이지 진입 동선**은 메인 화면 헤더에 추가한다(현재 헤더에는 사용자 식별 표시가 없다).
- **자동 부여 닉네임의 어휘 안전성**은 큐레이션된 단어 사전으로 보장한다. 단어 사전의 구체 어휘 선정은 구현 단계에서 수행하되 본 스펙의 FR-004 기준을 따른다.
- **사용자 직접 입력 닉네임의 안전성**은 금칙어 사전 기반 사전 차단으로 보장한다(FR-017). 금칙어 사전의 구체 어휘 선정도 구현 단계에서 수행한다. 완벽한 우회 방지(예: 변형 표기·자모 분리)는 v1 범위 밖이며, 핵심 금칙어 차단을 목표로 한다.
- **단어 사전 조합 공간**은 자동 생성 충돌이 실질적으로 드물 만큼 충분히 크다고 가정한다(수식어 × 명사 × 숫자).
