# Feature Specification: 자체 에디터 엔진 2라운드 — 마크(부분 스타일) · 혼합폰트 줄 측정

**Feature Branch**: `024-custom-editor` (R2 — 디렉토리 `024-custom-editor-r2`, 1라운드와 동일 브랜치)

**Created**: 2026-06-15

**Status**: Draft

**Input**: 브레인스토밍 산출 — 자체 EditContext 엔진의 2라운드. 1라운드(구조: 문단·제목·저장·줄단위 분할·undo·클립보드)가 실제 집필실에서 dogfooding 통과한 위에, 1라운드가 의도적으로 미룬 **"드래그한 부분만 굵게"** = 문자 단위 인라인 마크(bold/italic/underline/strike)와, 그로 인해 한 줄 안에서 글자 폭(advance)이 달라지는 **혼합 스타일 줄 측정/캐럿/선택**을 더한다. 데이터 구조는 리서치(Lexical·ProseMirror·Slate·Quill 전부 run/span 리스트) 근거로 **블록별 마크 run-list(정규형 비트마스크 run)** 로 확정.

## 배경 (왜 이 작업인가)

1라운드는 블록 단위 속성(문단/제목)만 모델링했다 — 한 블록 전체가 단일 폰트라는 전제 위에서 측정·캐럿·렌더가 단일 폰트 파라미터로 짜였다. 그래서 "한 문장 안에서 한 단어만 굵게"가 불가능하다. 이는 글쓰기에서 강조·인용·대사 처리에 필수인 기본 기능이다. 2라운드는 이 전제를 깬다: 한 블록 안에서 **구간마다 마크가 다른** 모델을 들이고, 측정·캐럿·선택·렌더를 **run 단위 가변 스타일**로 일반화한다. bold/italic은 같은 글자 크기라도 글리프 폭이 달라지므로(밑줄·취소선은 장식이라 폭 불변), 한 줄 안에 폭이 다른 run이 섞인 채로 줄바꿈·캐럿 좌표·클릭 hit-test가 정확해야 한다. 이것이 본 라운드의 "가장 무거운 엔진 작업"이다. 디스크 포맷(ProseMirror JSON)·자동저장·버전 토큰·충돌 감지는 1라운드처럼 무수정 재사용하고, 본 라운드는 마크 무손실 왕복만 경계 변환에 추가한다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 드래그한 부분만 굵게/기울임/밑줄/취소선 (Priority: P1)

작가가 본문에서 한 구절을 드래그로 선택하고 굵게(Cmd+B)·기울임(Cmd+I)·밑줄(Cmd+U)·취소선을 적용한다. 선택한 구간만 해당 스타일로 즉시 렌더되고, 그 구간이 굵어져 글자 폭이 늘어도 줄바꿈·다음 글자 위치·캐럿이 어긋나지 않는다. 저장 후 새로고침해도 부분 마크가 그대로 복원된다.

**Why this priority**: 이 작업의 양보 불가 핵심("드래그한 부분만 굵게"가 실제 집필 경험 안에서 동작 + 혼합 스타일 줄이 정확히 측정·렌더)이 바로 이 흐름이다. 이 한 스토리만 구현해도 자체 엔진이 문자 단위 마크를 모델·측정·렌더·저장까지 관통함을 증명하는 최소 산출물(MVP)이 된다. (§10 — 핵심을 첫 dogfoodable 산출물에서.)

**Independent Test**: 프레시 테스트 챕터에서 한 줄 안 일부를 드래그 선택 → Cmd+B → 그 구간만 굵게 렌더 확인 → 줄 끝까지 입력해 wrap 유발 → 굵은 구간 뒤 글자/캐럿 위치가 드리프트 없이 정확한지 확인 → 자동저장 후 새로고침 → 부분 마크 동일 복원 확인.

**Acceptance Scenarios**:

1. **Given** 한 블록 안에 텍스트가 있을 때, **When** 작가가 일부 구간을 선택하고 마크(굵게/기울임/밑줄/취소선)를 적용, **Then** 그 구간만 해당 스타일로 렌더되고 나머지는 평문으로 남는다.
2. **Given** 이미 마크가 적용된 구간, **When** 같은 마크를 그 구간 전체에 다시 적용(토글), **Then** 그 구간의 해당 마크가 해제된다.
3. **Given** 굵은 구간과 평문 구간이 한 줄에 섞여 있을 때, **When** 줄이 페이지 폭을 넘겨 wrap, **Then** 늘어난 글자 폭이 반영된 위치에서 줄바꿈되고, 캐럿·클릭 hit-test가 그 줄에서 어긋나지 않는다.
4. **Given** 부분 마크가 있는 챕터를 저장·재로드, **When** 다시 열림, **Then** 마크 구간·종류가 손실 없이 복원된다(왕복 무손실).
5. **Given** 마크 구간을 가로질러 텍스트를 입력·삭제, **When** 편집이 일어남, **Then** 마크 구간이 정확히 따라 늘어나거나 줄어들고(삽입·삭제 추종), 구간 경계가 깨지지 않는다.

### User Story 2 - 마크를 켜고 이어서 쓴다 (보류 마크 + 툴바) (Priority: P2)

작가가 선택 없이 굵게(Cmd+B 또는 툴바 버튼)를 누르면, 이후 입력되는 글자가 굵게 들어간다(표준 워드프로세서 방식). 툴바의 마크 버튼은 현재 캐럿 위치의 마크 상태를 활성/비활성으로 표시한다. 마크 적용 구간 안에 캐럿을 두고 타이핑하면 그 마크를 이어받는다.

**Why this priority**: P1의 "선택 후 적용"이 동작하는 위에 "켜고 이어 쓰기"라는 실제 집필 손맛을 얹는다. 보류 마크 상태와 툴바 활성 표시는 마크 모델·캐럿 위치 마크 조회에 의존하므로 P1 토대 위에서 검증된다.

**Independent Test**: 빈 줄에서 Cmd+B → 타이핑 → 굵게 들어가는지 확인 → Cmd+B 다시 → 평문으로 돌아오는지 확인. 마크 구간 안/밖으로 캐럿 이동 시 툴바 버튼 활성 표시가 전환되는지 확인.

**Acceptance Scenarios**:

1. **Given** 선택이 없고 캐럿만 있을 때, **When** 작가가 마크를 토글(Cmd+B 등), **Then** 이후 입력되는 글자에 그 마크가 적용된다(보류 마크).
2. **Given** 보류 마크가 켜진 상태, **When** 작가가 캐럿을 다른 위치로 이동하거나 보류 마크를 다시 토글, **Then** 보류 마크가 해제된다.
3. **Given** 캐럿이 특정 위치에 있을 때, **When** 화면을 보면, **Then** 툴바의 마크 버튼이 그 위치의 마크 상태(좌측 글자 기준)를 활성/비활성으로 정확히 표시한다.
4. **Given** 굵은 구간 한가운데에 캐럿, **When** 작가가 타이핑, **Then** 입력 글자가 그 구간의 마크를 이어받는다.

### User Story 3 - 줄바꿈·마크 경계에서 캐럿이 정확한 자리에 놓인다 (caret affinity) (Priority: P3)

긴 줄이 자동 줄바꿈(soft-wrap)되는 경계에서, 같은 글자 위치라도 작가가 "앞 줄 끝"에 있길 원하는지 "다음 줄 시작"에 있길 원하는지를 구분해 캐럿이 올바른 시각 위치에 놓인다. 1라운드의 `< vs <=` 임시 처리를 정식 affinity(앞/뒤 방향) 추적으로 대체한다.

**Why this priority**: 정확성 다듬기. 없어도 P1·P2의 마크 기능은 동작하지만, wrap 경계에서 캐럿이 한 줄 위/아래로 튀는 어색함을 제거해 집필 체감을 완성한다. 마크 경계도 같은 메커니즘으로 일관 처리된다.

**Independent Test**: 페이지 폭에 꽉 차 wrap되는 긴 줄에서 줄 끝으로 캐럿 이동(End) → 캐럿이 앞 줄 끝에 보이는지 확인. 다음 줄 시작으로 이동 → 다음 줄 머리에 보이는지 확인. 같은 offset이 두 시각 위치를 가질 때 방향에 맞게 렌더되는지 확인.

**Acceptance Scenarios**:

1. **Given** 자동 줄바꿈 경계의 offset, **When** 작가가 줄 끝(앞 줄) 쪽으로 도달, **Then** 캐럿이 앞 줄의 끝에 표시된다(upstream).
2. **Given** 동일 경계 offset, **When** 작가가 다음 줄 시작 쪽으로 도달, **Then** 캐럿이 다음 줄의 머리에 표시된다(downstream).
3. **Given** wrap 경계에서 좌/우 화살표·클릭으로 캐럿을 옮길 때, **When** 이동, **Then** affinity가 이동 방향과 일관되게 갱신되어 캐럿이 의도한 줄에 놓인다.

---

### Edge Cases

- **마크 경계와 wrap 경계가 겹침**: 굵은 구간이 정확히 줄 끝에서 끝날 때 캐럿·다음 글자 위치가 모호해지지 않아야 한다(affinity로 해소).
- **선택이 여러 블록/마크 구간을 가로지름**: 부분 마크 토글 시 "선택 전체가 이미 그 마크면 해제, 아니면 적용"(표준 토글)이 여러 구간에 걸쳐 일관 적용된다.
- **빈 선택에서 토글(보류 마크)인데 곧바로 캐럿 이동**: 보류 마크가 입력 없이 폐기된다(다음 글자에만 적용).
- **마크 구간 전체 삭제**: 구간이 0 길이가 되면 마크 run이 정규화로 사라진다(빈 run 잔류 금지).
- **밑줄/취소선만 적용(폭 불변)**: 측정은 변하지 않아야 하며(장식 마크), 불필요한 재측정/드리프트가 없어야 한다.
- **한글 IME 조합 중 마크 단축키**: 조합 중 Cmd+B 등이 조합을 깨지 않아야 한다(EditContext compositionstart/end 가드 — 1라운드 회귀룰, `e.isComposing` 신뢰 금지).
- **미지원 PM 마크가 섞인 기존 챕터**: 본 라운드 지원 4종 외 마크(예: link, highlight)는 평문 평탄화(1라운드 전제 유지) — 프레시 테스트 챕터 진입.

## Requirements *(mandatory)*

### Functional Requirements

**마크 모델·적용 (#65)**

- **FR-001**: 시스템은 한 블록 안에서 구간마다 다른 마크(bold/italic/underline/strike)를 보유하는 문서 모델을 유지해야 한다.
- **FR-002**: 시스템은 선택 구간 `[lo, hi)`에 마크를 적용/해제하는 토글을 제공해야 한다 — 구간 전체가 이미 그 마크면 해제, 아니면 적용(표준 토글).
- **FR-003**: 시스템은 텍스트 삽입·삭제 시 마크 구간이 정확히 추종(늘어남/줄어듦/분할/병합)하도록 보장하고, 편집 후에도 마크 표현이 정규형(인접 동일 run 병합·빈 run 제거)을 유지해야 한다.
- **FR-004**: 시스템은 특정 offset의 유효 마크(좌측 글자 기준)를 조회할 수 있어야 한다(툴바 활성 표시·입력 상속의 입력원).

**측정·렌더·캐럿(혼합 스타일) (#66)**

- **FR-005**: 시스템은 한 줄(블록) 안에 폰트(weight/style)가 다른 run이 섞여 있을 때, 각 run의 실제 글자 폭(advance)을 반영해 줄바꿈 위치를 계산해야 한다.
- **FR-006**: 시스템은 캐럿 좌표·클릭 hit-test·선택 사각형을 혼합 스타일 줄에서 렌더와 동일한 측정 방식으로 계산해, 굵은/기울인 구간 뒤에서도 누적 드리프트가 없어야 한다.
- **FR-007**: 시스템은 측정을 **실제 스타일이 적용된 오프스크린 DOM + Range** 로 수행해야 한다(canvas measureText 금지 — CJK 커닝·CSS 셰이핑 오차, 1라운드 회귀룰).
- **FR-008**: 시스템은 선택 구간만 해당 마크 스타일로 렌더해야 한다(굵게=font-weight, 기울임=font-style, 밑줄·취소선=text-decoration).
- **FR-009**: 밑줄·취소선(장식 마크)은 글자 폭을 바꾸지 않으므로 측정·줄바꿈에 영향을 주지 않아야 한다.

**툴바·단축키·보류 마크 (#67)**

- **FR-010**: 시스템은 마크 토글 UI(툴바 버튼)와 단축키(Cmd+B/I/U)를 제공해야 하며, 단축키는 IME 조합 가드 아래에서 동작해야 한다.
- **FR-011**: 시스템은 선택 없이 토글 시 "다음 입력에 적용될 보류 마크"를 보유하고, 입력이 그 마크를 소비하거나 캐럿 이동 시 폐기해야 한다.
- **FR-012**: 시스템은 툴바 마크 버튼에 현재 캐럿 위치의 마크 활성 상태를 표시해야 한다.

**캐럿 affinity (#68)**

- **FR-013**: 시스템은 캐럿을 `(offset, affinity ∈ {앞/뒤})` 로 표현해, soft-wrap 경계·마크 경계에서 같은 offset의 두 시각 위치를 구분해야 한다.
- **FR-014**: 시스템은 화살표·클릭·줄 이동 시 affinity를 이동 방향과 일관되게 갱신하고, 1라운드의 `< vs <=` 임시 처리를 대체해야 한다.

**저장·왕복(경계) (#67)**

- **FR-015**: 시스템은 bold/italic/underline/strike 마크를 ProseMirror JSON(`documents.bodyJson`)의 text node `marks`(`type`: `bold`/`italic`/`underline`/`strike`)로 무손실 왕복(로드↔저장)해야 한다.
- **FR-016**: 시스템은 마크를 포함한 모델↔PM JSON 왕복이 정규형으로 idempotent하도록 보장해, 로드 즉시 거짓 dirty가 나지 않아야 한다(1라운드 회귀룰).

**범위 가드·무회귀 (#69)**

- **FR-017**: 시스템은 1라운드 기능(문단·제목·저장·줄단위 분할·목차·undo/redo·복사/잘라내기/평문 붙여넣기·한글 IME)을 회귀 없이 유지해야 한다.
- **FR-018**: 기본 B형 집필실(TipTap)은 본 라운드에서 변경 없이 그대로 동작해야 한다.
- **FR-019**: 시스템은 백엔드·저장 API 계약·버전 토큰·공동집필 충돌 감지를 변경 없이 그대로 사용해야 한다(백엔드 변경 0).

### Key Entities *(include if feature involves data)*

- **마크 비트마스크(Mark mask)**: 한 run에 적용된 마크 집합을 한 정수로 압축한 값(bold/italic/underline/strike 각 1비트, Lexical 방식).
- **마크 run(Mark run)**: 같은 마크 마스크를 공유하는 연속 글자 구간. 블록을 빈틈없이 덮는 연속 run 분할(정규형: 인접 동일 마스크 없음, 0길이 run 없음 — Quill Delta·ProseMirror text node 시퀀스와 동형).
- **보류 마크(Pending marks)**: 선택 없이 토글한, 다음 입력에 적용될 마크 마스크. 문서 모델 밖 전이 상태(캐럿/조합 상태와 같은 계열), history·직렬화 비대상.
- **캐럿 affinity**: 같은 offset이 앞 줄 끝/다음 줄 시작 두 시각 위치를 가질 때 어느 쪽인지(앞/뒤). 캐럿 표현의 일부.
- **내부 문서 모델(Internal document model)**: 평문 buffer(텍스트·offset의 단일 진실원, EditContext offset API 정합) + 블록 속성(1라운드) + 블록별 마크 run-list(신규). 디스크 PM JSON과 경계에서 변환.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 한 줄 안에 굵은 구간과 평문 구간이 섞여 있을 때, 캐럿/클릭 위치가 브라우저 기준 위치와 다수 지점에서 어긋남 없이 일치한다(혼합 스타일 드리프트 0).
- **SC-002**: 부분 마크(구간·종류)를 가진 본문을 저장·재로드하는 왕복에서 손실이 0건이다(왕복 무손실, idempotent).
- **SC-003**: 선택 구간 마크 토글이 부분/전체/구간 경계/여러 구간 횡단에서 표준 토글 규칙대로 100% 정확하다.
- **SC-004**: 선택 없이 토글 후 타이핑 시 보류 마크가 입력에 적용되고, 툴바 활성 표시가 캐럿 위치 마크와 100% 일치한다.
- **SC-005**: 텍스트 삽입·삭제 후에도 마크 구간이 정확히 추종하고, 마크 표현이 항상 정규형이다(빈/중복 run 0).
- **SC-006**: soft-wrap 경계에서 캐럿이 affinity(앞/뒤)에 맞는 줄에 놓이며, 1라운드의 경계 캐럿 튐이 재현되지 않는다.
- **SC-007**: 1라운드 기능(저장·분할·목차·undo·클립보드·IME)이 본 라운드 전후로 회귀 없이 동일하다.
- **SC-008**: 기본 B형 집필실(TipTap)의 기존 동작이 본 라운드 전후로 회귀 없이 동일하다(기존 page.test GREEN 유지).

## Assumptions

- **마크 범위**: 본 라운드 마크는 정확히 bold/italic/underline/strike 4종이다. 임의 글꼴(font-family)·글자 크기의 구간별 선택은 범위 밖(후속). "혼합폰트"는 bold/italic이 한 줄 안에서 서로 다른 글리프 폭으로 렌더되어 혼합 측정이 필요하다는 뜻이지, 임의 글꼴 선택이 아니다.
- **마크 클립보드 비범위**: 마크가 적용된 텍스트의 복사/붙여넣기에서 **서식 전달은 본 라운드 범위 밖**이다 — 1라운드와 동일하게 붙여넣기는 평문(마스크 0)으로만 삽입한다(복사도 평문). 에디터 내부 서식 복붙은 후속 라운드 결정 대상(핸드오프 §5.6).
- **데이터 구조(리서치 근거)**: production 에디터(Lexical 비트마스크 per-run·ProseMirror 마크 가진 text node 시퀀스·Slate leaf·Quill Delta ops) 전부 run/span 리스트를 쓴다. 우리는 EditContext의 offset API 정합을 위해 **평문 buffer를 텍스트·offset SoT로 유지**하고, 마크는 **블록별 연속 run-list(정규형 비트마스크 run)** 로 얹는다. dense per-char 배열은 채택하지 않는다(production 선례 없음).
- **측정 기법(리서치 근거)**: 혼합 스타일 줄 측정은 실제 CSS(weight/style)가 적용된 오프스크린 DOM에 Range를 거는 방식이 표준(canvas는 CJK 커닝·CSS 셰이핑 오차). 1라운드 measure를 run 인지로 일반화한다.
- **캐럿 affinity(리서치 근거)**: 표준 모델은 `(offset, affinity)` 튜플(upstream/downstream — AppKit·Blink·Flutter, ProseMirror bias, CodeMirror assoc). 앞=줄끝, 뒤=줄시작.
- **IME 가드**: EditContext는 keydown `e.isComposing`을 설정하지 않으므로, 조합은 `compositionstart/compositionend`로 추적한 ref로 가드한다(1라운드 회귀룰). 마크 단축키도 이 가드 아래.
- **저장 인프라 재사용**: 자동저장(016)·버전 토큰·로컬 draft·저장 API·공동집필 충돌 감지를 무수정 재사용. 본 라운드는 마크 무손실 경계 변환만 추가한다.
- **토대 재사용**: 1라운드 모듈(`custom-editor/`)의 model·measure·pmConvert·CustomEditor를 run 단위로 일반화한다. layoutEngine은 무수정(줄별 height만 보므로 가변 줄높이 이미 처리).
- **대상 브라우저**: Chromium(Chrome/Edge 121+) 전용. Safari/Firefox는 후반 라운드.
- **격리 제약**: 023-export 브랜치 미커밋 패치(`BEditor.tsx`·`paper-editor.css`·`pageLayout.ts`·`b.css`·`write/page.tsx`)는 다른 세션 소유 — 절대 비접촉. 자체 엔진은 `custom-editor/`·`BStudioShell`·`/custom` 라우트만 건드린다.
- **검증 한계**: jsdom은 실제 글리프 폭이 0이라 혼합 advance를 단위테스트로 검증 못 한다. 줄분해/run 그룹핑·토글·왕복 등 순수 로직은 Vitest로, 실제 혼합 폭 정합은 CDP/사용자 dogfooding 게이트로 검증한다(1라운드와 동일).
