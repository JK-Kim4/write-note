# Feature Specification: 카드 관리 (Card Management)

**Feature Branch**: `048-card-management`

**Created**: 2026-07-01

**Status**: Draft

**Input**: User description: "board 기능 이외 단순한 문자열 메모를 추가·관리하는 기능. 카드는 단순 문자열 메모를 저장하고, 사용자는 보드와 별개로 카드만 관리하는 메뉴에 진입해 카드 목록과 각 카드의 소속 보드를 확인하며, 상세 화면에서 카드 종류·내용을 보고 수정·삭제(연결 있으면 삭제 경고)할 수 있다."

## 정황 요약 (Context)

- 본 프로젝트는 이미 **보드(board)** 기능을 제공한다. 보드는 무한 캔버스 위에 **카드(card)** 를 마인드맵 형태로 배치하고 카드끼리 **연결(link)** 하는 기능이다.
- 사용자가 말한 "단순 문자열 메모"는 **기존 보드 카드(card) 도메인**을 가리킨다 (근거: "카드 종류"·"카드 간 연결 삭제 경고"·"어느 보드 소속인지"는 모두 카드에만 있는 속성이며, 과거 폐기된 캡처 메모에는 없다).
- 사용자 결정(2026-07-01 인터뷰):
  - **보드 없는 독립 카드 허용** — 카드가 어느 보드에도 속하지 않고 떠 있을 수 있고, 나중에 보드에 붙일 수 있다.
  - **캡처 메모(memos)는 무변경** — 지금도 활성인 iOS 단축어/데스크탑 캡처 파이프라인과 그 데이터는 이 기능에서 건드리지 않으며, 이 화면에 노출하지 않는다.

## Clarifications

### Session 2026-07-01

- Q: 카드의 "소속 보드 변경" 범위 — 독립→보드 배정만인가, 연결 없는 카드의 양방향 재배정인가? → A: 연결(link)이 없는 카드는 자유 재배정(양방향) — 독립↔보드, 보드A↔보드B 모두 허용. 연결선이 걸린 카드만 범위 밖.
- Q: 독립 카드에 종류(타입)가 있는가? → A: 기존 4종(인물/장소/사건/주제) 재사용 + 무지정 기본 — 모든 카드가 동일한 종류 모델(nullable)을 공유. 독립 카드도 선택적으로 종류 지정 가능.
- Q: 카드 관리 진입점 배치? → A: `/boards` 화면의 하위 탭 [보드 | 카드](2026-07-01 목업 확정). 최상위 NAV 신규 항목 없음.
- Q: 집필 화면에서도 카드를 봐야 하나 / 어떤 범위? → A: 메인 진입점(보드 하위 탭)은 유지하고, **집필 화면의 "보드 참조" 패널에 "카드" 뷰를 추가**([보드 | 카드] 토글). 카드 뷰 범위 = **그 작품 관련 보드의 카드 + 독립 카드**(다른 작품에만 속한 카드 제외). 별도 라운드(R4)로 얹으며, 조회는 기존 계약 재사용(FE 전용).
- Q: 카드 목록 정렬·찾기? → A: `/boards` 카드 탭 기본 정렬 = **생성일 내림차순**(안정적 — 편집해도 순서 불변, 최근 수정순은 순서가 튀어 기각). + **문자열 검색**(카드 내용·보드 이름) + **필터**(소속: 전체/보드 소속/독립, 종류: 인물/장소/사건/주제/무지정). 집필 카드 뷰(R4) = **이 작품 보드 → 상위 시리즈 보드 → 독립 카드** 3단 그룹, 각 그룹 안 생성일 내림차순.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 카드 목록에서 모든 카드와 소속 보드 확인 (Priority: P1)

사용자는 보드와 별개의 "카드 관리" 진입점을 통해 카드 관리 화면에 들어가, 자신이 가진 모든 카드를 여러 보드를 가로질러 한 목록으로 보고, 각 카드가 어느 보드에 속해 있는지(또는 어디에도 속하지 않는지)를 확인한다.

**Why this priority**: 사용자가 요청의 첫머리에서 말한 "카드 정보만 관리하는 메뉴 진입 + 카드 목록 + 소속 보드 확인"에 해당하는 핵심 진입점이자, 이후 상세·수정·삭제·생성이 모두 얹히는 토대다.

**Independent Test**: 여러 보드에 흩어진 카드가 있는 사용자로 로그인해 카드 관리 화면에 진입하면, 모든 카드가 한 목록에 나오고 각 카드에 소속 보드 이름(또는 "속한 보드 없음")이 붙어 표시되는 것으로 단독 검증 가능하다. 이것만으로도 "내 모든 카드를 한눈에 본다"는 가치를 전달한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 서로 다른 두 보드에 각각 카드를 갖고 있고, **When** 카드 관리 화면에 진입하면, **Then** 두 보드의 카드가 모두 하나의 목록에 나타나고 각 카드에 해당 보드 이름이 표시된다.
2. **Given** 사용자가 어느 보드에도 속하지 않은 독립 카드를 갖고 있고, **When** 카드 관리 화면에 진입하면, **Then** 그 카드가 목록에 나타나고 소속 보드 자리에 "속한 보드 없음"(또는 동등한 표시)이 나온다.
3. **Given** 다른 사용자의 카드가 존재하고, **When** 내 카드 관리 화면을 보면, **Then** 내 카드만 나타나고 타인 카드는 보이지 않는다.
4. **Given** 사용자가 카드를 하나도 갖고 있지 않고, **When** 카드 관리 화면에 진입하면, **Then** 빈 상태 안내와 새 카드 만들기 유도가 표시된다.

---

### User Story 2 - 독립 카드(단순 문자열 메모) 생성 (Priority: P2)

사용자는 카드 관리 화면에서 어느 보드에도 속하지 않는 새 카드를 만들어 단순한 문자열 메모를 저장한다.

**Why this priority**: 사용자가 요청의 헤드라인에서 강조한 "단순 문자열 메모를 추가하는 기능". 다만 만든 카드를 확인하려면 목록(P1)이 먼저 있어야 하므로 P2로 둔다.

**Independent Test**: 카드 관리 화면에서 본문 텍스트를 입력해 저장하면, 보드 없이 소유된 독립 카드가 생성되어 즉시 목록에 나타나는 것으로 단독 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 사용자가 카드 관리 화면에 있고, **When** 새 카드에 본문 문자열을 입력하고 저장하면, **Then** 소속 보드 없는 독립 카드가 생성되고 목록에 즉시 나타난다.
2. **Given** 새 카드 생성 시, **When** 종류(타입)를 지정하지 않고 본문만 입력하면, **Then** 무지정 상태로 카드가 생성된다.
3. **Given** 새 카드 생성 시, **When** 본문을 비운 채 저장을 시도하면, **Then** 본문이 필요하다는 안내로 저장이 막힌다.

---

### User Story 3 - 카드 상세 확인 및 수정 (Priority: P3)

사용자는 목록에서 카드를 열어 종류와 본문 전체를 확인하고, 본문과 종류를 수정해 저장한다.

**Why this priority**: 요청 4항(상세에서 타입·내용 확인 + 수정)에 해당한다. 목록·생성이 있어야 대상이 존재하므로 P3.

**Independent Test**: 목록에서 카드를 열어 종류·본문이 표시되고, 본문/종류를 바꿔 저장한 뒤 다시 열면 바뀐 값이 유지되는 것으로 단독 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 목록에 카드가 있고, **When** 카드를 열면, **Then** 카드의 종류(무지정이면 "무지정")와 본문 전체가 표시된다.
2. **Given** 카드 상세에서, **When** 본문을 수정하고 저장하면, **Then** 변경된 본문이 저장되고 목록·상세에 반영된다.
3. **Given** 카드 상세에서, **When** 종류를 다른 값으로 바꾸거나 무지정으로 바꾸면, **Then** 변경된 종류가 저장된다.
4. **Given** 보드에 속한 카드든 독립 카드든, **When** 수정하면, **Then** 동일한 방식으로 본문·종류가 저장된다.

---

### User Story 4 - 카드 삭제 및 연결 경고 (Priority: P3)

사용자는 카드를 삭제한다. 삭제하려는 카드가 다른 카드와 연결되어 있으면, 확정 전에 연결이 함께 사라진다는 경고를 본다.

**Why this priority**: 요청 4항의 "다른 카드와 연결이 있는 경우 삭제 경고" 요구. 데이터 손실 관련이라 명확히 다뤄야 하나, 목록·상세가 선행돼야 하므로 P3.

**Independent Test**: 다른 카드와 연결된 보드 카드를 삭제하려 하면 경고와 연결 개수가 뜨고, 확정 시 카드와 연결이 함께 사라지는 것으로 단독 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 다른 카드와 1개 이상 연결된 카드가 있고, **When** 그 카드 삭제를 시도하면, **Then** 삭제 확정 전에 경고와 연결 개수가 표시된다.
2. **Given** 삭제 경고가 뜬 상태에서, **When** 삭제를 확정하면, **Then** 카드와 그 카드에 걸린 연결이 함께 제거된다.
3. **Given** 연결이 없는 카드(독립 카드 포함)가 있고, **When** 삭제를 시도하면, **Then** 연결 경고 없이 삭제된다(일반 삭제 확인은 있을 수 있음).
4. **Given** 카드를 삭제한 뒤, **When** 목록을 보면, **Then** 삭제된 카드는 사라지고 다른 카드·보드는 영향받지 않는다.

---

### User Story 5 - 카드의 소속 보드 변경 (연결 없는 카드) (Priority: P4)

사용자는 연결(link)이 없는 카드의 소속 보드를 바꾼다 — 독립 카드를 보드에 붙이거나, 보드 카드를 독립으로 떼거나, 다른 보드로 옮긴다.

**Why this priority**: 사용자가 고른 "나중에 보드에 붙임" + 연결 없는 카드의 양방향 자유 재배정(2026-07-01 Clarification). 핵심 관리 흐름(목록·생성·수정·삭제)이 갖춰진 뒤의 통합 단계라 P4.

**Independent Test**: 연결 없는 카드를 골라 다른 보드로 옮기거나 독립으로 떼면, 카드 관리 목록의 소속 보드 표시와 관련 보드 캔버스가 그에 맞게 갱신되는 것으로 단독 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 독립 카드와 사용자 소유의 보드가 있고, **When** 그 카드를 그 보드에 배정하면, **Then** 카드의 소속 보드가 해당 보드로 바뀌고 카드 관리 목록의 소속 보드 표시도 갱신되며 그 보드 캔버스에 기본 위치로 나타난다.
2. **Given** 연결이 없는 보드 카드가 있고, **When** 그 카드를 독립으로 떼면, **Then** 소속 보드가 "없음"이 되고 원래 보드 캔버스에서 사라진다.
3. **Given** 연결이 없는 보드 카드가 있고, **When** 그 카드를 다른 보드로 옮기면, **Then** 대상 보드에 기본 위치로 나타나고 원래 보드에서 사라진다.
4. **Given** 카드의 소속 보드를 바꿀 때, **When** 다른 사용자의 보드를 지정하려 하면, **Then** 변경이 허용되지 않는다.
5. **Given** 다른 카드와 연결(link)이 걸린 카드가 있고, **When** 소속 보드 변경을 시도하면, **Then** 그 카드는 변경 대상에서 제외된다(비활성 또는 안내).

### User Story 6 - 집필 중 카드 모아 보기 (Priority: P5)

작가는 집필 화면에서 보드를 곁에 두고 쓸 수 있듯이, **그 작품과 관련된 카드 + 독립 카드를 한데 모아 보는 뷰**에도 접근해 쓰면서 참조한다. (집필 화면의 기존 "보드 참조" 패널에 [보드 | 카드] 전환을 더한다.)

**Why this priority**: 집필 흐름을 끊고 보드 메뉴로 나가지 않아도, 쓰던 자리에서 카드/메모를 곁눈질하게 한다. 기존 "보드 참조"(그 작품 보드 + 상위 시리즈 보드)의 연장 — 보드 캔버스뿐 아니라 카드 묶음도. 관리 흐름(US1~US5)이 갖춰진 뒤의 집필 통합이라 마지막.

**Independent Test**: 집필 화면에서 카드 뷰를 열면 그 작품 관련 보드의 카드 + 독립 카드가 모여 보이고(다른 작품에만 속한 카드는 제외), 카드를 열어 종류·내용을 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 집필 화면의 참조 패널에서, **When** [카드] 뷰로 전환하면, **Then** 그 작품 관련 보드의 카드 + 사용자의 독립 카드가 모여 표시된다.
2. **Given** 카드 뷰에서, **When** 다른 작품에만 속한 카드는, **Then** 표시되지 않는다.
3. **Given** 카드 뷰에서, **When** 카드를 열면, **Then** 종류·내용을 확인할 수 있다.
4. **Given** 그 작품 관련 카드·독립 카드가 하나도 없으면, **When** 카드 뷰를 열면, **Then** 빈 상태 안내가 표시된다.

---

### Edge Cases

- **빈 목록**: 카드가 하나도 없을 때 — 빈 상태 안내 + 새 독립 카드 만들기 유도(화면 컨텍스트 유지).
- **연결된 카드 삭제**: 삭제 확정 시 그 카드에 걸린 연결이 함께 제거된다(연결 상대 카드는 보존).
- **무지정 카드**: 종류가 없는 카드는 목록·상세에서 "무지정"으로 표시.
- **카드 옮기기·떼기**: 연결 없는 카드를 다른 보드로 옮기면 원래 보드에서 사라지고 대상 보드에 기본 위치로 나타난다. 독립으로 떼면 소속 보드가 "없음"이 되어 어느 보드에도 나타나지 않는다. 이후 보드에서 이동 가능.
- **범위 밖(후속 결정)**: 다른 카드와 **연결선(link)이 걸린 카드**의 소속 보드 변경(연결선 처리 포함)은 본 스펙 범위 밖이다(아래 Assumptions 참조). 소속 보드 변경은 **연결이 없는 카드**에 한해 독립↔보드·보드↔보드 양방향으로 허용한다(2026-07-01 Clarification).

## Requirements *(mandatory)*

### Functional Requirements

**진입 및 목록 (US1)**

- **FR-001**: 인증된 사용자는 `/boards` 화면의 "카드" 탭([보드 | 카드])을 통해 카드 관리 화면에 들어갈 수 있어야 한다(보드 캔버스와 구분된 카드 전용 관리 화면). 추가로 집필 화면에서도 카드 뷰에 접근한다(US6/FR-019).
- **FR-002**: 카드 관리 화면은 사용자 소유의 모든 카드를 여러 보드를 가로질러 하나의 목록으로 보여줘야 한다(어느 보드에 속하든, 보드 없는 독립 카드 포함).
- **FR-003**: 목록의 각 카드는 본문 미리보기와 소속 보드를 표시해야 한다 — 보드에 속한 카드는 해당 보드 이름을, 독립 카드는 "속한 보드 없음"(또는 동등한 표시)을 보여준다.
- **FR-004**: 목록은 요청한 사용자 본인 소유의 카드만 보여야 한다(타인 카드 노출 금지).
- **FR-005**: 목록은 **생성일 내림차순**으로 정렬되어야 한다(동률 시 id 내림차순). 생성 순서라 카드를 편집해도 순서가 바뀌지 않는다(안정적).
- **FR-005a**: 카드 관리 화면은 카드를 찾기 위한 **문자열 검색**(카드 내용·보드 이름)과 **필터**를 제공해야 한다 — 소속(전체 / 보드 소속 / 독립) + 종류(인물·장소·사건·주제·무지정). 검색·필터는 정렬을 바꾸지 않고 표시 대상만 좁힌다.

**상세 (US3)**

- **FR-006**: 사용자는 목록에서 카드를 열어 상세를 볼 수 있어야 하고, 상세는 카드의 종류(타입)와 본문 전체를 보여줘야 한다.
- **FR-007**: 종류가 지정되지 않은 카드는 상세·목록에서 "무지정"으로 표시해야 한다.

**생성 (US2)**

- **FR-008**: 사용자는 카드 관리 화면에서 어느 보드에도 속하지 않는 독립 카드를 새로 만들 수 있어야 한다.
- **FR-009**: 독립 카드 생성 시 본문(문자열)은 필수이고, 종류(타입)는 선택이다 — 기존 4종(인물/장소/사건/주제)에서 고르거나 무지정(기본). 독립 카드와 보드 카드는 동일한 종류 모델을 공유한다.
- **FR-010**: 생성된 독립 카드는 생성 직후 목록에 나타나야 한다.

**수정 (US3)**

- **FR-011**: 사용자는 카드 상세에서 본문과 종류를 수정하고 저장할 수 있어야 한다(보드 카드·독립 카드 공통 동일 방식).

**삭제 및 경고 (US4)**

- **FR-012**: 사용자는 카드를 삭제할 수 있어야 한다.
- **FR-013**: 삭제 대상 카드가 다른 카드와 연결(link)되어 있으면, 시스템은 삭제 확정 전에 경고를 표시하고 연결된 개수를 알려야 한다.
- **FR-014**: 삭제가 확정되면 카드와 그 카드에 걸린 연결이 함께 제거되어야 한다(연결 상대 카드는 보존).
- **FR-015**: 연결이 없는 카드(독립 카드 포함)는 연결 경고 없이 삭제되어야 한다.

**소유 모델 (데이터 전제)**

- **FR-016**: 독립 카드는 어느 보드에도 속하지 않고도 특정 사용자의 소유로 존재하고 관리될 수 있어야 한다(소유·타인 격리는 다른 카드와 동일하게 보장). 소유자를 판별하는 구체 메커니즘은 plan 결정 사항이다.

**보드 붙이기 (US5)**

- **FR-017**: 사용자는 **연결(link)이 없는 카드**의 소속 보드를 변경할 수 있어야 한다 — 독립→보드(붙이기), 보드→독립(떼기), 보드A→보드B(옮기기) 모두. 배정/이동 대상 보드는 요청 사용자 본인 소유여야 한다(타인 보드 지정 금지). 카드가 보드에 배정되면 그 보드 캔버스에 기본 위치로 나타나고, 독립으로 떼면 소속 보드가 없어진다.
- **FR-017a**: 다른 카드와 연결(link)이 있는 카드는 소속 보드 변경 대상에서 제외되어야 한다(연결선 처리는 본 스펙 범위 밖). 이런 카드의 보드는 기존 보드 화면 흐름에서만 다룬다.

**집필 통합 (US6)**

- **FR-019**: 집필 화면의 참조 패널은 보드 뷰와 별개로 "카드" 뷰를 제공하여, 그 작품 관련 보드(그 작품 보드 + 상위 시리즈 보드)의 카드 + 사용자의 독립 카드를 모아 보여줘야 한다. 다른 작품에만 속한 카드는 제외한다. 카드를 열어 종류·내용을 확인할 수 있다(참조 목적).
- **FR-019a**: 집필 카드 뷰는 **이 작품 보드 → 상위 시리즈 보드 → 독립 카드** 순의 3단 그룹으로 나눠 표시하고, 각 그룹 안에서 생성일 내림차순으로 정렬한다.

**범위 밖(무변경)**

- **FR-018**: 폐기된 캡처 메모(memos) 도메인과 iOS 단축어/데스크탑 캡처 파이프라인은 본 기능에서 변경되지 않으며, 카드 관리 화면에 노출되지 않는다.

### Key Entities *(include if feature involves data)*

- **카드 (Card)**: 사용자의 단순 문자열 메모 단위. 주요 속성 — 본문(문자열, 필수), 종류(character/place/event/theme 중 하나 또는 무지정), 소속 보드(0 또는 1개 — 독립 카드는 없음), 소유 사용자. 본 기능은 기존 카드 도메인을 확장한다: 소속 보드가 선택적(있을 수도 없을 수도)이 되고, 독립 카드도 사용자 소유로 판별·격리될 수 있어야 한다(소유 판별 메커니즘은 plan 결정).
- **보드 (Board)**: 카드들을 마인드맵 형태로 배치하는 캔버스. 하나의 카드는 0 또는 1개 보드에 속한다.
- **연결 (Link)**: 같은 보드 안 두 카드 사이의 연결. 카드가 삭제되면 그 카드에 걸린 연결도 함께 제거된다. 독립 카드는 보드에 없으므로 연결을 가지지 않는다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자는 여러 보드에 흩어진 카드와 독립 카드를 카드 관리 화면 한 곳에서 모두 확인할 수 있다(누락 0).
- **SC-002**: 사용자는 진입 후 3단계 이내(진입 → 본문 입력 → 저장)로 새 독립 메모 카드를 만들 수 있다.
- **SC-003**: 목록에 나타난 모든 카드에 대해 소속 보드(또는 "없음")가 정확히 표시된다(정확도 100%).
- **SC-004**: 다른 카드와 연결된 카드를 삭제하려 할 때, 100% 경고가 먼저 표시되고 연결 개수가 함께 표시된다.
- **SC-005**: 삭제 확정 시 대상 카드와 그 연결만 제거되고, 연결 상대 카드·다른 카드·보드는 영향받지 않는다.
- **SC-006**: 연결 없는 카드의 소속 보드를 바꾸면(붙이기·떼기·옮기기), 카드 관리 목록의 소속 보드 표시와 관련 보드 캔버스가 그에 맞게 정확히 갱신된다.

## Assumptions

- **데이터 도메인**: 본 기능이 다루는 "카드"는 기존 보드 카드(card) 도메인이다. 과거 지원하다 프론트 UI만 폐기된 캡처 메모(memos)와 그 캡처 파이프라인은 별개 데이터이며 본 기능에서 무변경(FR-018).
- **소유·독립 데이터 모델**: 독립 카드(보드 없는 카드)를 허용하려면 데이터 모델 변경(마이그레이션)이 필요하다 — 최소한 카드의 보드 소속이 선택적이어야 하고, 보드를 거치지 않고도 소유·타인 격리가 보장돼야 한다. 구현 방식(카드의 보드 참조를 직접 nullable 로 두고 카드에 소유자 참조를 추가하는 방식 vs 사용자별 숨은 "인박스" 보드를 두어 소유를 보드 경유로 유지하는 방식)은 plan 단계에서 결정한다.
- **MVP 위상(plan 에서 명시)**: US1(기존 보드 카드를 여러 보드를 가로질러 읽기)은 스키마 변경 없이 가능하다 — 마이그레이션은 US2 이상(독립 카드)에서만 필요하다. plan 은 마이그레이션 없는 조회 MVP 를 먼저 끊고 그 위에 독립 카드를 얹는 위상으로 구성할 수 있다(BE 선행 배포순서 정합).
- **폐기 메모 프론트 잔재**: 과거 메모 UI 폐기 후 프론트에 미사용 고아 코드(`lib/query/useMemos.ts`, `lib/api/memo.ts`, `lib/memoView.ts`, `lib/electron-api/memos.ts`, `MemoResponse` 등)와 `/memos → /boards` redirect 가 남아 있다. plan 은 (a) 새 카드 관리 라우트명이 이 redirect 와 충돌하지 않도록, (b) 고아 코드의 정리/재사용 여부를 함께 결정한다.
- **카드 종류**: 기존 4종(character/place/event/theme)을 재사용하고 무지정을 허용한다(새 종류 체계 도입 없음).
- **정렬**: 목록 기본 정렬 = 최근 수정순(앱 내 다른 목록과 정합).
- **표시값 출처(plan 에서 확정)**: 소속 보드 이름과 "연결 N개" 카운트를 여러 보드를 가로질러 제공하려면, 기존 단일 보드 조회 외에 카드 목록/상세용 조회 경로를 신설해야 한다(프로젝트 룰 — 표시값 출처를 plan 에 명시).
- **진입점·화면 배치(2026-07-01 목업 확정 — `docs/research/2026-07-01-card-management-mockup.html`)**: 카드 관리는 **`/boards` 화면의 하위 탭**([보드 | 카드])으로 진입한다(최상위 NAV 신규 항목·신규 top-level 라우트 없음 — 044 가 메모·인물 메뉴를 보드로 통합한 방향과 정합). 목록 = **그리드**(종류색 틴트 타일 + 소속 보드 라벨 + 연결 배지, 최근 수정순). 상세 = **우측 슬라이드오버**(종류 변경·본문 수정·소속 보드 재배정·삭제). 연결 있는 카드는 재배정 **잠금**(FR-017a). 삭제 = 연결 있으면 "N개의 다른 카드와 연결" 경고 다이얼로그. 독립 생성 = 인라인 폼(빈 내용 막힘·IME 조합 가드). 빈 상태 = 화면 컨텍스트 유지 오버레이.
- **집필 화면 카드 뷰(R4, 2026-07-01 확정)**: 집필 화면의 "보드 참조" 패널에 [보드 | 카드] 토글을 더해, 그 작품 관련 보드(그 작품 보드 + 상위 시리즈 보드)의 카드 + 독립 카드를 모아 본다. 조회는 기존 계약(전체 카드 목록 `GET /api/cards` + 그 작품 참조 보드 `GET /boards/reference`)을 재사용해 FE 에서 범위 필터(card.boardId ∈ 참조 보드 id 집합 또는 독립) — 신규 백엔드 0. 표시는 **이 작품 보드 → 상위 시리즈 보드 → 독립** 3단 그룹, 각 그룹 안 생성일 내림차순(목업 `docs/research/2026-07-01-writing-card-view-mockup.html`). 참조 목적(열람·상세 확인) 중심이며, 관리(생성·재배정·삭제 등)의 주 무대는 `/boards` 카드 탭.
- **범위 밖(후속)**: (a) 이미 보드에 속하고 연결선이 걸린 카드를 다른 보드로 이동하거나 독립 상태로 되돌릴 때의 연결선 처리, (b) 캡처 메모(memos)와 카드의 데이터 통합은 본 스펙 범위 밖이며 필요 시 별도 작업으로 다룬다.
- **인증·소유 스코프**: 기존 인증·사용자 소유 판별 방식을 재사용한다(카드는 사용자 소유, 타인 데이터 격리).
