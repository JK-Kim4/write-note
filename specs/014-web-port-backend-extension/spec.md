## User Scenarios & Testing *(mandatory)*

본 작업의 사용자는 **web 버전으로 옮겨온 작가**다. desktop(로컬 단독 Electron 앱)에서 검증된 기능 4종이 web 으로 옮겨가도 **서버에 영속**되고 **계정 단위로 격리**되어 동일하게 동작해야 한다. desktop 은 단일 기기·단일 사용자였지만 web 은 다중 사용자·다중 기기다 — 이 차이에서 발생하는 소유권·동시성·비정상 종료 경계가 본 작업의 핵심이다.

각 User Story 는 desktop 기능 1종에 대응하며, 독립적으로 구현·테스트·검증 가능한 backend 능력 단위다.

### User Story 1 - "다음 장면" 메모 서버 저장 (Priority: P1)

작가가 한 작품에 대해 "다음에 쓸 장면" 한 줄을 적어 두면, 그 값이 서버에 저장되고 다른 기기에서 다시 로그인해도 그대로 보인다. 작품 목록(작품 벽 카드)과 재진입 시 "여기서부터 쓰면 된다"는 안내값으로 쓰인다.

**Why this priority**: desktop `projects.next_scene` 는 단순 텍스트 컬럼이며 기존 작품 수정 흐름(`PATCH /api/projects/{id}`)에 얹기만 하면 된다. 가장 작은 변경으로 "컬럼 추가 + 마이그레이션 + 소유권 검증" 의 backend 확장 패턴을 먼저 확립한다. 이후 곁쪽지 고정(US2)이 같은 패턴을 재사용한다.

**Independent Test**: 작가 계정으로 작품을 만들고 "다음 장면" 값을 저장한 뒤, 작품을 다시 조회하면 그 값이 그대로 반환됨을 확인한다. 다른 계정으로는 그 작품의 "다음 장면" 을 조회·수정할 수 없음을 확인한다.

**Acceptance Scenarios**:

1. **Given** 작가 A 의 작품 P 가 "다음 장면" 미설정 상태일 때, **When** A 가 "다음 장면" 에 한 줄을 저장하면, **Then** P 를 다시 조회했을 때 그 값이 반환된다.
2. **Given** "다음 장면" 이 설정된 작품 P, **When** A 가 빈 값으로 갱신하면, **Then** "다음 장면" 이 미설정(빈 값) 상태로 돌아간다.
3. **Given** 작가 A 의 작품 P, **When** 작가 B 가 P 의 "다음 장면" 수정을 시도하면, **Then** 거부되고(소유권 없음) P 의 값은 변하지 않는다.
4. **Given** "다음 장면" 만 갱신하는 요청, **When** 처리되면, **Then** 작품의 다른 메타(제목·장르 등)는 변하지 않는다(부분 수정).

### User Story 2 - 곁쪽지 고정(작품당 1개) (Priority: P1)

작가가 한 작품에 연결된 곁쪽지(메모) 중 하나를 "고정"하면 그 작품에서 항상 위로 보인다. 한 작품에는 **단 하나의 곁쪽지만 고정**될 수 있어, 새로 고정하면 이전 고정은 자동 해제된다. 고정 상태는 서버에 영속되어 기기 간 동일하게 보인다.

**Why this priority**: desktop `memo_projects.pinned` 는 메모↔작품 연결 단위의 플래그이며 "작품당 1개 고정" 불변식이 핵심이다. US1 과 같은 "컬럼 추가 + 토글 endpoint" 패턴을 재사용하되 불변식 검증이 더해진다. 곁쪽지는 작가의 재진입 맥락 유지에 직접 기여하므로 P1.

**Independent Test**: 한 작품에 곁쪽지 2개를 연결하고 하나를 고정한 뒤, 다른 하나를 고정하면 첫 번째 고정이 자동 해제되고 두 번째만 고정 상태임을 확인한다. 고정 상태가 조회 결과에 반영됨을 확인한다.

**Acceptance Scenarios**:

1. **Given** 작품 P 에 연결된 곁쪽지 M1(미고정), **When** 작가가 M1 을 고정하면, **Then** P 의 곁쪽지 조회에서 M1 이 고정 상태로 표시된다.
2. **Given** 작품 P 에서 M1 이 고정된 상태, **When** 작가가 같은 작품의 M2 를 고정하면, **Then** M1 은 자동 해제되고 M2 만 고정된다(작품당 1개 불변식).
3. **Given** 고정된 곁쪽지 M1, **When** 작가가 M1 의 고정을 해제하면, **Then** P 에 고정된 곁쪽지가 없는 상태가 된다.
4. **Given** 같은 메모 M 이 작품 P1·P2 에 연결되어 있고 P1 에서 고정될 때, **When** P1 에서 M 을 고정하면, **Then** P2 에서의 M 고정 상태는 영향받지 않는다(연결 단위 고정).
5. **Given** 작가 A 의 작품, **When** 작가 B 가 그 작품의 곁쪽지 고정을 시도하면, **Then** 거부된다(소유권).

### User Story 3 - 집필 기록(작품 로그) 영속·조회 (Priority: P2)

작가가 한 작업 세션을 마치며 남긴 짧은 기록(예: "3장 도입부 완성, 다음은 갈등 고조")이 서버에 시간순으로 쌓이고, 작품별로 다시 볼 수 있다. 작품 벽 카드에는 가장 최근 기록 한 줄이 요약으로 보인다.

**Why this priority**: desktop `project_logs` 는 신규 엔티티(테이블)다. US1·US2 의 단순 컬럼 추가와 달리 엔티티·조회 경로 신설이 필요해 P2. 집필 기록은 작업 세션 종료(US4)와 트랜잭션으로 묶이므로 US4 의 선행 조건이다.

**Independent Test**: 작품에 기록 여러 개를 시간 간격을 두고 생성한 뒤, 작품별 기록을 조회하면 최신순으로 반환됨을 확인한다. 작품 카드용 "최신 기록 1개" 조회가 올바른 최신값을 반환함을 확인한다. 타 계정은 조회 불가함을 확인한다.

**Acceptance Scenarios**:

1. **Given** 작품 P 에 기록이 없을 때, **When** 작가가 (세션과 무관하게) 기록을 직접 남기면, **Then** 그 기록이 작품 P 의 기록 목록에 나타난다.
2. **Given** 작품 P 에 기록 3개가 서로 다른 시각에 생성됨, **When** 작가가 P 의 기록을 조회하면, **Then** 생성 최신순으로 정렬되어 반환된다.
3. **Given** 작품 P 에 기록이 여러 개, **When** "최신 기록 1개" 를 조회하면, **Then** 가장 최근 1건이 반환된다(기록 없으면 "없음").
4. **Given** 작가 A 의 작품 기록, **When** 작가 B 가 조회를 시도하면, **Then** 거부된다(소유권).
5. **Given** 작품 P 가 삭제되면, **When** 처리되면, **Then** P 에 속한 모든 집필 기록도 함께 제거된다.

### User Story 4 - 작업 세션 추적(시작/종료/짧은세션 폐기/종료+기록/비정상정리) (Priority: P3)

작가가 한 작품의 집필실에 들어가면 작업 세션이 시작되고, 떠나면 종료된다. 너무 짧은(집중 작업으로 보기 어려운) 세션은 통계에서 버려진다. 작가가 "오늘 작업 마치며 기록 남기기" 를 하면 세션 종료와 집필 기록 저장이 한 번에(원자적으로) 처리된다. 브라우저를 그냥 닫는 등 정상 종료 신호 없이 끊긴 세션은 작업 시간 통계를 부풀리지 않도록 정리된다.

**Why this priority**: desktop `work_sessions` 는 신규 엔티티이면서 동작 규칙(임계값 폐기·원자적 종료+기록·비정상 정리)이 가장 복잡하다. web 의 다중 기기·비정상 종료 경계가 desktop 의 "앱 시작 시 정리" 와 1:1 매핑되지 않아 가장 큰 설계 판단을 요한다. 따라서 마지막(P3)에 두고, US3(집필 기록)에 의존한다.

**Independent Test**: 세션을 시작하고 임계값 미만에 종료하면 기록이 남지 않음(폐기)을, 임계값 이상에 종료하면 세션이 보존됨을 확인한다. "종료+기록" 호출 시 짧은 세션이라도 세션과 기록이 함께 보존됨을 확인한다. 한 작품에 열린 세션이 있는 상태에서 새 세션을 시작하면 이전 열린 세션이 정리됨을 확인한다.

**Acceptance Scenarios**:

1. **Given** 작품 P 에 열린 세션이 없을 때, **When** 작가가 세션을 시작하면, **Then** P 에 열린 세션 1개가 생긴다.
2. **Given** 작품 P 에 이미 열린 세션이 있을 때, **When** 작가가 P 의 새 세션을 시작하면, **Then** 이전 열린 세션이 먼저 정리되고(작품당 열린 세션 1개 불변식) 새 세션이 시작된다.
3. **Given** 시작된 지 임계값(desktop 기준 30초) 미만의 열린 세션, **When** 자동 종료되면, **Then** 그 세션은 폐기되어 통계에 포함되지 않는다.
4. **Given** 시작된 지 임계값 이상의 열린 세션, **When** 자동 종료되면, **Then** 종료 시각이 기록되고 세션이 보존된다.
5. **Given** 임계값 미만의 짧은 세션, **When** 작가가 "종료+기록" 을 하면, **Then** 짧아도 세션이 보존되고 집필 기록이 함께 저장된다(자동 종료의 폐기 규칙 우회, 원자적 처리).
6. **Given** "종료+기록" 처리 도중 집필 기록 저장이 실패하면, **When** 트랜잭션이 롤백되면, **Then** 세션 종료도 적용되지 않는다(부분 적용 없음).
7. **Given** 정상 종료 신호 없이 끊긴(dangling) 열린 세션이 현실적 최대 작업 길이를 초과해 열려 있을 때, **When** 서버의 주기적 정리 작업이 동작하면, **Then** 해당 세션은 작업 시간 통계를 부풀리지 않도록 폐기된다.

### Edge Cases

- **"다음 장면" 빈 값/공백**: 빈 문자열은 "미설정" 으로 취급(desktop 기본값 `''` 정합). 작가가 의도적으로 비울 수 있어야 한다.
- **곁쪽지 고정 동시성**: 같은 작품에서 두 기기가 거의 동시에 서로 다른 곁쪽지를 고정하면, "작품당 1개" 불변식이 최종적으로 일관되게 유지되어야 한다(마지막 고정이 승리).
- **존재하지 않는/타 계정 작품**: 모든 신규 동작은 작품 소유권 검증 실패 시 "찾을 수 없음(404)" 로 응답(기존 패턴 정합 — `ResourceNotFoundException`).
- **작업 세션 동시 시작(다중 기기)**: 같은 작품에 두 기기가 동시에 세션을 시작하면 "작품당 열린 세션 1개" 불변식이 유지되어야 한다(이전 열린 세션 정리 후 시작).
- **작품 삭제 시 연쇄**: 작품 삭제 시 그 작품의 집필 기록·작업 세션도 함께 제거된다(desktop `ON DELETE CASCADE` 정합).
- **집필 기록 빈 본문**: 빈 본문 기록 허용 여부 — desktop 은 `body TEXT NOT NULL` 이나 빈 문자열 허용 가능. 본문 검증 정책은 기존 메모 검증 패턴과 정합(가정 참조).

## Requirements *(mandatory)*

### Functional Requirements

**"다음 장면"(US1)**

- **FR-001**: 시스템은 작품마다 "다음 장면" 한 줄 텍스트를 저장할 수 있어야 하며, 미설정 기본값은 빈 값으로 취급해야 한다.
- **FR-002**: 작가는 기존 작품 부분 수정 흐름을 통해 "다음 장면" 을 설정·변경·비울 수 있어야 한다.
- **FR-003**: "다음 장면" 갱신은 해당 작품의 다른 메타데이터를 변경하지 않아야 한다(부분 수정).
- **FR-004**: 작품 조회 결과에 "다음 장면" 값이 포함되어야 한다(작품 벽 카드 요약·재진입 안내용).

**곁쪽지 고정(US2)**

- **FR-005**: 시스템은 (곁쪽지, 작품) 연결 단위로 고정 여부를 저장할 수 있어야 한다.
- **FR-006**: 작가는 한 작품에 연결된 곁쪽지의 고정을 켜고 끌 수 있어야 한다.
- **FR-007**: 한 작품에는 동시에 최대 1개의 곁쪽지만 고정될 수 있어야 하며, 새 고정 시 이전 고정은 자동 해제되어야 한다(작품당 1개 불변식).
- **FR-008**: 같은 곁쪽지가 여러 작품에 연결된 경우, 한 작품에서의 고정은 다른 작품에서의 고정 상태에 영향을 주지 않아야 한다.
- **FR-009**: 곁쪽지/작품 목록 조회 결과에 고정 상태가 반영되어야 한다.

**집필 기록(US3)**

- **FR-010**: 시스템은 작품별 집필 기록(자유 텍스트 본문 + 생성 시각)을 저장할 수 있어야 한다.
- **FR-011**: 작가는 작품의 집필 기록을 생성 최신순으로 조회할 수 있어야 한다.
- **FR-012**: 시스템은 작품의 "가장 최근 집필 기록 1건" 을 조회할 수 있어야 한다(작품 카드 요약용, 없으면 "없음").
- **FR-013**: 작품이 삭제되면 그 작품의 모든 집필 기록도 함께 제거되어야 한다.
- **FR-014**: 시스템은 집필 기록을 두 경로로 생성할 수 있어야 한다 — (1) 작가가 작업 세션과 무관하게 직접 기록을 추가하는 독립 생성 경로(마스터 설계 §4 의 POST 채택), (2) 작업 세션 "종료+기록"(FR-019) 경로. 두 경로 모두 동일한 집필 기록을 만든다.

**작업 세션(US4)**

- **FR-015**: 시스템은 작품별 작업 세션(시작 시각, 종료 시각)을 저장할 수 있어야 한다. 종료 시각이 없는 세션은 "열린 세션" 으로 취급한다.
- **FR-016**: 작가가 작품의 세션을 시작할 때, 그 작품에 이미 열린 세션이 있으면 먼저 정리한 뒤 새 세션을 시작해야 한다(작품당 열린 세션 1개 불변식).
- **FR-017**: 세션 자동 종료 시, 지속 시간이 임계값(desktop 기준 30초) 미만이면 그 세션을 폐기해야 한다(통계 미포함).
- **FR-018**: 세션 자동 종료 시, 지속 시간이 임계값 이상이면 종료 시각을 기록하고 세션을 보존해야 한다.
- **FR-019**: 작가의 "종료+기록" 동작은 세션 종료와 집필 기록 저장을 하나의 원자적 트랜잭션으로 처리해야 하며, 이 경로에서는 임계값 미만의 짧은 세션이라도 폐기하지 않고 보존해야 한다.
- **FR-020**: "종료+기록" 처리 중 일부가 실패하면 전체가 롤백되어야 한다(세션 종료/기록 저장의 부분 적용 없음).
- **FR-021**: 시스템은 서버 측 주기적 정리 작업으로, 현실적 최대 작업 길이를 초과해 열려 있는(dangling) 세션을 폐기해야 한다 — 정상 종료 신호 없이 끊긴 세션이 작업 시간 통계를 부풀리지 않게 한다. (정리 트리거 = 클라이언트 신호가 아닌 서버 스케줄 작업)
- **FR-021a**: 정리 기준이 되는 "최대 작업 길이" 임계값과 정리 작업 주기는 운영 가능한 설정값이어야 한다(구체 값·주기는 plan 단계에서 확정).

**소유권·계정 격리(전 항목 공통)**

- **FR-022**: 본 작업의 모든 신규 데이터(다음 장면·고정·집필 기록·작업 세션)는 작품 소유자(계정) 단위로 격리되어야 하며, 타 계정의 접근(조회·수정)은 거부되어야 한다(기존 JWT principal.userId 기반 소유권 검증 패턴 정합).

**계약(contract) — front 이식 선행 산출물**

- **FR-023**: desktop 의 IPC 채널(작업 세션 3종 `sessions:start`/`sessions:end`/`sessions:endWithLog`, 집필 기록 2종 `logs:list`/`logs:listByProject`, 곁쪽지 고정 `memos:setPin`, "다음 장면" 을 포함한 `projects:update`)과 그에 대응하는 서버 API 사이의 계약 매핑이 본 작업 산출물로 확정되어야 한다(하위 작업 2 front 이식의 입력 계약). FR-014 의 독립 집필 기록 생성 endpoint 는 desktop IPC 채널에 대응 항목이 없는 **web 추가 endpoint** 로 계약에 별도 표기한다.

### Key Entities *(include if feature involves data)*

- **작품(Project) — 확장**: 기존 작품 엔티티에 "다음 장면"(한 줄 텍스트, 미설정 시 빈 값) 속성이 추가된다. 소유자(계정)에 귀속.
- **곁쪽지-작품 연결(MemoProject) — 확장**: 기존 메모↔작품 연결에 "고정 여부" 속성이 추가된다. 작품당 최대 1개 고정 불변식을 가진다.
- **집필 기록(ProjectLog) — 신규**: 작품에 귀속되는 자유 텍스트 기록. 속성 — 본문, 생성 시각. 작품 삭제 시 연쇄 제거. 작품 단위 최신순 조회·최신 1건 조회 지원.
- **작업 세션(WorkSession) — 신규**: 작품에 귀속되는 집필 시간 구간. 속성 — 시작 시각, 종료 시각(열린 세션은 미설정). 작품당 열린 세션 1개 불변식, 임계값 미만 폐기 규칙, 작품 삭제 시 연쇄 제거.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: desktop 의 대상 기능 4종(다음 장면·곁쪽지 고정·집필 기록·작업 세션)이 web 서버 저장으로 100% 동등하게 동작한다 — 각 기능의 desktop 동작 규칙(빈 값 처리·작품당 1개 고정·임계값 폐기·원자적 종료+기록)이 모두 재현된다.
- **SC-002**: 한 작가 계정에서 저장한 4종 데이터는 다른 기기에서 같은 계정으로 접근 시 동일하게 보이고, 다른 계정에서는 0건 노출된다(계정 격리 100%).
- **SC-003**: 작업 세션 통계가 정상·비정상 종료 모두에서 정확하다 — 임계값 미만 자동 종료 세션과 dangling 세션은 작업 시간 합산에 포함되지 않는다.
- **SC-004**: "종료+기록" 동작은 원자적이다 — 부분 적용(세션만 종료되고 기록 누락, 또는 그 반대)이 0건이다.
- **SC-005**: 본 작업이 산출한 IPC↔API 계약 매핑으로 하위 작업 2(front 이식)가 추가 backend 설계 결정 없이 4종 기능을 결선할 수 있다(계약 공백 0건).

## Assumptions

- **범위**: 본 spec 은 마스터 설계의 하위 작업 1(backend 확장)만 다룬다. Front 이식·등장인물 UI·모바일 캡처·로컬 `.db` 가져오기·배포는 범위 밖(각자 별도 spec).
- **재활용 기반**: 기존 backend 의 인증(JWT)·멀티유저 소유권 검증 패턴(`findByIdAndUserId` → `ResourceNotFoundException`)·작품/메모 CRUD·마이그레이션 컨벤션(`V{N}__{name}.sql`)을 그대로 재사용한다. 본 작업은 V6 다음 신규 마이그레이션(컬럼 2건 + 테이블 2건)을 추가한다.
- **desktop = 데이터 모델 출처**: 4종 기능의 동작 규칙·기본값·불변식은 desktop 구현(`desktop/electron/db/`)을 정본(SoT)으로 삼는다 — "다음 장면" 빈 문자열 기본값, 곁쪽지 작품당 1개 고정, 작업 세션 30초 임계값 폐기, dangling 정리, 원자적 종료+기록.
- **다중 기기 세션 불변식**: desktop 은 단일 기기였으나 web 은 다중 기기다. "작품당 열린 세션 1개" 불변식은 **계정+작품 단위로 기기 무관하게** 유지한다(세션 시작 시 해당 작품의 기존 열린 세션을 정리). 기기별 동시 열린 세션은 허용하지 않는다.
- **집필 기록 본문 검증**: 본문 길이·빈 본문 허용 정책은 기존 메모 본문 검증 패턴과 정합한다(별도 요구 없으면 desktop 의 `NOT NULL` + 빈 문자열 허용 수준).
- **시각 기준**: 시작/종료/생성 시각은 서버 시계 기준(기존 엔티티 `createdAt`/`updatedAt` 패턴 정합). desktop 의 클라이언트 시계 기록은 서버 권위 시각으로 대체한다.
- **계약 산출물의 위치/형식**: IPC↔API 계약 매핑표의 구체 산출 위치·형식은 plan 단계(`/speckit-plan`)에서 contracts 산출물로 확정한다.
- **집필 기록 독립 생성(Q1 확정)**: 마스터 설계 §4 의 "POST" 를 채택해 세션과 무관한 독립 생성 endpoint 를 둔다. desktop front 는 현재 이 통로를 쓰지 않으므로(세션 종료+기록만 사용) backend 능력으로 제공하되, front 이식(하위 작업 2)에서의 노출 여부는 별도 결정이다.
- **dangling 세션 정리(Q2 확정)**: 클라이언트 heartbeat 가 아닌 **서버 측 주기 스케줄 작업**으로 정리한다(기존 `TokenCleanupService` 류 스케줄 패턴 정합). "최대 작업 길이" 임계값과 정리 주기의 구체 값은 plan 에서 확정하며, dangling 세션은 의미 있는 종료 시각이 없으므로 폐기(보존하지 않음)한다.
