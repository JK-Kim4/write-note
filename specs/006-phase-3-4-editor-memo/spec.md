## User Scenarios & Testing *(mandatory)*

본 기능은 write-note 의 본질 — *"컨텍스트가 안 죽는 작가용 작업공간"* — 의 두 핵심 흐름을 구현한다: **(1) 끊김 없이 본문을 쓰는 작성 경험**, **(2) 떠오른 생각을 마찰 없이 잡아 나중에 큐레이션하는 메모 흐름**. 사용자는 본인(단일 작가)이며, 데스크탑에서 작성하고 모바일/데스크탑 어디서나 메모를 캡처한다.

진행은 005 와 동일한 MVP 분할 방식을 따른다: **US1(본문 작성+자동저장)을 먼저 완성·검증한 뒤** 나머지로 확장한다.

### User Story 1 - 본문을 끊김 없이 쓰고 자동 저장한다 (Priority: P1)

작가가 프로젝트를 열면 그 프로젝트의 본문이 바로 편집 가능한 상태로 나타난다. 글을 쓰는 동안 별도의 "저장" 동작 없이 변경이 자동으로 보존되고, 현재 분량과 목표 대비 진행률이 항상 보인다. 한국어 입력(조합 중인 자모, 한자 변환 포함)이 깨지거나 유실되지 않는다.

**Why this priority**: 작성은 이 도구의 존재 이유다. 자동 저장 + 한국어 안정성이 없으면 나머지 기능은 무의미하다. 이 하나만으로도 "본문을 쓰고 다음에 와도 그대로 있다"는 최소 가치를 단독으로 제공한다.

**Independent Test**: 로그인 → 프로젝트 열기 → 본문 입력 → 일정 시간 멈춤 → 페이지 새로고침 시 마지막 내용이 그대로 복원되는지, 작성 중 자수/진행률이 갱신되는지로 단독 검증 가능.

**Acceptance Scenarios**:

1. **Given** 작가가 자신의 프로젝트를 열었을 때, **When** 본문을 입력하고 작성을 잠시 멈추면, **Then** 명시적 저장 동작 없이 변경이 보존되고 다시 열었을 때 마지막 내용이 복원된다.
2. **Given** 작가가 한국어를 빠르게 입력하며 조합 중 서식(굵게)을 토글하거나 한자 변환을 할 때, **When** 자동 저장이 일어나면, **Then** 글자 깨짐·유실 없이 입력한 그대로 저장된다 (PoC 0-1 의 4 케이스).
3. **Given** 본문에 목표 분량이 설정된 프로젝트에서, **When** 작가가 글을 쓰면, **Then** 현재 자수와 목표 대비 진행률이 실시간으로 갱신된다.
4. **Given** 같은 본문이 다른 탭/기기에서 먼저 수정되어 저장된 상태에서, **When** 작가가 오래된 버전 위에 자동 저장을 시도하면, **Then** 시스템이 충돌을 감지해 경고하고 "다른 곳에서 수정됨 — 다시 불러오기 / 내 것으로 덮어쓰기" 선택을 작가에게 제시한다 (자동 덮어쓰기하지 않음).

### User Story 2 - 원고지 격자 위에서 쓰고 매수를 본다 (Priority: P2)

작가가 설정에서 작성 모드를 "원고지"로, 크기를 200/400/1000자 중 하나로 정하면, 작성 화면이 그 모드로 고정된다. 원고지 모드에서는 격자(행·열) + 컬럼 마커 + 행 번호가 보이고, 현재 매수가 자동 계산된다. 크기를 바꾸면 같은 글이 새 격자 기준으로 재배치되고 매수가 자동 변환된다.

**Why this priority**: 원고지는 한국 작가(특히 단막극·공모전 제출)의 실제 작업 양식으로, 이 도구의 차별점이다. 단 본문 작성(US1)이 동작해야 의미가 있으므로 P2.

**Independent Test**: 설정에서 원고지 모드 + 400자 선택 → 작성 화면에 20×20 격자/마커/행번호 표시 → 글 입력 후 매수 표시 확인 → 200자로 변경 시 글 유실 없이 재배치 + 매수 재계산으로 단독 검증.

**Acceptance Scenarios**:

1. **Given** 설정에서 원고지 모드 + 400자(20×20)를 선택한 작가가, **When** 작성 화면에 들어가면, **Then** 격자 + 5/10/15/20 컬럼 마커 + 행 번호가 표시된다.
2. **Given** 원고지에 글을 쓴 상태에서, **When** 설정에서 크기를 200자로 바꾸면, **Then** 글 유실 없이 새 격자에 재배치되고 매수가 자동 변환된다.

---

### User Story 3 - 떠오른 생각을 마찰 0 으로 캡처한다 (Priority: P2)

작가가 일상 중(이동/취침 전) 떠오른 장면·대사·관찰을 모바일에서 본문만 던지면 즉시 inbox 에 들어간다. 어떤 필드도 강제되지 않는다. 데스크탑에서는 단축키로 같은 방식으로 캡처하며, 당시 작업 중이던 프로젝트가 자동 기록된다. 캡처된 메모는 inbox 에 시각·출처와 함께 쌓인다.

**Why this priority**: "마찰 0 캡처 → 나중 큐레이션"은 write-note 의 가장 독창적 본질이다. 캡처가 inbox 에 도착하는 것까지가 P2 의 독립 슬라이스(큐레이션은 US4).

**Independent Test**: 모바일 캡처 수단으로 본문 전송 → inbox 에 1건 도착 확인 / 데스크탑 단축키로 캡처 → 활성 프로젝트가 함께 기록되는지 / 같은 캡처를 중복 전송 시 1건만 생성되는지로 단독 검증.

**Acceptance Scenarios**:

1. **Given** 작가가 모바일에서, **When** 본문 텍스트만 전송하면, **Then** 다른 입력 없이 메모가 inbox 에 도착하고 시각·출처(모바일)가 자동 기록된다.
2. **Given** 작가가 데스크탑에서 특정 프로젝트를 작업 중일 때, **When** 빠른 입력으로 메모를 캡처하면, **Then** 당시 활성 프로젝트가 메모에 자동 기록된다.
3. **Given** 네트워크 재시도 등으로 같은 캡처 요청이 중복 도착할 때, **When** 시스템이 이를 처리하면, **Then** inbox 에는 메모가 1건만 생성된다.

---

### User Story 4 - 쌓인 메모를 프로젝트·인물·태그로 큐레이션한다 (Priority: P3)

며칠~몇 주 후, 작가가 inbox 의 미분류 메모를 한 건씩 정리한다. 한 메모를 여러 프로젝트에 동시에 연결하고, 연결된 프로젝트들의 인물 합집합에서 등장인물을 고르고, 자유 태그와 "왜 적었나" 한 줄을 단다. 필터(전체/미분류/프로젝트/인물/태그/텍스트)로 메모를 찾을 수 있고, 미분류 상태로 영구히 두는 것도 허용된다.

**Why this priority**: 캡처된 메모의 가치를 살리는 단계지만, 캡처(US3)가 선행해야 한다. 분류 없이도 캡처 자체로 가치가 있으므로 P3.

**Independent Test**: inbox 메모 1건을 두 프로젝트에 연결 + 인물/태그/이유 입력 → 저장 → 각 프로젝트 필터에서 모두 보이는지 / 미분류 필터에서 사라지는지로 단독 검증.

**Acceptance Scenarios**:

1. **Given** inbox 에 미분류 메모가 있을 때, **When** 작가가 한 메모를 두 개의 프로젝트에 연결하고 저장하면, **Then** 두 프로젝트 필터 모두에서 해당 메모가 보인다.
2. **Given** 한 메모가 두 프로젝트에 연결된 상태에서, **When** 등장인물을 연결하려 하면, **Then** 그 두 프로젝트의 인물 합집합에서만 고를 수 있다.
3. **Given** 메모를 어떤 프로젝트에도 연결하지 않을 때, **When** 작가가 그대로 두면, **Then** 메모는 미분류 상태로 영구 보존된다.

---

### User Story 5 - 모바일 캡처용 토큰을 발급·관리한다 (Priority: P3)

작가가 설정에서 모바일 캡처용 토큰을 발급받는다. 발급 시 원본 토큰이 1회만 표시되며, 이후 목록에서는 식별용 접두사·라벨·마지막 사용 시각만 보인다. 토큰에 라벨을 붙이거나 해지할 수 있고, 해지된 토큰으로는 더 이상 캡처할 수 없다.

**Why this priority**: US3 의 모바일 캡처가 실제로 동작하려면 토큰이 필요하지만, 본인 1대 셋업이라 우선순위는 낮다. 데스크탑 캡처(US3)는 토큰 없이 동작하므로 분리 가능.

**Independent Test**: 토큰 발급 → 원본 1회 표시 확인 → 그 토큰으로 모바일 캡처 성공 → 토큰 해지 → 같은 토큰으로 캡처 시 거부됨으로 단독 검증.

**Acceptance Scenarios**:

1. **Given** 작가가 설정에서 토큰 발급을 요청할 때, **When** 발급이 완료되면, **Then** 원본 토큰이 1회만 표시되고 이후 목록에는 다시 표시되지 않는다.
2. **Given** 발급된 토큰을 해지한 뒤, **When** 그 토큰으로 모바일 캡처를 시도하면, **Then** 캡처가 거부된다.

---

### Edge Cases

- 본문 작성 중 네트워크가 끊겼다가 복구되면, 보존되지 않은 변경은 어떻게 되는가? (자동 저장 재시도 / 사용자 인지)
- 빈 본문(글자 0)을 자동 저장하면 어떻게 처리되는가?
- 원고지 크기를 줄였을 때 기존 글이 새 격자보다 길면 어떻게 표시되는가?
- 한국어 IME 조합이 끝나기 전에 자동 저장 시점이 도래하면? (조합 완료 후 저장)
- 미분류 메모가 수백 건 쌓였을 때 inbox 필터/목록 성능은?
- 큐레이션 중 연결된 프로젝트를 해제하면, 그 프로젝트에만 속하던 인물 연결은 어떻게 되는가?
- 해지·만료된 토큰, 형식이 잘못된 토큰으로 캡처 시도 시 응답은?
- 메모를 삭제하면 그 메모의 프로젝트/인물 연결도 함께 사라지는가?

## Requirements *(mandatory)*

### Functional Requirements

**본문 작성 / 자동 저장 (US1)**

- **FR-001**: 작가가 프로젝트를 열면 시스템은 그 프로젝트 식별자로 해당 프로젝트의 본문을 직접 조회하여(프로젝트당 본문 1개), 바로 편집 가능한 상태로 표시해야 한다.
- **FR-002**: 시스템은 작성이 잠시 멈춘 뒤 명시적 저장 동작 없이 본문 변경을 자동으로 보존해야 한다.
- **FR-003**: 시스템은 한국어 입력(빠른 타자, 조합 중 서식 토글, 한자 변환, 자모 단위 삭제) 중에도 글자 깨짐·유실 없이 본문을 보존해야 한다.
- **FR-004**: 시스템은 현재 자수(공백 제외 기준)와 목표 분량 대비 진행률을 작성 화면에 표시해야 한다.
- **FR-005**: 시스템은 본문 제목을 본문 내용과 분리하여 수정할 수 있게 해야 한다.
- **FR-006**: 같은 본문이 더 최신 버전으로 이미 저장된 상태에서 오래된 버전 위에 저장이 시도되면, 시스템은 충돌을 감지하고 자동으로 덮어쓰지 않으며, 작가에게 "다시 불러오기 / 내 것으로 덮어쓰기" 선택을 제시해야 한다 (`03-backend-requirements.md` §3-4 정합).

**원고지 / 작성 모드 (US2)**

- **FR-007**: 작가는 설정에서 작성 모드(에디터/원고지)와 원고지 크기(200/400/1000자)를 한 번 정할 수 있고, 작성 화면은 그 모드로 고정되어야 한다 (작성 화면에 모드 토글 없음).
- **FR-008**: 원고지 모드에서 시스템은 선택된 크기에 맞는 격자, 컬럼 마커(5/10/15/20), 행 번호를 표시해야 한다.
- **FR-009**: 시스템은 원고지 매수를 자동 계산하고, 크기 변경 시 본문 유실 없이 새 격자 기준으로 재배치하고 매수를 재계산해야 한다.

**작성 화면 사이드 패널 (US1/US2 보조)**

- **FR-010**: 작성 화면 사이드 패널은 해당 프로젝트의 메타(제목·장르·시놉시스 등)와 등장인물 목록을 실제 데이터로 표시해야 한다.

**메모 캡처 (US3)**

- **FR-011**: 시스템은 모바일에서 본문 텍스트만으로(다른 필드 강제 없이) 메모를 캡처하여 inbox 에 넣을 수 있어야 한다.
- **FR-012**: 시스템은 데스크탑에서 빠른 입력 수단으로 메모를 캡처하고, 당시 활성 프로젝트를 자동 기록해야 한다.
- **FR-013**: 시스템은 캡처 시점의 시각과 출처(모바일/데스크탑)를 자동 기록해야 한다.
- **FR-014**: 시스템은 동일한 캡처 요청이 중복 도착해도 메모를 1건만 생성해야 한다 (멱등성).

**메모 inbox / 큐레이션 (US4)**

- **FR-015**: 시스템은 미분류 메모를 모은 inbox 와 필터(전체/미분류/프로젝트별/인물별/태그별/텍스트 검색)를 제공해야 한다.
- **FR-016**: 시스템은 한 메모를 여러 프로젝트에 동시에 연결할 수 있게 해야 한다.
- **FR-017**: 메모에 등장인물을 연결할 때, 시스템은 그 메모가 연결된 프로젝트들의 인물 합집합에서만 선택하도록 보장해야 한다.
- **FR-018**: 작가는 메모에 자유 태그와 "왜 적었나" 이유 노트를 달 수 있어야 한다.
- **FR-019**: 시스템은 어떤 프로젝트에도 연결되지 않은 메모를 미분류 상태로 영구 보존해야 한다.
- **FR-020**: 작가는 메모의 본문/태그/이유 노트를 수정하고 메모를 삭제할 수 있어야 하며, 삭제 시 그 메모의 프로젝트·인물 연결도 함께 정리되어야 한다.

**모바일 캡처용 토큰 (US5)**

- **FR-021**: 작가는 모바일 캡처용 토큰을 발급할 수 있고, 발급 시 원본 토큰은 1회만 표시되어야 한다.
- **FR-022**: 작가는 본인 토큰 목록(식별용 접두사·라벨·마지막 사용 시각)을 조회하고, 라벨을 수정하거나 해지할 수 있어야 한다.
- **FR-023**: 시스템은 해지된·형식이 잘못된 토큰으로의 캡처를 거부해야 한다.

**공통 권한**

- **FR-024**: 모든 본문·메모·토큰 접근은 본인 소유 자원으로 격리되어야 하며, 타인 자원 접근은 "없음"으로 응답해야 한다 (정보 노출 회피).

### Key Entities *(include if feature involves data)*

- **Document(본문)**: 프로젝트당 1개. 본문 내용(구조화 텍스트), 제목, 자수, 동시 편집 충돌 감지를 위한 버전을 가진다. *(entity·테이블은 이미 존재 — 본 기능은 조회/저장 흐름 신설)*
- **Memo(메모)**: 작가가 캡처한 한 조각의 생각. 본문 텍스트, 출처(모바일/데스크탑), 캡처 시각, 캡처 당시 활성 프로젝트, 태그 목록, "왜 적었나" 이유 노트를 가진다.
- **MemoProject(메모-프로젝트 연결)**: 메모와 프로젝트의 다대다 연결. 연결 0개 = 미분류.
- **MemoProjectCharacter(메모-프로젝트-인물 연결)**: 특정 메모-프로젝트 연결에 묶인 등장인물. 해당 프로젝트의 인물이어야 한다.
- **ApiToken(모바일 캡처용 토큰)**: 장기 유효 토큰. 식별용 접두사, 라벨, 마지막 사용 시각, 해지 시각을 가진다. 원본은 발급 시 1회만 노출.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 작가가 본문을 작성한 뒤 별도 저장 동작 없이, 페이지를 새로 열어도 마지막 내용이 100% 복원된다.
- **SC-002**: 한국어 본문 작성(PoC 0-1 의 4 케이스: 빠른 타자/조합 중 서식/한자 변환/자모 삭제) 중 글자 깨짐·유실이 0건이다.
- **SC-003**: 작가가 떠오른 생각을 모바일에서 본문 외 필드 입력 없이 캡처할 수 있고, 캡처 동작이 수 초 내에 완료된다.
- **SC-004**: 캡처된 메모가 누락 없이 inbox 에 100% 도착한다.
- **SC-005**: 한 메모를 여러 프로젝트에 연결하면 각 프로젝트 필터에서 모두 조회된다.
- **SC-006**: 원고지 크기를 바꿔도 작성한 글이 유실 없이 새 격자에 재배치되고 매수가 올바르게 재계산된다.
- **SC-007**: 동일 메모를 중복 캡처해도 inbox 에 1건만 생성된다.
- **SC-008**: 해지된 토큰으로의 캡처 시도는 100% 거부된다.

## Out of Scope *(본 spec 제외 — Week 5~6 영역으로 명시 분리)*

- **메모 핀** (본문 특정 구절에 메모를 고정) — 본문 안의 핀 위치 추적은 별도의 본문 마크 + 편집 시 위치 이동 메커니즘에 의존하므로 Week 5 영역.
- **세션 노트 흐름** — "다음 세션을 위한 한 줄", 작성 화면의 지난 세션 인용 띠, 홈의 "지난 세션" hero. Week 5 영역.
- **미리보기 view 완성** — 페이지 분할·하단 footer(진행률/페이지/목차/이동). 라우트 골격만 존재, 완성은 Week 6.
- **메모 분류 후 슬라이드/페이드 애니메이션의 세부 모션 튜닝** — 동작은 포함하되 정교한 모션은 후속.
- **본문 내 텍스트 검색** — 메모 검색은 포함, 본문 검색은 Tier 2(Week 5 후순위).

## Assumptions

- 사용자는 본인 1명(단일 작가)이며, 작성은 데스크탑, 캡처는 모바일/데스크탑에서 이뤄진다 (모바일 에디터는 범위 외 — DESIGN.md 107).
- 프로젝트와 본문은 1:1이며, 본문은 프로젝트 생성 시 이미 자동으로 만들어져 있다 (004 에서 구현됨). 다중 문서는 V1.5.
- 기존 로그인(인증)과 프로젝트·등장인물 관리 기능을 그대로 재사용한다.
- 자수는 공백 제외 기준으로 서버에서 계산한다 (`03-backend-requirements.md` §2-2 Document 정의).
- 데스크탑 빠른 입력(⌘+N) 모달과 모바일 캡처 연동(iOS Shortcut)의 화면은 미디자인 영역으로, 본 기능 진행 중 최소 형태로 디자인한다. iOS Shortcut 실기기 셋업은 사용자(작가 본인)의 수동 작업 영역이다.
- 메모-프로젝트, 메모-인물 연결의 무결성(인물은 연결된 프로젝트 소속)은 애플리케이션 레벨에서 검증한다.
