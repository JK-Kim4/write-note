## 0. 유비쿼터스 언어 (Ubiquitous Language)

이 프로젝트는 **코드·DB·API·문서·대화·화면이 같은 용어**를 쓴다. 번역 계층을 두지 않는다.

| 도메인 개념 | 유비쿼터스 언어 | 화면 노출 | 비고 |
|---|---|---|---|
| 캔버스 한 장 | **Board / 보드** | "보드" | 시리즈 또는 작품에 매달림 |
| 보드 위의 한 장 | **Card / 카드** | "카드" (종류 붙이면 "인물 카드") | 과거의 "메모/노드"를 대체 |
| 카드를 잇는 선 | **Link / 연결** | (동사) "잇기·이어요" | 과거의 "엣지" |
| 카드의 종류 | **CardType** | "이건 뭔가요?" → 인물·장소·사건·테마 | 기본 무지정 |
| ~~메모 / 인물 (기존 메뉴)~~ | (폐기) | — | 보드의 Card로 흡수 |
| ~~node / edge~~ | (React Flow 어댑터 내부에서만) | — | §8 어댑터 경계 참조 |

> **핵심**: 기존 `메모`·`인물` 메뉴는 사라지고 **보드 위 Card로 통합**된다. "인물"은 별도 기능이 아니라 **`CardType = 인물`인 Card**일 뿐이다.

## 1. 배경 (Background)

기존 서비스는 `메모`(title + content 평면 노트)와 `인물`을 각각 별도 메뉴로 관리했다. 이 둘은 작품/시리즈와 느슨하게만 연결돼, 이야기 구조를 설계하는 도구로는 작동하지 못했다.

목표는 흩어진 메모·인물을, **작품/시리즈에 연결해 캔버스에 펼쳐 보고 잇는 통합 공간 = 보드(Board)** 로 재구성하는 것이다. 작가는 집필 전에 보드에서 인물·사건·설정(=Card)을 배치하고 연결(Link)해 플롯을 설계하고, 집필 중에는 보드를 참조한다.

레퍼런스: 탐정 보드의 핀-실 연결, Obsidian Canvas, Scrivener 코르크보드.

설계 원칙: **"책상에 카드를 툭 던져놓는 느낌"이어야지 "DB를 설정하는 느낌"이면 안 된다.**

## 2. 서비스 구조 맥락

상단 메뉴: `홈 · 작품 · 메모 · 인물 · 기록 · 공지`
→ **`메모`·`인물`이 보드로 개념 통합되어 제거**된다.

작품 계층:
```
작품(메뉴) → 시리즈 목록 → 시리즈 선택 → 개별 집필(작품 원고) 목록
```

보드는 이 계층 중 **시리즈 또는 개별 작품**에 매달린다(§5).

## 3. 목표 / 비목표

### 목표
- 카드를 보드에 자유롭게 배치하고 위치를 영속화한다.
- 카드를 잇는 연결(Link)로 관계를 표현한다.
- 카드에 종류(인물·장소·사건·테마)를 (선택적으로) 부여한다.
- 보드를 시리즈/작품에 1:N으로 매달고, **집필 중 보드를 참조**한다.
- 특정 카드 기준 연결된 카드(백링크 포함)를 본다.

### 비목표 (1차 제외)
- 실시간 공동 편집(단일 사용자 기준 설계). *§11 검토*
- 시간순/집필순 시퀀스·타임라인 뷰(후속).
- 인라인 `[[ ]]` 위키링크(후속).
- **같은 Card를 여러 보드에 동시 표시(N:N).** 1차는 Card가 한 보드에 전속.

## 4. 핵심 결정 (Confirmed)

1. **Card ↔ 시각 카드는 1:1.** Card는 정확히 한 보드에 속한다. 위치는 Card 행에 직접 저장(별도 배치 테이블 불필요).
2. **Board ↔ (시리즈 또는 작품)는 1:N.** 한 시리즈/작품이 보드를 여러 개 가질 수 있다.
3. **보드는 미소속(owner = null)도 허용한다 = "아이디어 보드".** 어느 작품이 될지 모르는 막연한 구상을 먼저 펼치고, 나중에 작품/시리즈에 붙일 수 있다(§5.3).
4. 좌표는 화면 픽셀이 아니라 **캔버스(flow) 좌표계**(음수·소수 허용).
5. 저장 타이밍은 **드래그 종료 / 연결 완료 / 종류 선택 시점**. 드래그 중 BE 미호출.
6. 도메인 전역은 `Board/Card/Link`. **React Flow의 `node/edge`는 어댑터 경계 안에서만** 등장(§8).
7. **보드 진입점은 세 곳**(전역 메뉴 / 작품·시리즈 내부 / 집필 참조 패널), 모두 동일한 보드 실체를 가리킨다(§5.4).

## 5. 보드–시리즈/작품 매핑 ★신규

### 왜 양쪽(시리즈·작품) 둘 다인가
보드가 담는 내용이 두 층위로 성격이 다르다.
- **시리즈 보드** — 전체를 관통하는 것: 세계관, 인물 장기 아크, 권을 넘는 떡밥, 시리즈 연표. *여러 작품에서 함께 참조.*
- **작품 보드** — 그 한 편의 구조: 이 편의 사건 순서, 회수할 복선, 장면 배치. *이 편에서만.*

둘을 한 보드에 욱여넣으면 시리즈가 길어질수록 "털뭉치"가 된다. 그래서 보드는 시리즈·작품 **어느 쪽에도 매달릴 수 있어야** 하고, 한 주체가 **여러 보드(1:N)** 를 갖는다(관점별로 단순하게 분리).

### 1:N의 사용성 이점
보드 1개만 만드는 작가는 전환 없이 단순하게 쓰고, 필요한 작가만 "인물 관계 보드 / 1부 사건 보드"처럼 늘린다. 단순한 사용자는 단순하게, 복잡한 사용자는 확장 가능.

### 5.3 미소속 보드 = "아이디어 보드" ★신규
보드는 시리즈/작품에 **반드시 매달릴 필요가 없다**. 작가의 실제 흐름은 *"아직 어느 작품이 될지 모르는 구상을 일단 펼쳐보고, 나중에 작품이 정해지면 붙인다"* 이다. 작품이 정해져야만 보드를 만들 수 있으면 이 초기 구상 단계를 막아버린다.

- `owner_type` / `owner_id`가 **둘 다 null**이면 미소속 = **아이디어 보드**.
- 제약: owner_type/owner_id는 **항상 함께 채워지거나 함께 null**(둘 중 하나만 채워지면 안 됨) — DB CHECK 또는 검증에 둔다.
- **"나중에 붙이기"는 필수 짝 기능**이다. 미소속 보드를 작품/시리즈에 연결하는 액션(`PATCH /boards/:id`로 owner 지정)이 없으면 미아 보드만 쌓인다. "먼저 끄적이고 나중에 정리"의 *정리* 부분.

### 5.4 보드 진입점 (세 곳) ★신규
세 진입점은 목적이 다르지만 **모두 같은 보드 실체**를 연다.

| 진입점 | 보여주는 것 | 의도 | 비고 |
|---|---|---|---|
| **상단 `보드` 메뉴** (전역 허브) | 내 모든 보드(시리즈·작품·아이디어 가로질러) | "그 보드 어디 있더라" — 찾기 | 소속 라벨 + 최근순 + 키워드 검색 |
| **작품/시리즈 내부 `보드` 탭** | 그 주체에 매달린 보드만 | 맥락 안에서 — 관리 | 생성 시 owner 자동 결정 |
| **집필 화면 참조 패널** | 그 작품 + 상위 시리즈 보드 | 곁눈질 — 읽기 | §9, 읽기 중심 |

**전역 목록 표시 규칙**
- **소속 라벨**: 각 보드 카드에 출처를 단다 — "○○시리즈", "○○작품", 미소속은 **"아이디어"**. (보드 이름만으로는 출처를 알 수 없으므로 필수)
- **정렬**: 기본 **최근 연 순**(전역 진입의 의도가 "위치 기억 안 남"이므로).
- **키워드 검색**: **작품명 / 시리즈명 / 보드명**을 OR로 매칭. *주의 — 아이디어 보드는 작품·시리즈명이 없으므로 보드명으로만 잡힌다. 소속 필드가 null이어도 검색에서 누락되지 않게 한다.*

**owner 결정의 차이 (마찰 설계)**
- **작품/시리즈 내부 생성** → owner 자동 결정, **이름만 입력**(권장 기본 동선, 마찰 적음).
- **전역 메뉴 생성** → owner를 **먼저 선택**(이 작품 / 시리즈 전체 / 아이디어).

## 6. 사용자 시나리오

1. 작가가 시리즈 또는 작품에서 보드를 연다 → 카드들이 캔버스에 펼쳐진다.
2. 카드를 드래그해 배치한다 → 손 떼는 순간 위치 저장.
3. 카드 A에서 카드 B로 선을 그어 잇는다 → 연결 저장.
4. 카드를 선택해 "이건 뭔가요?"로 종류를 정한다 → 카드에 색/아이콘 반영.
5. **집필 화면에서 보드를 참조한다** → 그 작품의 보드 + 상위 시리즈 보드를 오가며 원고와 나란히 본다(§9).
6. 다시 열어도 카드 위치·연결·마지막 뷰포트가 복원된다.

## 7. 데이터 모델 (제안)

> Card↔카드 1:1 → 위치를 Card 행에 직접 저장. Board는 `owner_type/owner_id`로 시리즈·작품 양쪽을 받는다.

### board
| 컬럼 | 타입 | 비고 |
|------|------|------|
| id | PK | |
| owner_type | enum nullable `'series' \| 'work' \| null` | 보드가 매달리는 주체. **null이면 아이디어 보드** |
| owner_id | FK nullable | series.id 또는 work.id. owner_type과 항상 함께 채워지거나 함께 null |
| name | text | 예: "인물 관계", "1부 사건 흐름" |
| viewport_zoom / viewport_x / viewport_y | double | 마지막으로 본 화면 |
| created_at / updated_at | timestamptz | |

> **제약**: `CHECK ((owner_type IS NULL AND owner_id IS NULL) OR (owner_type IS NOT NULL AND owner_id IS NOT NULL))` — owner는 짝으로만 존재.
| 컬럼 | 타입 | 비고 |
|------|------|------|
| id | PK | |
| board_id | FK → board.id | Card는 한 보드에 전속 |
| title | text | |
| content | text | 향후 `[[ ]]` 도입 시 구조화 포맷 검토 |
| card_type | enum nullable | `character \| place \| event \| theme`, 기본 null(무지정) |

| z_index | int default 0 | 겹침 순서 |

### link  (기존 `엣지`)
| 컬럼 | 타입 | 비고 |
|------|------|------|
| id | PK | |
| board_id | FK → board.id | |
| source_card_id | FK → card.id | |
| target_card_id | FK → card.id | |
| relation_type | text nullable | 후속: 원인·결과 / 복선·회수 |

## 8. FE 아키텍처 & React Flow 어댑터 경계

| 관심사 | 도구 | 비고 |
|---|---|---|
| 캔버스 | React Flow (`@xyflow/react`) | React 19 호환. `"use client"` + `dynamic(ssr:false)` |
| 드래그 중 transient | Zustand | 프레임 단위. **BE 미호출** |
| 서버 상태/영속화 | React Query | optimistic update + 롤백 |

**어댑터 경계 (유비쿼터스 언어 유지의 핵심)**
도메인 전역은 `Card/Link`로만 다루고, **React Flow에 넘기기 직전 얇은 어댑터 한 겹**에서만 라이브러리 형태로 변환한다.
```ts
// 도메인 → React Flow (어댑터 안에서만 node/edge라는 단어가 등장)
const toFlowNode = (card: Card) => ({ id: card.id, position: { x: card.pos_x, y: card.pos_y }, data: card });
const toFlowEdge = (link: Link) => ({ id: link.id, source: link.source_card_id, target: link.target_card_id, data: link });
```
이 파일 바깥에서는 `node`·`edge`라는 단어를 쓰지 않는다.

## 9. 집필 중 보드 참조 ★신규

"작업 중 보드 참고"의 알맹이는 스키마가 아니라 **동선**이다.

- 작품 집필 화면에서 참조 가능한 보드 = **그 작품의 보드 + 상위 시리즈의 보드** 모두. (이 편을 쓰며 시리즈 연표·인물 보드도 봐야 함)
- 보드가 여러 개면 **탭/드롭다운으로 전환**. 보드 1개면 선택 단계 없이 바로 그것(1:N이어도 N=1이면 1:1처럼 동작).
- **그 작품에서 마지막으로 본 보드를 기억**했다가 기본으로 연다(매번 고르는 마찰 제거).
- 집필 화면 한쪽에 **참조 패널/분할 뷰**로 보드를 띄워 원고와 나란히 본다.

> 주의: 집필 화면 에디터는 자체 EditContext 엔진(별도 화면). 보드는 React Flow 기반 별도 컴포넌트로, dynamic import로 격리해 집필 번들에 영향 주지 않는다.

## 10. API 설계 (제안)

| 메서드 | 경로 | 설명 |
|--------|------|------|
| GET | `/boards/mine` | **전역 허브**: 내 모든 보드(시리즈·작품·아이디어). 기본 최근순. `?q=` 키워드 검색(작품·시리즈·보드명 OR), `?owner_type=` 필터 |
| GET | `/boards?owner_type=&owner_id=` | 주체(시리즈/작품)에 속한 보드 목록 |
| GET | `/works/:id/reference-boards` | 집필 참조용: 그 작품 보드 + 상위 시리즈 보드 |
| POST | `/boards` | 보드 생성. `owner_type`/`owner_id`는 **선택**(생략 시 아이디어 보드) |
| PATCH | `/boards/:id` | 보드 메타 수정 + **owner 지정(아이디어 보드를 작품/시리즈에 붙이기)** |
| GET | `/boards/:id` | 보드 메타 + 뷰포트 + 카드 목록 + 연결 목록 (하이드레이션) |
| PATCH | `/boards/:id/cards` | **배치** 위치 업데이트 `[{ card_id, pos_x, pos_y, z_index? }]` |
| PATCH | `/boards/:id/viewport` | `{ zoom, x, y }` |
| POST | `/boards/:id/cards` | 카드 생성 + 초기 위치 |
| PATCH | `/cards/:id` | 제목/본문/종류(card_type) 수정 |
| DELETE | `/cards/:id` | 카드 삭제 (연결 cascade) |
| POST | `/boards/:id/links` | 연결 생성 |
| DELETE | `/links/:id` | 연결 삭제 |

## 11. 미결정 사항 (Open)

1. **단일 / 멀티 유저** — 1차 단일 사용자 가정. 동시 편집 시 충돌 처리 + 뷰포트를 사용자별 저장 필요.
2. **CardType 1차 노출 범위** — 인물/장소/사건/테마 전부 vs 일부.
3. **시퀀스/타임라인 뷰 우선순위** — 그래프는 "순서"를 못 보여줌. 어느 단계에 둘지.
4. **content 포맷** — 후속 `[[ ]]` 도입 시 text → 구조화 마이그레이션 전략.
5. **N:N 확장 시점** — "같은 Card를 여러 보드에서" 수요가 확인되면 위치를 배치 테이블로 분리.

> 해결됨: 보드 진입점(전역/내부/집필참조 세 곳, §5.4) · 미소속 보드 허용(아이디어 보드, §5.3).

## 12. 단계별 계획

- **Phase 1 (MVP)**: 보드 캔버스 + 카드 배치/영속화 + 수동 연결 + 백링크(기본) + 보드–시리즈/작품 매핑 + **아이디어 보드(미소속) & 나중에 붙이기** + **세 진입점(전역 허브·내부 탭·집필 참조)** + 전역 목록(소속 라벨·최근순·키워드 검색) + 집필 중 보드 참조(전환·마지막 본 보드 기억). 기존 메모·인물 마이그레이션.
- **Phase 2**: CardType 시각 구분 + Link `relation_type` + 인라인 `[[ ]]` + 필터링.
- **Phase 3**: 시퀀스/타임라인 뷰 + 로컬 그래프(N-hop) + 구조 템플릿.
