write-note · desktop scaffold

Electron 데스크톱 앱의 FE / BE 아키텍처 웹의 "클라이언트 ↔ 서버" 가 아니라, 한 프로세스 트리 안의 "Renderer ↔ Main" 입니다.

Electron 앱은 하나의 실행 파일 안에 Chromium(웹 화면) 과 Node.js(시스템 권한) 를 함께 담습니다. 그래서 "프론트엔드 / 백엔드" 의 경계가 네트워크 가 아니라 프로세스 로 그어지고, 그 사이를 preload + IPC 라는 좁은 다리가 잇습니다.

write-note desktop 의 실제 구조. 외부 서버 없이 한 대의 노트북 안에서 모두 돈다.

03 다리 — preload + contextBridge + IPC

FE 와 BE 역할이 직접 손잡으면 안 된다. 좁고 명시적인 창구만 연다.

preload Renderer 가 로드되기 직전 실행되는 특수 스크립트. 한 발은 Node, 한 발은 화면에 걸친 유일한 코드. → electron/preload.ts
contextBridge preload 가 화면 쪽 window.electronAPI 에 정해진 함수만 얹어주는 안전 통로. raw ipcRenderer 통째 노출은 금지.
IPC Inter-Process Communication. ipcRenderer.invoke('저장', data) → Main 의 ipcMain.handle('저장', …) 가 받아 처리 후 결과 반환.
현재 상태(Phase 1). 아직 로컬 저장 기능이 없어서 다리는 최소 골격만 있습니다. preload.ts 는 OS 이름(platform) 하나만 read-only 로 노출합니다. 실제 파일/DB 저장용 IPC 채널은 Phase 2 에서 추가됩니다.

04 데이터 한 바퀴 — "원고를 저장한다" (Phase 2 예상 흐름)

사용자가 글을 쓰고 멈추면 자동저장이 도는 가상의 시나리오. 색이 곧 어느 프로세스인지를 가리킨다.

1
Renderer · FE

사용자가 TipTap 에디터에 타이핑 → 700ms 멈춤 감지 → window.electronAPI.saveDoc(json) 호출

2
Preload · Bridge

contextBridge 가 노출한 saveDoc 이 내부에서 ipcRenderer.invoke('doc:save', json) 으로 변환

3
Main · BE 역할

ipcMain.handle('doc:save', …) 가 받아 Node 의 fs 또는 로컬 SQLite 에 기록 (전체 권한 사용)

4
Main · BE 역할

저장 성공/실패 결과를 return { ok: true, savedAt } 로 응답 — IPC 가 자동으로 되돌려보냄

5
Renderer · FE

await 가 결과를 받아 저장 상태를 'saved' 로 갱신 → Dock 에 "저장됨" 표시

지금 코드와의 차이. 현재 App.tsx 의 자동저장은 setTimeout 으로 상태만 바꾸는 시뮬레이션(2~5단계 없음)입니다. 실제 IPC 결선이 들어오면 위 흐름이 됩니다.

05 "그럼 진짜 서버(BE)는 어디?" — 두 가지 선택지

Electron 의 Main 은 BE "역할"을 하지만, 별도의 원격 서버를 둘 수도 있다. write-note desktop 은 ① 을 택했다.

① 로컬 전용 (현재 선택)

  • 데이터가 노트북 안에만 산다 — 파일/로컬 DB
  • Main process 가 곧 백엔드. 외부 서버·로그인·네트워크 불필요
  • 오프라인 동작, 빠름, 프라이버시 ↑
  • 단점: 기기 간 동기화·협업이 기본 제공 안 됨

② 원격 BE 병행 (선택지)

  • Renderer/Main 이 fetch 로 원격 API(Spring/REST) 호출
  • 웹 버전(frontend/backend)과 데이터 공유·동기화 가능
  • 클라우드 백업, 멀티 디바이스
  • 단점: 서버·인증·네트워크 의존, 오프라인 한계
한 앱에 둘 다 가능. 보통 로컬(Main)로 즉시 저장해 빠르게 반응하고, 뒤에서 원격 BE 로 동기화하는 로컬 우선(local-first) 하이브리드를 씁니다. write-note desktop Phase 1~2 범위는 ① 로컬 전용에 집중합니다 (docs/phase/01-desktop-scaffold/README.md "제외" 항목 참조).

08 네트워크가 끊기면? — 실행은 된다

자주 나오는 오해. Electron 앱은 브라우저처럼 매번 서버에서 페이지를 받지 않는다. 런타임이 전부 노트북 안에 번들된다.

결론. 인터넷이 끊겨도 앱은 정상으로 뜨고 모든 기능이 돈다. Chromium + Node.js + 앱 코드가 설치 시점에 모두 번들되고, 배포 빌드는 dist/index.html 을 file:// 로 로드하기 때문(main.ts). 게다가 이 프로젝트는 로컬 전용이라 원격 서버 호출이 0건 — 오프라인에서 기능적으로 완전하다.

오프라인에서 깨질 수 있는 건 "외부에 의존하는 부분"뿐이다. 현재 코드에선 폰트 하나가 유일하다.

외부 의존오프라인 시이 프로젝트
원격 BE API (fetch)기능 마비없음 — 로컬 전용, 호출 0건
CDN 폰트 / 스크립트폰트만 fallback있음 — index.html Google Fonts 3줄
자동 업데이트 체크업데이트만 스킵없음
실제로 보이는 차이. 지금 코드 그대로 오프라인 실행하면 — 앱·기능은 멀쩡하지만 Gowun Batang · Noto Sans KR 웹폰트를 못 받아 시스템 fallback 폰트(Apple SD Gothic Neo 등)로 글자가 표시된다. 의도한 명조체 집필 분위기가 살짝 달라지는 정도다.

dev 모드 (pnpm dev)

  • Main 이 Vite dev server localhost 를 loadURL 로 띄움 (main.ts:28)
  • localhost 는 인터넷이 아니라 OS 내부 루프백 → 인터넷 단절과 무관하게 동작

배포 빌드 (file://)

  • 빌드된 dist/index.html 을 loadFile (main.ts:30)
  • 네트워크를 아예 타지 않음 → 더더욱 무관
해결 방침은 이미 박혀 있음. 새 문제가 아니다. index.html:7 주석과 README §6 가 "프로덕션은 woff2 subset 을 로컬 번들" 로 명시한다. CDN → 로컬 폰트 전환을 마치면 오프라인에서 폰트까지 의도대로 나온다 — 외부 의존이 0 이 된다.

이 문서는 desktop/ 실제 코드(main.ts · preload.ts · App.tsx · vite.config.ts)를 읽어 작성했습니다. Phase 1 scaffold 기준이며, IPC 데이터 흐름(04)·로컬 저장(05)은 Phase 2 예상 설계입니다.
