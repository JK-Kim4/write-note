## 1. 배경 · 목표

작가가 작성한 작품을 외부 포맷으로 내보낸다. **핵심 요구(사용자 확정):**

> 작가가 집필 화면에서 **본 시각 형식 그대로** 결과물이 산출되어야 한다.

이때 "동일한 시각"의 보장 수준은 포맷의 구조적 본질에 따라 차등한다. PDF는 고정 레이아웃이라 화면을 픽셀 그대로 박을 수 있지만, hwpx·docx는 워드프로세서(한컴/Word)가 자기 폰트 메트릭으로 본문을 다시 흘리는 흐름 레이아웃이라 페이지 분할 위치를 픽셀 동일하게 재현하는 것은 기술적으로 불가능에 가깝다. 따라서 **PDF는 픽셀 그대로, 워드 문서는 형식 최대한 + 구조 보존**으로 목표를 분리한다.

### 사용자 결정 사항 (2026-06-14 브레인스토밍)

| 결정 | 내용 |
|---|---|
| "설정 규격"의 의미 | **용지·페이지 규격(형식)** — 용지 크기·페이지 분할·여백·폰트·서식. 등장인물/세계관/메모 등 작품 "설정 자료"는 **범위 밖** |
| 시각 충실도 | PDF = 화면 픽셀 그대로 / hwpx·docx = 형식 최대한 + 챕터·제목·문단 구조 보존 |
| 시각 장식(줄노트·종이색) | **export 시 설정값으로 작가가 선택**(줄노트 줄 포함 토글). 다크모드는 항상 라이트로 출력 |
| PDF 생성 방식 | **화면 충실 우선** — 브라우저 native 인쇄(저장 시 인쇄 대화상자 1회 노출 허용) |
| 구현 순서 | **hwpx 스파이크 → PDF → DOCX → HWPX** |

## 2. 범위

### 2-1. 포함 (In Scope)

- 본문 + **챕터 구조**(챕터 제목·순서) export
- **용지/페이지 형식 재현**: 용지 크기(A4/B4/A3/A2)·좌우 여백·명조체 폰트·페이지 분할·챕터 제목·본문 서식(H1~H3, bold 등)
- **Export 설정창**: 형식 선택 + 챕터 선택·순서 + 옵션(줄노트 토글)
- 3개 포맷: **PDF · HWPX · DOCX**

### 2-2. 비범위 (Out of Scope)

- 등장인물·세계관·곁쪽지(메모) 등 작품 **설정 자료**의 export (= "설정 규격은 형식을 의미"로 확정)
- 구 `.hwp` 바이너리 export (개방표준 `.hwpx`로 대체)
- 표지·목차 자동 생성 (후속 후보)
- 다른 작품의 챕터 가져오기/작품 횡단 합본 (챕터 설계에서 이미 비범위)
- 다크모드 결과물 (항상 라이트 출력)
- 운영(Supabase/Render) 배포 — 기존 전략대로 Round 4. 단 **hwpx는 백엔드 의존**이므로 운영 작동은 Round 4 이후(아래 §7 제약)

## 3. Export 설정창 (UI)

**진입점:** 집필실 좌측 챕터 목록 아래 **"내보내기"** 버튼. (A형 `/projects/[id]/write` · B형 `/b/works/[id]` 양쪽 — plan 단계에서 두 집필실 구조 확인)

**구성 (브레인스토밍에서 목업 승인):**

```
내보내기 — 「작품명」
─────────────────────────────
① 형식      [ PDF ]  [ HWPX ]  [ DOCX ]
② 챕터      ☑ 제1장 · 출발        3,200자  ⌃⌄
            ☑ 제2장 · 낯선 도시    5,800자  ⌃⌄
            ☐ 제3장 · 초고(미완)     900자  ⌃⌄
③ 옵션      줄노트 줄 포함         [토글]
            용지                  A4 (작품 설정)
─────────────────────────────
            [ 형식으로 내보내기 ]
```

- **① 형식** — PDF/HWPX/DOCX 택1. PDF에 "화면 그대로", HWPX "한컴", DOCX "Word" 보조 라벨.
- **② 챕터** — 활성(미삭제) 챕터 목록. **기본 전체 선택**. 체크로 일부 제외, ⌃⌄로 export 순서 조정. 글자수 표시.
  - 순서 기본값 = 집필실 챕터 순서(`sort_order`). 이 창의 순서 조정은 **이번 export 한정**(원본 챕터 순서 불변).
  - 최소 1개 선택 강제(0개 선택 시 내보내기 비활성).
- **③ 옵션** — 줄노트 토글(기본 OFF 권장 — 인쇄 실용성), 용지는 작품 속성(`projects.paper_size`) 표시만(읽기전용).

## 4. 충실도 정책 (포맷별)

| 요소 | PDF | DOCX | HWPX |
|---|---|---|---|
| 용지 크기(A4/B4/A3/A2) | ✅ 동일 | ✅ 문서 속성 지정 | ✅ 문서 속성 지정 |
| 좌우 여백 | ✅ 동일 | ✅ 지정 | ✅ 지정 |
| 명조체 폰트 | ✅ 동일 | ✅ 지정(폰트 미설치 시 대체) | ✅ 지정 |
| **페이지 분할 위치** | ✅ **화면과 동일** | ⚠️ Word가 재계산 | ⚠️ 한컴이 재계산 |
| 챕터 제목 | ✅ | ✅ 제목 스타일 | ✅ 제목 스타일 |
| 본문 서식(H1~H3·bold) | ✅ | ✅ | ✅ |
| 줄노트 줄(옵션 ON) | ✅ | ✖️(워드 본질상 미지원) | ✖️ |
| 종이 배경색 | ✖️(흰 종이 인쇄 기본) | ✖️ | ✖️ |

> 줄노트·배경색 같은 화면 장식은 PDF에서만 의미가 있다. 워드 문서는 "편집 가능한 원고"가 본질이라 장식보다 **내용·구조 보존**이 목표.

## 5. 기술 아키텍처

### 5-1. PDF — 인쇄 전용 합본 뷰 + native 인쇄

- 선택한 챕터들을 **순서대로 이어붙인 인쇄 전용 합본 뷰**를 렌더. 집필실의 `PaperEditor` 읽기전용 렌더링(같은 페이지 분할 CSS)을 재사용해 화면과 동일한 시각을 보장한다.
- `@page` 인쇄 스타일시트 + 기존 페이지 분할 CSS(`column-wrap`/`pageLayout.ts` 기하)로 인쇄 시 분할을 화면과 일치시킨다.
- 브라우저 native 인쇄(`window.print()`) → 사용자가 "PDF로 저장". 라이브러리 0.
- 줄노트 토글: 인쇄 뷰의 `.sheet--lined` 클래스 on/off로 반영.
- **위치: 프론트(Next.js)만. 백엔드 의존 0.**

### 5-2. DOCX — JS 라이브러리 변환

- JS `docx` 라이브러리로 챕터 본문(TipTap 본문 구조) → docx 변환. 챕터 제목 = Heading 스타일, 문단/H1~H3/bold 매핑.
- 용지·여백은 docx section 속성으로 지정.
- **위치: 프론트만. 백엔드 의존 0.**

### 5-3. HWPX — 백엔드 JVM

- 백엔드(Kotlin)에서 `hwpxlib`(한컴 개방표준 OWPML) 로 hwpx 생성. 본문 구조 → OWPML 매핑.
- 신규 endpoint(예: `POST /api/projects/{projectId}/export/hwpx`, body = 선택 챕터 id·순서·옵션) → hwpx 바이너리 응답.
- **위치: 백엔드. JVM 라이브러리라 프론트 불가.**

### 5-4. 데이터 — 챕터 본문 합치기

- export는 **선택 챕터들의 본문(body)**을 순서대로 모아야 한다.
- 현재 챕터 목록 API(`GET /api/projects/{projectId}/documents`)는 전송량 절약을 위해 **본문(body) 제외**.
- **plan 단계 검증 필수(agent-workflow-discipline §9):** 선택 챕터들의 본문을 가져오는 데이터 경로가 기존에 있는지 확인. 없으면 다음 중 택1을 plan에서 결정:
  - (a) 챕터 단건 본문 조회를 챕터 수만큼 호출(집필실 챕터 전환이 이미 쓰는 경로 재사용 가능성)
  - (b) export 전용 합본 조회 endpoint 신설(`GET .../documents?include=body` 또는 export payload 조회)

## 6. 구현 순서

| 순서 | 작업 | 산출/게이트 |
|---|---|---|
| 1 | **hwpx 스파이크(#42)** | `hwpxlib`로 최소 hwpx 1개 생성(한컴오피스에서 열림 확인) + 본문→OWPML 매핑 가능 범위 문서화. **결과로 hwpx 추정·범위 확정.** 불가/과도 시 hwpx 범위 재논의(예: 후속 이관·docx 우선) |
| 2 | **PDF(#51)** | 인쇄 합본 뷰 + @page CSS + 설정창. 핵심 요구 "화면 그대로" 첫 dogfoodable 산출물 |
| 3 | **DOCX(#43)** | `docx` 변환 + 설정창 형식 추가 |
| 4 | **HWPX(#44)** | 스파이크 결과 기반 백엔드 본구현 + 설정창 형식 추가 |

> §10(양보불가 핵심 우선) 정합: hwpx 스파이크는 코드량이 작은 리스크 검증이라 PDF(핵심)를 크게 지연시키지 않으며, 스파이크 직후 PDF로 핵심을 가장 먼저 증명한다.

## 7. 리스크 / 제약

- **hwpx 실현가능성 미확정** — 스파이크 전 미확정(최대 불확실). 액션: 순서 1번으로 검증, 결과로 추정 갱신. 불가 시 hwpx 범위 재논의.
- **챕터 본문 합치는 데이터 경로** — §5-4. plan 단계에서 기존 조회 경로 grep 후 결정(추측 금지).
- **백엔드 운영 미배포** — web 백엔드가 Render에 아직 미배포(Round 4). **hwpx는 로컬 dogfooding만 가능**, 운영 작동은 Round 4 배포 후. **PDF·DOCX는 프론트라 Vercel만으로 즉시 작동.**
- **PDF 인쇄창 UX** — native 인쇄라 저장 시 브라우저 인쇄 대화상자 1회 노출. "화면 100% 동일"의 대가(사용자 수용).
- **명조체 폰트 임베딩** — docx·hwpx에서 명조체(Nanum Myeongjo) 미설치 환경 대체. 폰트 임베딩 가능 여부는 라이브러리별 plan 확인.
- **두 집필실(A형·B형) 진입점** — 양쪽 모두 "내보내기" 진입 필요. plan에서 두 구조 확인 후 공통/분리 결정.
