## 2. 어떻게 했는가 (접근)

- QA는 자동화 검증과 실제 Electron 창 기반 Computer Use 확인을 함께 사용했다.
- 디자인 평가는 `impeccable`의 critique 흐름을 사용했고, 사용자의 우선순위 답변을 기준으로 brainstorming visual companion 목업으로 전환했다.
- A/B 방향을 먼저 보여준 뒤, 사용자가 선호한 A를 단일 목업으로 확장했다.
- 마지막에는 최신 사용자 피드백인 "더 작업실 답게가 필요"를 다음 세션의 최우선 설계 조건으로 고정했다.

## 3. 잘 된 점

1. QA와 디자인 개선 대상을 분리했다. QA에서 발견한 메모 연결 UI 반영 지연은 수정 완료 상태로 정리했고, 디자인 고도화는 별도 흐름으로 남겼다.
2. 사용자가 선호한 A 방향을 파일 산출물로 남겼다. 다음 세션이 말뿐인 요약이 아니라 실제 목업 파일을 보고 이어갈 수 있다.
3. 핸드오프 문서에 "다음 세션에서 구현하지 말고 목업부터"라는 gate를 명시했다. brainstorming 흐름이 끊기지 않도록 했다.

## 4. 어긋난 점

- 사용자가 "뭘 확인하라고?"라고 물은 시점이 있었다. 선택해야 하는 것이 기술 검증인지 디자인 방향인지 충분히 명확히 말하지 못했다.
- A 기반 목업은 사용자의 최신 피드백 기준으로 아직 실용 UI 느낌이 남았다. `오늘 붙일 메모`, 큐, 메모 처리 같은 표현이 작업실보다 관리 화면 쪽으로 읽힐 수 있다.
- visual companion 클릭 이벤트 파일은 없었고, 터미널 답변을 기준으로 판단했다. 이 점은 문제는 아니지만, 다음 세션에서는 사용자가 실제 화면에서 선택했는지 터미널로 답했는지 구분해 기록하는 편이 낫다.
- `.superpowers/`와 `.impeccable/` 산출물이 미추적 상태로 남아 있다. 다음 세션에서 추적 여부를 결정해야 한다.

## 5. 다음 작업에 남길 교훈

### 5-1. 작업별 교훈 (이 프로젝트 한정)

- "작업실답게"는 배경 질감보다 조작 밀도와 정보 언어에서 결정된다. 큐, 통계, 처리 상태를 먼저 꺼내면 작업실이 아니라 관리 도구가 된다.
- 재진입 강화의 첫 정보는 날짜가 아니라 마지막 문장, 다음 장면, 오늘 붙일 쪽지다.
- 사용자가 목업 피드백을 줄 때는 "무엇을 확인하면 되는지"를 한 문장으로 명확히 써야 한다.

### 5-2. 룰 갱신 후보 (사용자 컨펌 필요)

- 갱신 대상 파일: `AGENTS.md` 또는 프로젝트 디자인 규칙 문서
- 후보 룰: desktop app 디자인 목업에서 "재진입 강화"를 다룰 때는 관리형 단어(`queue`, `통계`, `처리`)보다 작업 재료형 단어(`쪽지`, `마지막 문장`, `다음 장면`)를 우선한다.
- 근거 회귀 사례: A 방향 선호 후 만든 목업이 여전히 실용 UI 쪽으로 치우쳐, 사용자가 "더 작업실 답게가 필요"라고 정정했다.

사용자 컨펌 전까지 위 룰 후보는 실제 규칙 파일에 반영하지 않는다.
