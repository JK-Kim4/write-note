## 1. 배경 (Background)

서비스는 집필 결과를 **시리즈**와 **작품** 단위로 관리한다. 시리즈는 여러 작품을 포함하고, 작품은 **독립 존재** 또는 **시리즈 소속**으로 존재한다.

공유 기능은 작가가 **공유 링크를 생성해 외부인에게 전달**하고, 외부인이 **로그인 없이** 작품/시리즈를 열람하게 한다. 공유받은 사람이 **회원**이면 본문 특정 위치에 **댓글**을 남겨 피드백할 수 있다.

목적: 작가가 글을 신뢰하는 독자·동료에게 보여주고 **위치 기반 피드백을 받는** 순환을 만든다.

## 2. 구현 컨텍스트 — 코드베이스 검증 (2026-06-28)

> 본 절은 PRD 가 전제한 사실들을 **실제 코드에서 검증**한 결과다(추측 아님). 충돌 시 코드가 SoT.

### 2.1 용어 ↔ 실제 코드 매핑

PRD 가 쓰는 도메인 용어와 코드 식별자가 다르다. 설계·구현 시 코드 식별자를 쓴다.

| PRD 용어 | 실제 코드 | 근거 |
|---|---|---|
| work (작품) | `Project` / `projects` 테이블 (PK `Long`) | `Project.kt` — `userId`·`categoryId` FK |
| series (시리즈) | `Category` / `categories` 테이블 (PK `Long`) | `Category.kt`, UI 노출명 = "시리즈" |
| 본문 | `documents.body` (ProseMirror/PM JSON, **암호화 저장**) | `Document.kt`, `DocumentService.kt` |
| 작품-본문 관계 | **1:1** (033 에서 챕터 제거) | `DocumentController` = `GET /api/projects/{id}/document` 단수, 챕터 endpoint 부재 |
| user | `users` (PK `Long`) | — |
| 댓글의 `work_id` | `project_id FK → projects.id` | — |

**유리한 사실:** 작품-본문 1:1 → 댓글 앵커가 "어느 챕터"를 추적할 필요 없이 작품의 단일 본문(스냅샷)만 가리키면 된다.

### 2.2 검증된 구현 기반 (재사용 가능)

| 기반 | 상태 | 위치 |
|---|---|---|
| 읽기 전용 본문 정적 렌더 | **있음** (재사용) | `printLayout.tsx` `relayout(model)` + `renderRuns` (PDF/인쇄용) |
| 추측 불가 랜덤 토큰(slug) 생성 | **있음** (재사용) | `ApiTokenHasher` — `SecureRandom` base62 32자 |

| Flyway 마이그레이션 | 신규 = **V27** | `backend/src/main/resources/db/migration/` (현재 V26) |

### 2.3 구현 전 해소해야 할 장애물

1. **[해소 — 스냅샷으로 우회] 블록 ID 앵커링.** PRD 의 핵심 전제(안정적 블록 ID)가 살아있는 에디터에는 **부재**했다(블록 = `\n` 경계 + 배열 인덱스). POC 로 주입 가능(침습도 중)함을 확정했으나, **공유본 = 스냅샷(§3-6)** 결정으로 *불변 스냅샷의 블록 인덱스* 가 곧 안정 앵커가 되어 **살아있는 에디터를 건드릴 필요가 없어졌다**. 블록 ID 주입(POC 옵션 A)은 미래의 "원문 수정 실시간 반영" 기능용으로 보류.
2. **[신규 구성] 선택적 인증(optional auth).** 공유 페이지 = 비로그인 열람 + 로그인 회원만 댓글. 현재 `SecurityConfig` 는 permitAll/authenticated **이분법만** 있고 optional-auth 패턴이 없다. `/api/shared/**` 를 permitAll 로 열되 토큰이 있으면 principal 을 채우는 로직을 새로 구성해야 한다.
3. **[확정] 스냅샷의 암호화 처리.** 스냅샷도 본문과 동일하게 **owner 키로 암호화 저장**하고, 비로그인 공개 read 시 **서버측에서 링크 소유자(`owner_id`) 키로 복호화**해 평문을 내려준다(at-rest 평문 0 유지 — 034 본문암호화 posture 일관, 2026-06-28 결정). 서버는 KEK 보유 → owner DEK 복호 가능. `BodyCipherService.encrypt/decryptToPlain` 재사용.

## 3. Phase 1 핵심 결정 (Confirmed)

1. **접근 통제 = 링크 공개.** 링크를 가진 사람은 누구나 열람(비로그인 포함). 비밀번호·만료 없음.
2. **공유 단위 = 작품 또는 시리즈.** 시리즈 공유 시 **작가가 공개할 작품을 직접 선택**(전체 자동 공개 아님).
3. **공유 페이지는 읽기 전용.** 외부인이 보는 동안 본문은 고정되며, 작가의 원문 편집과 분리된다.
4. **피드백 = 레벨 1 "위치 지정 댓글".** 회원이 본문의 특정 문단/구간에 댓글을 단다. 읽기 전용·스냅샷이라 앵커링이 안정적이다.
5. **링크 끄기(revoke) 포함.** 유출 시 차단 수단으로 1차에 포함한다.
6. **공유본 = 공유 시점 스냅샷 (옵션 B 확정, 2026-06-28).** 공유본은 **공유(작품 공개) 시점의 본문을 동결한 사본**이다. 작가가 이후 원문을 고쳐도 **공유본에는 반영되지 않는다.** 갱신하려면 새로 공유(끄기 후 재생성 = 새 스냅샷)한다. → 살아있는 에디터 무변경 + 댓글 앵커 영구 안정 + orphan 처리 불필요.

## 4. Phase 1 범위

### 들어가는 것 (In scope)
- 작가가 **작품/시리즈 공유 링크 생성·끄기**.
- 시리즈 공유 시 **공개 작품 선택**(선택 시점에 각 작품 본문 스냅샷 동결).
- 비로그인 외부인의 **읽기 전용 열람**(스냅샷 렌더, 시리즈는 공개 작품을 개별 단위로).
- 회원의 **위치 지정 댓글** 작성·조회·삭제(본인 댓글).
- 작가가 자기 글의 댓글을 **모아 보고 해당 위치로 이동**.

### 빠지는 것 (Out of scope — 이번엔 안 함)
- 비밀번호 / 만료 기한 / 열람자 화이트리스트.
- **원문 수정의 공유본 실시간 반영**(= 블록 ID 라이브 앵커링, POC 옵션 A — 미래 기능).
- 공유본 "스냅샷 갱신" 전용 액션(현재는 끄기 후 재생성으로 충족).
- 레벨 2 수정 제안(첨삭): 원문 변경 제안·수락/거절·머지.
- 대댓글 / 해결(resolve) 표시 / 작가 댓글 알림.
- 열람 통계(조회수 등), 실시간 협업 편집.

## 5. 접근 통제

- 공유 링크는 **추측 불가능한 랜덤 토큰**(긴 slug)을 쓴다. URL에 순번 ID를 노출하지 않는다(열거 공격 방지). → `ApiTokenHasher` 패턴 재사용.
- 링크를 아는 사람은 누구나 열람한다.
- 공유 페이지에 **`noindex`** 를 두어 검색엔진 색인을 막는다(원치 않는 노출 방지).
- 작가는 링크를 **끄기(revoke)** 할 수 있다. 꺼진 링크로 접근하면 "더 이상 볼 수 없는 링크" 안내를 보여준다.
  - 재발급(새 토큰 생성)은 "끄기 후 새 링크 생성"으로 충족하며, 이때 **새 스냅샷**이 동결된다.

## 6. 공유 단위 & 가시성

### 6.1 작품 공유
- 단일 작품 공유 링크. 독립 작품·시리즈 소속 작품 모두 가능.
- 시리즈 소속 작품도 **단독으로 따로 공유 가능**(시리즈 공유와 독립적으로 동작).
- 링크 생성 시점에 그 작품 본문을 **스냅샷 동결**한다.

### 6.2 시리즈 공유
- 공유 페이지는 **공개로 선택된 작품 목록**을 보여주고, 각 작품을 **개별 단위로 열람**한다.
- **작가가 공개 작품을 직접 선택**한다(예: 10화 중 1~3화만 공개). **선택(공개 목록에 추가) 시점에 그 작품 본문을 스냅샷 동결**한다.
- 시리즈에 **새 작품이 추가돼도 자동 포함하지 않는다.** 작가가 공개 목록에 명시적으로 추가해야 노출되며, 추가하는 그 순간의 본문이 스냅샷된다(미완성분 의도치 않은 노출 방지).

### 6.3 읽기 전용 동작 (스냅샷)
- 공유 페이지 본문은 **공유 시점 스냅샷을 읽기 전용으로 렌더**한다(작가의 살아있는 원문이 아니라 동결 사본). → `printLayout.relayout` 정적 렌더로 스냅샷 PM JSON 을 렌더.
- 스냅샷은 불변이므로 **열람 중에도, 작가의 이후 편집에도 변하지 않는다.**
- 스냅샷이 동결돼 있으므로 **댓글 앵커는 항상 유효**하다(구조 변경으로 인한 앵커 소실이 발생하지 않음 → §7.2).

## 7. 피드백 — 레벨 1 위치 지정 댓글

### 7.1 동작
- **회원**인 열람자가 본문의 특정 **문단/구간을 선택**해 그 위치에 댓글을 남긴다.
- **비회원은 읽기만** 가능(댓글 불가). 댓글에는 작성자 신원이 남아야 하므로 로그인 회원으로 한정.
- 작가는 자기 글의 댓글을 모아 보고, 클릭하면 본문(스냅샷)의 해당 위치로 이동한다.
- 작성자는 **본인 댓글을 삭제**할 수 있다(수정·대댓글·resolve는 이번 범위 밖).

### 7.2 앵커링 — 스냅샷 블록 위치 (단순화됨)

> 공유본이 불변 스냅샷이므로, v0.3 의 "살아있는 원문에 안정적 블록 ID 주입"(POC 옵션 A)이 **불필요**해졌다. 댓글은 동결된 스냅샷 안의 블록을 가리키기만 하면 된다.

- 댓글 위치는 **스냅샷 본문의 블록 인덱스**(top-level 블록 배열의 위치)로 저장한다. 스냅샷이 절대 변하지 않으므로 인덱스는 **영구적으로 안정**하다(글자 오프셋 방식의 취약성·블록 ID 주입의 침습성 둘 다 회피).
- `anchor_meta` 에 블록의 텍스트·타입 일부를 함께 저장해 표시(미리보기)와 검증에 쓴다.
- **orphan 불필요:** 스냅샷이 불변이라 앵커가 깨지지 않는다 → v0.3 의 `is_orphaned`·content fingerprint 백스톱은 Phase 1 에서 제거.
- 댓글은 그 댓글이 작성된 **스냅샷에 귀속**된다(자신이 본 불변 본문에 항상 정합). 새 공유(새 스냅샷)는 댓글 없이 시작한다.

### 7.3 신원 처리
- 공유 페이지 접근 시 로그인 여부를 판별한다(§2.3-2 선택적 인증). 로그인 회원이면 댓글 입력 UI를 노출, 비회원이면 숨긴다(읽기 전용).

## 9. API 설계 (Phase 1)

> 모든 경로에 `/api` prefix. 공개 엔드포인트(`/api/shared/...`)는 토큰 유효성·`is_active` 를 먼저 검증한다. 댓글 작성은 토큰 유효 + 로그인 회원을 동시에 만족해야 한다(§2.3-2 선택적 인증).

| 메서드 | 경로 | 인증 | 설명 |
|--------|------|------|------|
| POST | `/api/share-links` | 작가 | 공유 링크 생성(`target_type`, `target_id`). 작품 공유면 즉시 스냅샷 동결 |
| PATCH | `/api/share-links/:id` | 작가 | 끄기(`is_active=false`) |
| PUT | `/api/share-links/:id/works` | 작가 | 시리즈 공유의 공개 작품 목록 설정. **추가된 작품을 그 시점 본문으로 스냅샷 동결**, 제거된 작품 스냅샷 삭제 |
| GET | `/api/share-links/mine` | 작가 | 내 공유 링크 목록/상태 |
| GET | `/api/shared/:token` | 공개 | 공유 페이지 데이터(대상 메타 + 시리즈면 공개 작품 목록). 꺼진 링크면 비활성 응답 |
| GET | `/api/shared/:token/works/:projectId` | 공개 | 공유된 작품의 **스냅샷 본문**(읽기 전용) + 댓글 목록. 스냅샷 암호화 시 서버측 owner 키 복호화 |
| POST | `/api/shared/:token/works/:projectId/comments` | 회원 | 위치 지정 댓글 작성(`anchor_block_index`, `content`) |
| DELETE | `/api/comments/:id` | 작성자 | 본인 댓글 삭제 |

## 10. 미결정 사항 (Open)

1. **[해결됨] 스냅샷 정책 → 옵션 B 확정 (2026-06-28).** 공유본 = 공유 시점 스냅샷, 작가 원문 수정 미반영(§3-6). 블록 ID 라이브 앵커링(POC 옵션 A)은 미래 기능으로 보류.
2. **[해결됨] 스냅샷 암호화 처리 → 암호화 저장 확정 (2026-06-28).** `body_snapshot` 을 owner 키로 암호화 저장하고 공개 read 시 서버측 복호(§2.3-3). at-rest 평문 0 유지.
3. **댓글 권한 범위** — 링크를 아는 회원 누구나 댓글 가능(현재 가정) vs 작가가 댓글 권한을 통제.
4. **구간 선택 단위** — 블록(문단) 전체 단위만 허용할지, 블록 내 텍스트 범위(`anchor_meta`)까지 지원할지. 1차는 **블록 단위**로 시작 권장.
