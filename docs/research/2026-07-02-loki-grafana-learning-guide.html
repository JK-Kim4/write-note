Grafana Loki + Grafana 로그 모니터링 학습 가이드

작성일 2026-07-02 · 대상: Loki/Grafana를 처음 사용하는 개발자 · 목적: 소설비(write-note) 백엔드 로그 모니터링 설계에 앞선 기술 학습

핵심 요약

  • Loki는 "로그 본문을 색인하지 않는" 로그 저장소다. 라벨(메타데이터)만 색인하기 때문에 Elasticsearch보다 훨씬 가볍고, 대신 본문 검색은 그때그때 훑는(grep) 방식이다.
  • 스택은 3개 구성요소로 이뤄진다: Alloy(수집) · Loki(저장·색인) · Grafana(조회·시각화·알림).
  • 로그는 LogQL이라는 질의 언어로 검색한다. Prometheus의 PromQL과 닮았다.
  • 가장 중요한 설계 원칙 하나만 기억하면 된다: 라벨 종류(카디널리티)를 적게 유지하라. 이걸 어기면 Loki의 장점이 전부 사라진다.
  • 기존 수집기 Promtail은 2026-03-02부로 지원 종료(EOL). 신규 구축은 무조건 Grafana Alloy를 쓴다.

1. 큰 그림 — 세 구성요소의 역할

로그 모니터링 스택은 어떤 제품을 쓰든 세 단계로 이뤄진다: 수집(collect) · 저장(store) · 조회(query). Grafana 진영에서 이 세 역할을 맡는 것이 각각 Alloy, Loki, Grafana다.

구성요소역할비유
Grafana Alloy서버·컨테이너에서 로그를 긁어모아 Loki로 보내는 수집기(agent). Docker 소켓을 통해 컨테이너 stdout 로그를 자동 발견·수집할 수 있다.우편배달부
Loki받은 로그를 압축 저장하고, 질의가 오면 찾아주는 로그 저장소(서버). Go로 짜인 단일 바이너리라 가볍다.도서관 서고
GrafanaLoki에 질의를 날려 결과를 보여주는 웹 UI. 로그 탐색(Explore), 대시보드, 알림(Alerting)을 담당한다.열람실 + 사서

세 구성요소는 전부 독립 프로세스(각각 Docker 컨테이너)로 돌며, 서로 HTTP로만 통신한다. 하나가 죽어도 나머지는 영향받지 않는다 — 예를 들어 Grafana가 내려가도 로그 수집·저장은 계속된다.

2. Elasticsearch(ELK)와 무엇이 다른가

앞선 조사에서 ELK 스택과 비교했으니, 근본 차이를 정확히 짚고 넘어가자. 핵심은 "무엇을 색인(index)하느냐"다.

  • Elasticsearch는 로그 본문의 모든 단어를 역색인(inverted index)한다. 어떤 단어든 즉시 찾지만, 색인 자체가 원본보다 커질 수 있고 그걸 만들고 유지하는 데 CPU와 메모리(JVM 힙)를 크게 쓴다.
  • Loki는 본문을 색인하지 않는다. 로그에 붙은 라벨(예: 어느 컨테이너에서 왔나)만 색인하고, 본문은 압축해서 통째로 저장한다. 검색할 때는 라벨로 범위를 좁힌 뒤 그 범위의 본문을 훑는다(분산 grep).
Elasticsearch (ELK)Loki
색인 대상본문 전체 (역색인)라벨만 (본문은 압축 저장)
본문 검색 방식색인 조회 — 항상 빠름범위를 좁힌 뒤 스캔 — 소규모에선 충분히 빠름
상주 자원무거움 (JVM, 최소 힙 1GB+)가벼움 (Go 단일 바이너리, 수백 MB)
저장 비용색인 때문에 원본의 수 배압축으로 원본보다 작음
질의 언어Query DSL / KQLLogQL (PromQL 유사)
적합한 규모대량 로그 + 복잡한 전문 검색중소 규모 + 라벨 기반 탐색 (우리 케이스)

트래픽이 적고 "백엔드에서 방금 무슨 에러가 났지?"가 주 용도인 우리 상황에서는, 전문 검색 색인의 비용을 낼 이유가 없다. 이것이 Loki를 선택한 근거다.

3. Loki 핵심 개념 — 라벨, 스트림, 청크

3-1. 로그 한 줄의 구조

Loki에 저장되는 로그 한 줄(entry)은 세 부분으로 이뤄진다.

구성예시색인 여부
라벨 (labels) {container="wn-be-green", level="ERROR"} 색인됨 — 검색의 진입점
타임스탬프 2026-07-02T12:31:05.123Z 정렬 기준
본문 (log line) NullPointerException at ShareService.kt:42 ... 색인 안 됨 — 압축돼 청크에 저장

3-2. 스트림(stream) — 라벨 조합이 곧 파일 하나

라벨 키·값의 고유한 조합 하나가 스트림 하나를 만든다. 같은 라벨 조합으로 들어온 로그는 같은 스트림에 시간순으로 쌓이고, 일정량이 모이면 청크(chunk)라는 압축 블록으로 굳어져 디스크에 쓰인다.

가장 중요한 규칙 — 라벨 카디널리티(cardinality)를 낮게 유지하라

라벨 값의 종류가 늘어나는 만큼 스트림 수가 곱셈으로 늘어난다. 예를 들어 user_id를 라벨로 쓰면 사용자 수만큼 스트림이 생기고, 각 스트림이 메모리 버퍼와 자잘한 청크를 만들어 Loki가 급격히 무거워진다. 라벨로 좋은 것: container, app, env, level 처럼 값이 몇 개 안 되는 것. 라벨로 쓰면 안 되는 것: user_id, trace_id, request_path, IP 주소 등 값이 무한히 늘어나는 것 — 이런 값은 본문에 남겨두고 질의 시점에 필터·파서로 걸러낸다.

3-3. 저장과 보존 (retention)

  • 저장 위치: 단일 서버에서는 로컬 디스크(filesystem)에 저장한다. 규모가 커지면 S3 같은 오브젝트 스토리지로 바꿀 수 있다 — 설정만 다르고 개념은 같다.
  • 색인 방식: 최신 스키마는 tsdb(v13)다. 설정에 스키마 버전을 명시하게 되어 있다.
  • 보존 기간: compactor(청크 정리 담당 내부 구성요소)가 오래된 로그를 지운다. 설정 예시:
# loki-config.yaml 중 보존 관련 부분
compactor:
  working_directory: /loki/retention
  delete_request_store: filesystem   # 삭제 요청 기록 위치 (retention 켤 때 필수)
  retention_enabled: true

limits_config:
  retention_period: 30d              # 30일 지난 로그는 삭제

3-4. 배포 모드 — 우리는 "모놀리식" 하나만 알면 된다

Loki 문서를 보면 배포 모드가 3가지 나온다: 모놀리식(monolithic, 단일 바이너리), 단순 확장형(simple scalable), 마이크로서비스. 이건 같은 코드를 몇 개의 프로세스로 쪼개 돌리느냐의 차이일 뿐이다. 단일 서버·소규모에서는 모놀리식 모드(프로세스 1개, -target=all)가 공식 권장이고, 우리도 이걸 쓴다. 문서에서 distributor, ingester, querier 같은 용어를 만나면 "모놀리식 안에 다 들어있는 내부 부품 이름"이라고 이해하면 된다.

4. 수집기 — Grafana Alloy

Promtail은 끝났다 (EOL 2026-03-02)

인터넷 자료 대부분이 수집기로 Promtail을 소개하지만, Promtail은 2026년 3월 2일부로 지원 종료(EOL)됐고 코드가 Grafana Alloy에 흡수됐다(Loki 공식 문서 명시). 튜토리얼·블로그를 볼 때 Promtail 기반 글이면 오래된 자료라고 판단하면 된다. 신규 구축은 Alloy 하나만 보면 된다.

Grafana Alloy는 Grafana의 통합 수집기다. 로그뿐 아니라 메트릭·트레이스도 수집할 수 있는 범용 에이전트(OpenTelemetry 호환)인데, 우리는 로그 수집 기능만 쓴다.

4-1. 동작 원리 — 부품(component)을 파이프라인으로 잇는다

Alloy 설정 파일(config.alloy)은 "부품 선언 + 부품끼리 연결"로 읽는다. Docker 컨테이너 로그를 수집하는 실제 공식 예시가 이렇다:

// 1) Docker 소켓으로 떠 있는 컨테이너를 자동 발견
discovery.docker "backend" {
  host             = "unix:///var/run/docker.sock"
  refresh_interval = "5s"
}

// 2) 컨테이너 이름을 container 라벨로 변환
discovery.relabel "backend" {
  targets = []
  rule {
    source_labels = ["__meta_docker_container_name"]
    regex         = "/(.*)"
    target_label  = "container"
  }
}

// 3) 발견한 컨테이너들의 stdout/stderr 로그를 읽는다
loki.source.docker "backend" {
  host          = "unix:///var/run/docker.sock"
  targets       = discovery.docker.backend.targets
  relabel_rules = discovery.relabel.backend.rules
  forward_to    = [loki.write.default.receiver]
}

// 4) Loki로 전송
loki.write "default" {
  endpoint {
    url = "http://loki:3100/loki/api/v1/push"
  }
}

읽는 법: discovery.docker가 컨테이너 목록을 뽑고, loki.source.docker가 그 컨테이너들의 로그를 읽어, forward_to로 지정된 loki.write에 넘긴다. 앱 코드는 한 줄도 건드리지 않는다 — 컨테이너가 stdout에 찍는 로그를 밖에서 받아가는 구조다.

4-2. 대안 — 앱이 Loki로 직접 쏘기

수집기 없이 Spring Boot 앱이 Loki HTTP API로 직접 로그를 보내는 방법도 있다(logback용 Loki appender 라이브러리 사용). 다만 이 방식은 앱과 로깅 인프라가 결합되고(Loki가 죽으면 앱 로깅 경로에 영향), 백엔드 외 다른 컨테이너(PostgreSQL 등) 로그는 못 담는다. 일반적으로는 Alloy 방식(수집기가 밖에서 수거)이 표준이고, 우리 설계에서도 그쪽이 유력하다 — 최종 선택은 설계 단계에서.

5. LogQL — 로그 질의 언어

LogQL은 Grafana 화면에서 로그를 검색할 때 쓰는 언어다. 구조는 항상 "스트림 선택 + 파이프(|)로 가공" 2단이다. 셸에서 cat file | grep ... 하는 감각과 같다.

5-1. 스트림 셀렉터 — 어느 로그를 볼 것인가 (필수)

{container="wn-be-green"}                  # 백엔드 컨테이너의 모든 로그
{container=~"wn-be-.*"}                    # 정규식 매칭 (blue/green 둘 다)
{container="wn-be-green", level="ERROR"}   # 라벨 두 개로 좁히기

중괄호 안이 라벨 매처다. 모든 질의는 최소 한 개의 라벨 매처로 시작해야 한다 — 색인된 라벨로 먼저 범위를 좁혀야 하기 때문이다.

5-2. 라인 필터 — 본문에서 골라내기

{container="wn-be-green"} |= "ERROR"          # "ERROR" 포함 라인만
{container="wn-be-green"} |= "ERROR" != "test" # 포함 + 제외 연쇄
{container="wn-be-green"} |~ "Exception|OOM"   # 정규식 필터
연산자의미
|=문자열 포함
!=문자열 미포함
|~정규식 일치
!~정규식 불일치

5-3. 파서 — 본문을 필드로 쪼개기

로그 본문이 JSON이나 key=value(logfmt) 형식이면, 파서를 파이프에 끼워 본문 안의 필드를 질의에 쓸 수 있다. 저장할 때 색인하지 않았어도 질의 시점에 구조를 뽑아내는 것 — 이게 Loki의 "나중에 파싱(schema on read)" 철학이다.

# 본문이 JSON일 때: {"level":"ERROR","logger":"ShareService","message":"..."}
{container="wn-be-green"} | json | logger="ShareService"

# 파싱한 필드로 숫자 비교도 가능
{container="wn-be-green"} | json | duration_ms > 1000

5-4. 메트릭 질의 — 로그를 숫자로 바꿔 그래프 그리기

로그 줄 수를 세어 시계열 그래프로 만들 수 있다. 대시보드와 알림의 재료가 된다.

# 5분 구간마다 에러 로그 개수
sum(count_over_time({container="wn-be-green"} |= "ERROR" [5m]))

# 초당 로그 발생률
rate({container="wn-be-green"}[1m])

# 레벨별로 나눠 세기 (JSON 로그 가정)
sum by (level) (count_over_time({container="wn-be-green"} | json [5m]))

PromQL을 써봤다면 rate, sum by가 그대로라 익숙할 것이고, 안 써봤어도 "구간을 정해 세고, 라벨별로 묶는다"만 이해하면 된다.

6. Grafana — 조회·시각화·알림

Grafana는 Loki 전용 도구가 아니라 범용 시각화 플랫폼이다. Loki를 데이터소스(data source)로 등록하면 그때부터 로그를 다룰 수 있다. 처음에 쓰게 될 화면은 세 가지다.

6-1. Explore — 로그 조회의 주 무대

왼쪽 메뉴의 Explore가 로그를 뒤지는 화면이다. 데이터소스를 Loki로 고르면 LogQL 입력창이 나오고, 라벨 브라우저(어떤 라벨·값이 있는지 클릭으로 탐색)와 질의 빌더를 제공해서 LogQL을 다 외우지 않아도 된다. 실시간으로 흘러오는 로그를 보는 Live tail 기능도 여기 있다. "방금 배포했는데 에러 나나?"를 볼 때 docker logs -f 대신 쓰게 되는 화면.

6-2. 대시보드 — 반복해서 볼 화면을 저장

메트릭 질의(5-4)를 시계열 패널로, 로그 질의를 로그 패널로 만들어 한 화면에 배치한다. 예: "시간대별 에러 수 그래프 + 최근 ERROR 로그 목록 + 5xx 응답 카운트"를 한 장에. 커뮤니티가 만든 대시보드를 가져올 수도 있다.

6-3. 알림 (Alerting) — 사람이 안 보고 있어도

메트릭 질의에 조건을 걸어 알림 규칙을 만든다. 예: "5분간 ERROR 로그가 10건 넘으면" 이메일·Slack·Discord 등으로 통지. 솔로 운영에서 사실상 가장 가치 있는 기능이다 — 대시보드는 안 볼 수 있지만 알림은 찾아온다.

참고 — Grafana 로그인과 노출

Grafana는 자체 계정 체계(기본 admin 계정)가 있는 웹 앱이다. 운영 서버에 올릴 때는 외부에 직접 노출하지 않고 Caddy 뒤에 두거나 SSH 터널로만 접근하는 방식을 설계 단계에서 정한다(8절).

7. 직접 만져보기 — 로컬 10분 실습

운영 서버를 건드리지 않고, 지금 쓰는 Mac에서 스택 전체를 띄워 감을 잡는 실습이다. 끝나면 docker compose down -v로 흔적 없이 지워진다. (Apple Silicon에서 세 이미지 모두 arm64를 지원한다.)

준비물: Docker Desktop 실행 중일 것. 아래 3개 파일을 빈 디렉토리(예: ~/loki-playground)에 만든다.

파일 1 — loki-config.yaml (모놀리식 + 로컬 디스크 저장 최소 설정)

auth_enabled: false

server:

실습 단계

  1. 스택 기동 — 파일 3개가 있는 디렉토리에서: 
    docker compose up -d
    컨테이너 3개(loki, alloy, grafana)가 Started로 뜬다. docker compose ps로 확인.
  2. Grafana 접속 — 브라우저에서 http://localhost:3200 접속, 아이디 admin / 비밀번호 admin.
    Grafana 홈 대시보드가 보인다.
  3. 데이터소스 등록 — 좌측 메뉴 Connections에서 Data sources로 들어가 Add new data source, Loki 선택. URL에 http://loki:3100 입력(컨테이너끼리는 서비스 이름으로 통신) 후 Save & test.
    "Data source successfully connected." 초록 메시지.
  4. 첫 질의 — 좌측 메뉴 Explore, 데이터소스 Loki 선택, 질의창에 입력 후 실행(Shift+Enter): 
    {container=~".+"}
    이 Mac에서 돌고 있는 컨테이너들(loki, alloy, grafana 자신 포함)의 로그가 쏟아진다. Alloy가 docker.sock으로 전부 수집하고 있다는 증거.
  5. 필터 연습 — 앞 절에서 배운 것들을 실제로 쳐본다: 
    {container="grafana"} |= "error"
sum by (container) (count_over_time({container=~".+"}[1m]))
    두 번째 질의는 로그가 아니라 컨테이너별 분당 로그 수 그래프가 그려진다 — 로그가 숫자가 되는 순간.
  6. 선택: 소설비 로컬 dev와 연결해 보기 — 로컬에서 docker compose up -d --wait postgres + 백엔드 bootRun을 돌리고 있다면, postgres 컨테이너 로그가 이미 수집되고 있다. {container="write-note-postgres"}로 조회해 보라.
  7. 정리: 
    docker compose down -v
    컨테이너·볼륨 모두 삭제. Mac에 남는 것 없음.

8. 우리 인프라(OCI free-a1)에 적용할 때

2026-07-02 실측 기준으로 확인된 사실과, 설계 시 짚을 지점들이다.

8-1. 자원은 충분하다 (실측 + 추정)

항목값근거
서버 유휴 메모리약 10GB / 12GB실측 백엔드 516MB + PostgreSQL 52MB + Caddy 등만 상주
Loki 예상 상주200~500MB추정 소규모 모놀리식 기준
Alloy 예상 상주100~200MB추정
Grafana 예상 상주200~400MB추정
디스크185GB 여유실측 보존 30일이면 현 트래픽에서 수 GB 미만 예상

합계 약 1GB 내외로 ELK(약 3GB+) 대비 3분의 1 이하. 다만 서버에 스왑이 없고 운영 서비스와 동거하므로, 세 컨테이너 모두 메모리 상한(mem_limit)을 걸어 올리는 것이 안전하다 — 이 원칙은 설계에 반영한다.

8-2. 보안 — 노출 범위를 좁힌다

  • Loki(3100)와 Alloy는 외부 노출 금지 — Docker 내부 네트워크에서만 통신하면 되고, 호스트 포트 바인딩 자체가 필요 없다.
  • Grafana만 사람이 접근해야 하는데, 방식은 설계에서 결정: (a) Caddy에 서브도메인 추가해 HTTPS 노출(+Grafana 로그인), (b) SSH 터널로만 접근(포트 미노출, 가장 보수적).
  • Alloy가 /var/run/docker.sock을 읽는다 — 이는 Docker 제어 권한과 맞먹는 강한 권한이므로 read-only 마운트로 제한한다.

8-3. 로그 포맷 — JSON 구조화가 LogQL 활용도를 결정한다

백엔드(Spring Boot)가 지금처럼 사람용 텍스트 로그를 찍으면 |= 필터까지만 편하고, | json 파서의 힘(레벨별 집계, 필드 비교)은 못 쓴다. Spring Boot는 구조화 로깅(JSON 출력)을 설정으로 지원하므로(3.4부터 내장 — 우리 버전(4.x)에서의 정확한 설정 키는 설계 단계에서 확인), 운영 프로파일만 JSON 출력으로 바꾸는 것을 설계에서 검토한다. 로컬 개발은 사람이 읽는 텍스트 유지.

8-4. 기타

  • 세 이미지 모두 arm64(우리 서버 아키텍처)를 공식 지원한다.
  • Docker의 json-file 로그 로테이션(max-size)은 Loki 도입과 별개로 걸어두는 것이 좋다 — Alloy가 읽는 원본이 무한히 커지지 않게.
  • blue-green 배포 시 컨테이너 이름이 wn-be-blue/wn-be-green으로 바뀐다 — 라벨 설계에서 container=~"wn-be-.*"로 묶거나 고정 라벨(app="backend")을 부여하는 처리가 필요하다. 설계 포인트.
