# 페이지 분할(0-4 안1) dogfooding 보고서

- 작성 시각: 2026-06-07 23:09 KST
- 기준 문서: `docs/poc/0-4-page-split-dogfooding.md`
- 검증 방식: Computer Use로 Electron 실앱 화면 조작 + DB 읽기 확인
- 대상 앱: 기존 실행 중이던 `desktop` Electron dev 앱

## 환경 메모

- 문서 명령(`cd desktop && ... && pnpm dev`)을 실행했으나, 기존 `5173` dev 서버가 이미 떠 있어 새 Vite는 `5174`로 기동했다.
- Computer Use가 실제로 잡은 창은 `localhost:5173/`에 연결된 기존 Electron 프로세스였다.
- 검증 중 `5174` 세션은 종료했다. 기존 `5173` 앱/서버는 그대로 둔 상태다.
- DB 경로: `~/Library/Application Support/write-note-desktop/write-note.db`

## 요약 판정

전체 GREEN 아님. A 한글 IME hard-gate와 E 줌, F 신규 입력 저장 검증을 통과로 볼 수 없다.

```text
A 한글IME: A1 △ / A2 △ / A3 △ / A4 × / A5 △
B 분할:    B1 ○ / B2 △ / B3 ○ / B4 △
C 클릭:    C1 ○ / C2 △ / C3 △ / C4 ×
D 외관:    D1 ○ / D2 ○ / D3 △ / D4 ○ / D5 ○ / D6 ○ / D7 ○
E 줌:      E1 × / E2 ×
F 저장:    F1 × / F2 △ / F3 △
G 기존:    G1 △
```

범례:
- `○`: Computer Use 화면/접근성 트리로 확인
- `×`: 문제 관찰 또는 조작 불가로 요구사항 미충족
- `△`: 이번 환경에서 충분히 검증하지 못함

## 관찰 상세

### A. 한글 IME

- `press_key`로 실제 키 이벤트를 보냈을 때 현재 입력기는 한글 상태였다. `r` 입력은 화면에 `ㄱ`, 이어 `k` 입력은 `가`로 조합됐다.
- 이후 빠른 연속 입력, Backspace, Enter 처리에서 화면/접근성 트리/DB 반영이 일관되지 않았다.
- `BackSpace`를 여러 번 보냈지만 화면상 삭제 변화가 보이지 않았다. `Backspace` 표기는 도구에서 `keyNotFound`로 실패했다.
- 한자 변환 키는 Computer Use 도구에서 직접 보낼 수 있는 키가 확인되지 않아 검증하지 못했다.
- 결론: A는 hard-gate GREEN으로 판단할 수 없다. 특히 A4는 RED로 기록한다.

### B. 실시간 페이지 분할

- 접근성 트리에 `1쪽`~`5쪽` 컨테이너가 노출됐다.
- 화면상 여러 장의 독립 종이, 장 사이 책상 배경, 하단 쪽번호가 확인됐다.
- 한 장 넘게 새로 길게 입력하는 검증은 입력 반영이 안정적으로 되지 않아 충분히 수행하지 못했다.
- 중간 삭제로 장 수가 줄어드는 B4는 Backspace/Delete가 기대대로 작동하지 않아 검증하지 못했다.

### C. 노트식 마우스 클릭

- 빈 줄 좌표 클릭 후 커서가 해당 빈 줄로 이동했다. 이 점은 C1 GREEN으로 본다.
- 글자 위 클릭, 드래그 선택은 이번 라운드에서 별도 확인하지 못했다.
- 줌 변경이 동작하지 않아 C4는 RED로 본다.

### D. 외관

- 줄노트 ON 상태에서 줄선은 종이 내부에 보였고 책상 간격에는 줄선이 보이지 않았다.
- 종이는 둥근 모서리/그림자/여백을 가지고 독립 장으로 보였다.
- 하단 쪽번호 `1`~`5`와 종이 밖 제목 `6얼 벚꽃동산`이 확인됐다.
- 본문 포커스 중 하늘색 장별 외곽선은 보이지 않았다.
- 빈 새 종이의 줄선 full-page 여부는 별도 빈 문서로 확인하지 못했다.

### E. 줌

- 보기 메뉴는 열렸고 `100%`, `+`, `-`, 줄노트/테마/자동저장 토글이 화면에 보였다.
- 하지만 접근성 트리와 실제 화면 상태가 불일치했고, 좌표로 `+`를 눌러도 확대 변화가 확인되지 않았다.
- 따라서 E1/E2는 RED로 기록한다.

### F. 저장/복원/전환

- 입력 시도 후 상단 상태는 계속 `저장됨 · 261자`로 남았다.
- DB 조회 결과 현재 문서 `6얼 벚꽃동산`도 `wordCount=261`이고, 검증 중 입력한 조합/문구가 저장본 tail에 반영되지 않았다.
- 기존 저장본이 앱 화면에 로드되는 것은 확인됐지만, 신규 입력 저장과 재시작 복원은 GREEN으로 볼 수 없다.

### G. 기존 문서

- 기존 문서 2개가 작품 목록/DB에 존재했고, `6얼 벚꽃동산` 문서는 5쪽으로 분할 렌더링됐다.
- 제목/목록/인용 등 서식 섞인 문서를 별도로 준비해 열지는 못했다.

## RED / 리스크

1. A4: Backspace/Delete가 화면상 삭제로 이어지지 않았다.
2. E1/E2: 보기 메뉴의 줌 조작이 실제 확대/축소로 확인되지 않았다.
3. F1: 입력 후 저장 상태와 DB 저장본이 갱신되지 않았다.
4. 검증 환경에 기존 `5173` 앱과 새 `5174` 앱이 동시에 떠 있었다. 실제 조작 대상은 `5173`이었다.

## 다음 확인 권장

1. 기존 `5173` Electron/Vite 세션을 종료하고 단일 `pnpm dev` 세션만 띄운 뒤 재검증한다.
2. A 항목은 사용자가 실제 물리 키보드로 한자키 포함 재검증해야 한다. Computer Use만으로는 한자키를 충분히 보낼 수 없었다.
3. 줌/줄노트 메뉴는 접근성 상태와 실제 화면 상태 불일치부터 재현 범위를 좁힌다.
4. 입력 후 `onUpdate`/`editor.view.composing` guard가 조합 종료 후 저장 이벤트를 다시 발생시키는지 확인한다.
